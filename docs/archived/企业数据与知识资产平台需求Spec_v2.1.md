## 一、文档说明

本文档基于 [企业数据与知识资产平台nexus_v7.0.md](/home/bjbodao/projects/nexus/docs/企业数据与知识资产平台nexus_v7.0.md) 和 [企业数据与知识资产平台技术选型和架构nexus_v2.1.md](/home/bjbodao/projects/nexus/docs/企业数据与知识资产平台技术选型和架构nexus_v2.1.md) 输出平台需求规格说明，定义一期建设范围、角色、功能需求、非功能需求、数据要求、接口要求和验收口径。

本文档用于：

1. 统一业务、产品、研发、测试、实施对平台一期范围的理解。
2. 作为研发实施、测试验收、原型设计和交付评审的依据。
3. 为后续版本迭代提供可追溯的需求基线。
4. 为试点上线提供可验证的业务成功标准。

### 1.1 v2.1 需求优化原则

1. 需求必须与架构 v2.1 的主数据模型一致，不能在产品需求中引入冗余反向指针或旧状态口径。
2. 需求必须区分 NEXUS、MinerU、RAGFlow、爬虫系统、钉钉通讯录适配器和上层业务系统的职责边界。
3. 需求必须覆盖原始留存、标准化契约、作业状态、可配置治理规则、治理决策追踪、权限审计、质量复核和检索引用追溯。
4. 需求必须明确 P0、P1、P2 优先级，发布、监控、告警、容量规划等运维业务不得作为一期产品化范围。
5. 角色模型必须收敛，平台管理与数据治理由统一角色承担，授权业务访问统一纳入 API 调用方模型。
6. 分类、分级、标签、组织范围、质量准入、复核触发、索引准入规则必须可配置，不允许写死在接入或处理流程中。
7. 数据资产治理和数据质量评分应采用 AI 主导、规则护栏、人工辅助的方式，人工不再作为全量必经审核节点。
8. AI 输出必须结构化、可解释、可追溯、可抽检，并保留证据引用、置信度、模型配置和提示词版本。
9. 验收标准必须可测试、可演示、可追踪，避免只停留在能力描述。

## 二、本轮 Review 结论与 v2.1 优化

### 2.1 Review 发现

| 序号 | Review 发现 | 风险 | v2.1 优化动作 |
|------|-------------|------|--------------|
| P-01 | v1.2 仍引用企业 IAM / SSO | 与架构 v2.0 不一致，形成外部身份平台强依赖 | 改为 NEXUS 本地用户、组织、角色和 API 调用方；钉钉仅作为可选通讯录同步源。 |
| P-02 | 运维观测、告警、Runbook、容量展示被写入一期产品范围 | 超出一期架构边界，扩大交付范围 | 将运维业务降级为架构预留；一期只要求健康检查、结构化日志、请求追踪、作业状态和基础运行状态。 |
| P-03 | 资产版本仍使用“发布 / 现行有效”等旧产品流，且出现旧状态语义 | 容易被实现为人工发布流，增加审核负担 | 对齐 `processing`、`available`、`review_required`、`archived`、`disabled`、`failed`；默认自动进入 `available`，异常才复核。 |
| P-04 | 资产当前版本和标准化引用在需求中没有明确“派生读取”约束 | 产品与接口可能重新要求当前版本字段和标准化引用反向字段 | 明确当前版本通过唯一 `available` 版本读取，标准化引用通过 `normalized_asset_ref.version_id` 读取。 |
| P-05 | 分类、分级、标签需求停留在“配置字典和审核”，缺少规则配置闭环 | 无法指导治理规则页面、规则发布、重跑治理和决策追踪开发 | 新增治理规则配置、规则集版本、规则校验、规则发布、重跑治理、治理决策追踪需求。 |
| P-06 | 组织范围治理没有作为独立自动治理对象 | 权限范围可能依赖人工填写，影响自动可用 | 将组织范围纳入自动治理规则和复核触发条件。 |
| P-07 | Prototype 需要页面级输入，但 Spec v1.2 未明确规则配置页面和决策追踪页面 | 原型无法覆盖架构 v2.0 的关键新能力 | 更新控制台信息架构和功能需求，明确治理规则、规则决策日志、当前版本读取展示。 |
| P-08 | 数据资产治理和质量评分尚未明确 AI 主导链路 | 仅靠规则和人工候选难以支撑规模化治理，质量评分也难以解释和闭环 | 新增 AI 治理建议、AI 质量评分、证据引用、置信度、模型配置、人工反馈回灌和 AI 重评分需求。 |
| P-09 | AI 能力若没有规则护栏和人工抽检，可能产生幻觉、越权和错误采纳 | 错误分类、错误分级或敏感内容误放行会直接影响权限和检索结果 | 明确 AI 输出必须经过 Schema 校验、规则护栏、敏感策略、质量阈值和状态机判定后才能采纳。 |

### 2.2 v2.1 产品结论

1. 一期产品目标是打通 D1-D4 数据从接入、治理、索引到检索开放的最小可用闭环，不以知识图谱、SFT、评价标准库和运维中心产品化为上线目标。
2. P0 必须保障资产可入库、可治理、可自动进入可用、可检索、可追溯、可控权、可审计、规则可配置。
3. 治理中心从“标签审核中心”升级为“AI 治理建议 + 质量评分 + 规则护栏 + 治理待办 + 决策追踪 + 人工反馈”的统一入口。
4. 资产版本默认由 AI 生成治理建议和质量评分，再由规则护栏自动采纳；只有质量失败、规则冲突、低置信度、高敏风险、组织范围不明、抽检命中或索引准入阻断时进入人工复核。
5. API 与控制台均不得暴露或依赖 `current_version_id`、`normalized_ref_id`、`quality_report_id` 这类反向指针字段；产品上展示“当前版本”和“当前标准化引用”时，应理解为读取模型结果。
6. AI 输出不得直接等同于正式治理结果；正式结果必须能说明 AI 建议、规则采纳、人工覆盖和最终状态之间的关系。
7. 需求验收采用“功能演示 + 样本数据 + 指标验证 + 权限用例 + AI 治理用例 + 规则治理用例 + 故障用例 + 文档交付”的组合方式。

## 三、项目背景、目标与成功指标

### 3.1 背景问题

当前企业内外部非结构化数据存在以下共性问题：

1. 数据分散在网盘、NAS、系统库表、爬虫结果和业务文件中，缺少统一入口。
2. 相同文档在不同项目中重复处理，缺少原始留存和标准化资产沉淀。
3. 文档解析结果、治理规则、检索索引和权限控制之间没有统一口径。
4. 分类、分级、标签和组织范围过度依赖人工确认，难以支撑规模化资产入库。
5. 数据质量评分缺少可解释维度和证据引用，难以判断问题来源和修复优先级。
6. 上层业务系统和 AI 应用需要重复建设知识库、问答语料和检索服务。
7. 数据资产和知识资产缺少版本、质量、权限、审计和来源引用的一致口径。

### 3.2 建设目标

平台建设目标如下：

1. 构建统一的数据源接入层、原始数据持久化层和作业编排处理层。
2. 构建统一的标准化资产模型，形成数据资产与知识资产的共同输入件。
3. 建立分类、分级、标签、组织范围、质量准入、索引准入、版本、权限、审计等治理体系。
4. 引入 AI 大模型作为数据资产治理和质量评分的主判断能力，输出建议、评分、证据和置信度。
5. 提供可配置治理规则能力，作为 AI 输出的规则护栏，支持规则发布、规则追踪、冲突处理和重跑治理。
6. 对外提供统一的资产查询、检索、问答、作业、规则治理和权限校验接口。
7. 为 AI 助教、知识检索、教材生成、图谱构建、评价标准化等场景提供底座支撑。
8. 通过 `nexus-console` 承载控制面治理入口，通过 `nexus-api` 承载统一服务开放入口。

### 3.3 一期业务成功指标

| 指标 | 目标 | 说明 |
|------|------|------|
| 首批数据域覆盖 | D1-D4 均至少完成一类样本数据接入 | 验证分类、分级、标签、组织范围和索引链路适配不同业务域。 |
| 原始对象落库成功率 | 不低于 95% | 按试点接入样本统计。 |
| 标准化资产可追溯率 | 100% | 任一标准化资产可回溯到原始对象、资产版本和标准化引用。 |
| 自动治理通过率 | 联调阶段不低于 60%，试点稳定后不低于 75% | 规则命中、质量达标、无冲突的资产自动进入 `available`。 |
| AI 治理自动采纳率 | 联调阶段不低于 60%，试点稳定后不低于 75% | AI 建议经规则护栏采纳后自动进入后续准入流程。 |
| AI 质量评分可解释率 | 100% | 每份质量报告必须包含维度分、证据引用、置信度和阻断原因。 |
| 人工复核可解释率 | 100% | 任一 `review_required` 资产必须能查看触发规则、冲突或失败原因。 |
| 人工覆盖反馈留存率 | 100% | 人工修改 AI 建议或质量分时必须记录原因和反馈标签。 |
| Top-5 召回率 | 联调不低于 60%，试点后不低于 80% | 基于业务专家确认的检索问题集。 |
| 问答结果引用率 | 100% | 问答结果必须带来源引用。 |
| 权限误放行率 | 0 | 任一未授权内容返回均为阻塞问题。 |
| 作业失败可定位率 | 100% | 失败必须能定位阶段、原因、重试次数和关联对象。 |
| 试点用户满意度 | 业务专家评审通过 | 以试点验收会结论为准。 |

## 四、术语定义

| 术语 | 定义 |
|------|------|
| 数据资产 | 经过接入、登记、留存和治理后，可被统一管理和复用的数据对象。 |
| 知识资产 | 在数据资产基础上经过切片、抽取、结构化或审核加工后形成的可直接消费对象。 |
| 原始对象 `raw_object` | 原始文件、原始 JSON 包、原始媒体引用等可信留存对象。 |
| 文档资产 `document_asset` | 平台侧资产主实体，承载长期资产身份、资产级治理基线和生命周期，不保存当前版本指针。 |
| 资产版本 `document_version` | 文档资产的版本实例，是处理、治理和索引边界。 |
| 当前版本 | 由同一资产下唯一 `available` 的 `document_version` 派生，不是 `document_asset` 主表字段。 |
| 标准化文档 `normalized_document` | 面向 PDF、Word、PPT、图片等文档输入的统一标准对象。 |
| 标准化记录 `normalized_record` | 面向 JSON、Excel、数据库同步数据、Webhook 事件的统一标准对象。 |
| 标准化引用 `normalized_asset_ref` | 通过 `version_id` 关联资产版本，记录标准化对象 URI、schema、摘要和状态。 |
| AI 模型配置 `ai_model_profile` | 治理和质量评分使用的模型、提示词版本、输出 Schema、脱敏策略和任务类型配置。 |
| AI 治理执行 `ai_governance_run` | 记录 AI 对某资产版本生成的治理建议、质量评分、证据引用、置信度和采纳状态。 |
| AI 治理建议 | AI 基于标准化对象输出的分类、分级、标签、组织范围、索引准入建议和理由。 |
| AI 质量评分 | AI 基于标准化对象输出的质量维度分、综合分、问题列表、证据引用和修复建议。 |
| 规则护栏 | 对 AI 输出执行 Schema、枚举、分级、组织范围、敏感策略、质量阈值和索引准入校验的规则集合。 |
| 质量报告 `quality_report` | 通过 `version_id` 关联资产版本，记录 AI 质量评分、规则检查、维度分、证据引用、置信度和准入结论。 |
| 治理规则集 `governance_rule_set` | 一组可发布、可回滚、可追踪的治理规则版本。 |
| 治理规则 `governance_rule` | 分类、分级、标签、组织范围、质量准入、复核触发、索引准入的条件与动作配置。 |
| 治理结果 `governance_result` | 资产版本的正式分类、分级、标签、组织范围和索引准入结论。 |
| 治理决策日志 `governance_decision_log` | 记录 AI 建议、规则命中、候选结果、冲突、置信度、采纳状态、最终结果和人工覆盖。 |
| 知识切片 `knowledge_chunk` | 用于检索、问答和知识加工的标准切片对象。 |
| 索引清单 `index_manifest` | 记录索引分区、同步状态、失败原因与同步时间的清单对象。 |
| API 调用方 | 上层业务系统、智能应用、集成方和授权业务访问入口的统一平台消费角色。 |
| 钉钉通讯录适配器 | 可选的组织用户同步源，不能作为平台运行强依赖。 |
| MinerU | 非结构化文档解析执行引擎。 |
| RAGFlow | 切片、索引与检索执行引擎。 |
| 控制面 | 接入、元数据、规则、作业、权限、审计、运营和管理能力集合。 |
| 执行面 | 解析、标准化、AI 治理与质量评分、规则护栏、索引同步、检索执行和知识加工能力集合。 |

## 五、角色与使用方

### 5.1 平台角色

v2.1 延续角色收敛规则：原“平台管理员”和“数据管理员”合并为“平台/数据管理员”；授权业务访问不再单列为独立角色，统一合入“API 调用方”。

| 角色 | 说明 | 主要职责 |
|------|------|---------|
| 平台/数据管理员 | 平台级管理与数据治理合并角色 | 账号与角色管理、组织范围配置、数据源注册、资产审核、AI 治理建议采纳、质量评分复核、治理规则配置、规则发布、版本管理、权限配置、审批处理、审计查看。 |
| 业务专家 | 业务审核角色 | 标签修订、治理规则建议、AI 建议抽检、AI 评分校准、规则效果确认、知识资产审核、质量抽检、试点场景定义、业务验收。 |
| 运维人员 | 平台维护角色 | 一期仅查看基础运行状态、作业失败、健康检查和故障排查信息；发布、监控、告警、容量规划不作为一期产品化业务。 |
| API 调用方 | 能力消费角色 | 上层业务系统、智能应用、集成方和授权业务访问入口，通过 API Key 或 JWT 调用资产、检索、问答、作业接口。 |

### 5.2 API 调用方子类型

API 调用方不拆成平台角色，但产品和权限配置需识别以下子类型：

| 子类型 | 典型对象 | 主要诉求 | 权限特点 |
|--------|----------|----------|----------|
| 业务系统调用方 | 教学平台、实训平台、题库系统、学情分析系统 | 稳定检索、资产查询、问答引用 | 按系统、组织、数据域和接口授权。 |
| 智能应用调用方 | AI 助教、教材生成、智能体搭建平台 | 检索上下文、问答、知识资产消费 | 需要更严格的引用、脱敏和频控。 |
| 数据接入调用方 | 爬虫系统、同步连接器、批量导入工具 | 批量推送、接入状态、作业回查 | 只允许访问接入和作业接口。 |
| 授权业务访问入口 | 上层应用中的教师、教研人员、项目成员 | 查询授权资产、查看引用、反馈问题 | 由上层应用代理认证，NEXUS 按调用方和用户上下文联合鉴权。 |

### 5.3 身份与组织来源

| 来源模式 | 一期要求 |
|----------|----------|
| 本地维护 | P0。平台必须支持本地组织、用户、角色、API 调用方和组织范围维护。 |
| 钉钉同步 | P1 / 可选。仅作为组织用户初始化或同步来源，同步后落入本地主数据。 |

## 八、业务场景

### 8.1 场景一：静态文档入库并自动可用

平台/数据管理员或系统连接器提交教材、产业报告、人才培养方案等文档，平台完成原始留存、解析、标准化、AI 治理建议、AI 质量评分、规则护栏执行、索引和资产入库。

验收输出：资产、版本、原始对象、解析产物、标准化引用、AI 治理执行、质量报告、治理结果、决策日志、知识切片、索引清单均可查询。满足准入条件的版本自动进入 `available`。

### 8.2 场景二：爬虫动态数据接入

爬虫系统按批次推送产业政策、岗位招聘和人才需求数据，平台完成批次登记、原始包留存、标准化处理、AI 治理与质量评分、规则护栏治理与索引更新。

验收输出：批次号、作业号、原始 JSON 包、标准化记录、AI 建议、质量维度分、治理决策、索引状态可回查。

### 8.3 场景三：AI 主导的数据资产治理与质量评分

平台对标准化对象自动调用 AI 治理能力，生成分类、分级、标签、组织范围、质量评分、证据引用、问题列表和置信度。规则护栏对 AI 输出进行合法性校验和准入判断，高置信且无阻断的资产自动进入后续可用流程。

验收输出：AI 模型配置、提示词版本、输入摘要、AI 建议、质量维度分、证据引用、置信度、采纳状态和人工反馈均可查询。

### 8.4 场景四：可配置治理规则

平台/数据管理员配置分类、分级、标签、组织范围、质量准入、复核触发和索引准入规则，校验后发布规则集。新接入资产先生成 AI 建议，再使用 active 规则集执行护栏治理。

验收输出：规则集版本、规则发布审计、AI 建议采纳记录、规则命中记录、治理结果、冲突处理结果均可查询。

### 8.5 场景五：资产人工复核

当 AI 治理出现低置信度、规则冲突、高敏风险、组织范围不明、质量不达标、证据缺失或抽检命中时，资产版本进入 `review_required`，平台/数据管理员或业务专家查看 AI 建议、质量评分、证据引用和决策日志并人工处理。

验收输出：复核原因、AI 候选值、质量维度分、命中规则、人工修改前后值、复核结论、反馈标签和审计记录。

### 8.6 场景六：知识检索与问答

API 调用方通过 `nexus-api` 或上层应用入口执行检索和问答，平台返回带引用、带权限过滤、带脱敏处理的结果。

验收输出：结果包含来源文档、版本、标准化引用、切片 ID、引用位置、权限说明和请求 ID。

### 8.7 场景七：重处理、重治理与版本切换

当模型配置升级、规则升级、解析失败、索引失效、质量评分需校准或人工复核发现问题时，平台触发重处理、AI 重评分或重治理。新版本满足准入条件后进入 `available`，旧可用版本自动归档。

验收输出：旧版本 `archived`，新版本 `available`；当前版本通过读取模型展示；索引投影失效或重建。

### 8.8 场景八：知识资产加工

基于标准化资产和知识切片，平台生成问答语料、流程语料、图谱结构和评价标准库，并经过审核后进入对应消费场景。

一期边界：检索知识库为 P0，问答草稿和流程模型为 P1，图谱和评价标准库为 P2。

### 8.9 场景九：基础故障排查

运维人员或平台/数据管理员通过工作台、作业中心、审计日志查看基础运行状态、失败作业、错误码、请求 ID 和 trace ID，并执行授权范围内的重试或重处理。

验收输出：故障可定位、可追踪、可重试；不要求独立告警中心或运维观测平台。

### 9.1 角色、认证与权限主体需求

| 编号 | 需求 | 优先级 |
|------|------|--------|
| FR-RBAC-001 | 平台必须将平台管理和数据治理能力统一归入“平台/数据管理员”角色 | P0 |

| FR-RBAC-003 | 平台必须支持本地控制台用户登录、API 调用方 API Key / JWT 认证、后台作业账号认证 | P0 |
| FR-RBAC-004 | 平台必须支持本地组织、用户、角色、API 调用方和组织范围维护 | P0 |

### 13.1 功能验收

一期验收通过需同时满足以下条件：

1. 文档可统一接入、留存、解析、标准化、AI 治理与质量评分、规则护栏、索引和检索。
2. 爬虫数据可批量推送、原始包留存、标准化处理、AI 治理、规则治理和索引更新。
3. 控制台可完成数据源管理、作业查询、AI 建议查看、质量评分查看、规则配置、治理复核、资产查看、权限管理和审计查询。
4. API 可完成资产查询、当前版本查询、检索、问答、作业查询、AI 治理执行查询、重处理触发、规则治理和权限校验。
5. 权限与审计链路可运行，L4 数据默认脱敏，未授权内容不返回。
6. RAGFlow 集成链路完整，`index_manifest` 可回查。
7. 任一检索和问答结果都能追溯到资产版本、标准化引用、知识切片和原始对象。
8. AI 治理建议和 AI 质量评分均可追溯到模型配置、提示词版本、输入摘要、证据引用、规则采纳状态和人工反馈。
9. 平台角色配置符合 v2.1 收敛规则，不再拆分平台管理与数据治理角色，不再单列授权业务访问角色。
10. 至少完成一次作业失败重试、索引失败重建、AI 重评分、规则重治理和权限隔离测试。
11. 无外部统一身份平台时，平台仍可完成本地组织用户维护、接入、治理、检索和 API 调用主链路。

| E2E-010 API 幂等 | 重复提交同一 `idempotency_key` 接入请求 | 不产生重复有效资产，返回同一批次或作业引用。 |
| E2E-011 本地身份闭环 | 不配置钉钉同步，手工维护组织用户 | 可完成登录、授权、治理、检索和审计。 |
| E2E-012 AI 重评分与反馈 | 对低质量样本提交人工评分校准并触发 AI 重评分 | 生成新 `ai_governance_run` 和 `quality_report`，保留人工反馈和评分变化。 |

### 13.4 试点验收样本

试点验收至少覆盖以下样本：

1. D1 产业政策或行业报告。
2. D2 岗位招聘或职业能力分析数据。
3. D3 人才培养方案或教学标准。
4. D4 教材、教案、课件或实训指导书。
5. 至少 20 个业务检索问题。
6. 至少 5 个权限隔离测试用例。
7. 至少 3 个失败重试或重处理测试用例。
8. 至少 3 条治理规则配置与命中用例。
9. 至少 5 个 AI 治理建议自动采纳样本。
10. 至少 3 个 AI 质量评分可解释样本。
11. 至少 2 个规则冲突或 AI 低置信进入复核用例。
12. 至少 2 个 API 调用方联调用例。

### 13.5 文档验收

一期交付必须同时包含以下文档：

1. 标准化资产规范。
2. 切片规范。
3. 元数据与主数据规范。
4. 治理规则配置规范。
5. AI 治理与质量评分规范。
6. RAGFlow 集成规范。
7. 技术选型和架构文档。
8. API 接口文档。
9. 权限与审计设计说明。
10. 部署说明和基础维护说明。

### 13.6 Go / No-Go 标准

| 决策项 | Go 标准 | No-Go 条件 |
|--------|---------|------------|
| 权限安全 | 权限误放行率为 0 | 任一未授权内容返回。 |
| 数据追溯 | 标准化资产可追溯率 100% | 任一验收样本无法回溯原始对象。 |
| AI 治理 | AI 治理建议和质量评分可追溯率 100% | 任一 AI 结论无法说明模型配置、证据引用或采纳状态。 |
| 治理追踪 | 治理决策可追溯率 100% | 任一自动治理或人工覆盖无法说明依据。 |
| 检索质量 | Top-5 召回率达到联调基线 | 关键试点问题大量无结果或引用错误。 |
| 作业稳定 | 失败可定位且可重试 | 作业失败原因不可查或状态丢失。 |
| 审计合规 | 关键动作审计完整 | 高敏访问、规则发布、权限变更无审计。 |
| 身份组织 | 本地身份组织闭环可运行 | 无外部身份源时平台不可用。 |

## 十五、依赖、假设与风险

### 15.1 外部依赖

| 依赖 | 影响 | 责任方 |
|------|------|--------|
| MinerU 服务可用 | 文档解析链路 | 技术团队 / 运维 |
| RAGFlow 服务可用 | 切片、索引、检索链路 | 技术团队 / 运维 |
| AI 大模型服务可用 | AI 治理建议和质量评分链路 | 技术团队 / 运维 |
| 爬虫系统推送契约 | D1/D2 动态数据接入 | 爬虫系统团队 |
| 钉钉通讯录权限 | 可选组织用户同步 | 企业 IT / 平台团队 |
| 业务专家参与 | 分类分级、标签、规则确认、试点验收 | 业务部门 |
| GPU 资源 | Hybrid / VLM 解析能力 | 运维 / 资源管理 |

### 15.2 关键假设

1. 一期试点数据以 D1-D4 为主，D5/D6 只做模型和适配器预留。
2. RAGFlow 可以满足一期切片、索引和检索执行需求。
3. AI 模型可以通过 `llm-gateway` 输出稳定结构化结果，且支持必要的私有化或脱敏调用策略。
4. 业务专家可在第 1 周完成分类、分级、标签、组织范围和质量评分样本确认。
5. 单节点或三节点环境可提供所需 GPU、对象存储和数据库资源。
6. 上层业务系统愿意通过 `nexus-api` 接入，不直接依赖 RAGFlow 原生接口。
7. 钉钉同步不可用时，平台可通过本地维护完成组织用户闭环。

### 15.3 产品风险与缓解

| 风险 | 表现 | 缓解策略 |
|------|------|----------|
| 范围膨胀 | 图谱、SFT、评价标准库、运维中心被要求一期全部生产化 | 以 P0/P1/P2 控制范围，P2 不作为上线阻塞。 |
| 治理规则滞后 | 分类分级和组织范围规则不清导致大量资产进入复核 | 第 1 周设置规则确认 Go/No-Go，先用简单规则覆盖高频数据源。 |
| AI 建议不稳定 | 同类资产出现分类、分级或标签建议不一致 | 固定提示词版本、使用结构化输出、引入规则护栏、建立人工抽检和反馈回灌。 |
| AI 质量评分不可解释 | 只有分值没有证据，业务专家无法信任 | 强制维度分、证据引用、阻断原因和人工校准记录。 |
| 敏感内容外发风险 | 外部模型收到未脱敏 L3/L4 明文 | 默认脱敏或元数据调用，高敏样本使用私有化模型或进入人工复核。 |
| 人工复核压力过大 | 低置信度或规则冲突样本过多 | 优化规则阈值、补充数据源默认提示、批量复核和重跑治理。 |
| 检索效果不稳定 | Top-5 召回达不到试点目标 | 建立问题集、调参、标签修订和切片重建闭环。 |
| 权限模型复杂 | API 调用方、上层用户、组织范围混淆 | 强制所有调用携带调用方与用户上下文，统一由 NEXUS 求值。 |
| 数据质量不稳定 | 文档格式复杂、扫描件质量差 | 质量评分、人工复核、解析策略升级和重处理。 |

## 十六、版本演进说明

### 16.1 v2.1 范围结论

v2.1 的核心定位是：

1. 对齐架构 v2.1 的主数据模型、AI 治理链路和状态基线。
2. 打通接入、解析、标准化、AI 治理与质量评分、规则护栏、索引和检索主链路。
3. 完成企业级治理与服务开放的最小可用闭环。
4. 明确 NEXUS、MinerU、RAGFlow、爬虫系统、钉钉通讯录和上层业务系统边界。
5. 收敛角色模型，降低权限配置和验收复杂度。
6. 用 AI 主导、规则护栏、人工辅助降低人工审核负担。
7. 用可配置治理规则和 AI 质量评分增强治理可解释性。
8. 用 P0/P1/P2、业务 KPI、端到端用例和 Go/No-Go 标准提升需求可执行性。

### 16.2 后续版本方向

后续版本沿以下方向演进：

1. D5 / D6 业务数据正式接入。
2. 问答语料、流程语料、图谱和评价标准库规模化加工。
3. 治理规则效果分析、规则模拟器、灰度发布和自动优化。
4. AI 模型效果分析、提示词在线优化、模型 A/B、主动学习和批量重评分。
5. 钉钉通讯录同步生产化。
6. 独立运维观测中心、告警中心和容量规划系统。
7. 数据库、检索、AI 推理和观测体系高可用升级。
8. 引入更完整的知识图谱 API、Agent 流程模板服务和评价标准服务。
