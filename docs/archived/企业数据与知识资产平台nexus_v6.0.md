# 企业数据与知识资产平台 v6.0 — NEXUS

## 一、项目定位

NEXUS 面向企业现有非结构化数据，建设统一的数据沉淀、治理、复用与开放平台，解决企业数据分散、处理重复、管理粗放、使用门槛高的问题。

平台沉淀两类核心资产：

1. 面向业务应用的数据资产
2. 面向大模型和智能应用的知识资产

平台作为企业级数据与知识资产基础设施，统一支撑资产管理和能力开放两类核心场景。

---

## 二、产品结构

平台采用"一个统一数据资产底座 + 两类终端"的产品结构，并集成两个外部系统。

```
┌─────────────────────────────────────────────────────────────────┐
│                      外部系统集成层                               │
│   ┌─────────────────────┐   ┌─────────────────────────────┐    │
│   │   爬虫系统           │   │   RAGFlow 系统               │    │
│   │ 产业政策 / 人才需求  │   │ 检索 / 召回 / 知识组织        │    │
│   └──────────┬──────────┘   └──────────────┬──────────────┘    │
└──────────────┼───────────────────────────── ┼───────────────────┘
               ▼                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                    统一数据资产底座                                │
│                                                                  │
│  数据源接入 → 文档解析与清洗 → 分类分级与标签治理 →              │
│  权限与可见范围控制 → 检索、召回与知识组织                        │
│                                                                  │
└────────────────────────┬────────────────────────────────────────┘
                         │
          ┌──────────────┴───────────────┐
          ▼                              ▼
┌──────────────────┐          ┌──────────────────────┐
│  nexus-console   │          │     nexus-api         │
│  数据管理员终端   │          │  业务系统 API 终端     │
└──────────────────┘          └──────────────────────┘
```

### 外部系统集成

**爬虫系统**
定期爬取产业政策数据和行业人才需求数据，作为平台实时动态数据源的输入通道，通过标准数据接口推送至底座接入层。

**RAGFlow 系统**
提供检索、召回和知识组织能力，支撑平台统一的知识服务链路。底座完成文档解析与切片后，将结构化结果推送至 RAGFlow 构建检索索引，nexus-api 通过 RAGFlow 标准接口获取检索与问答结果。

## 三、统一数据资产底座

统一数据资产底座承载平台核心能力，是整个平台的数据处理与服务基础。

### 3.1 底座整体架构

底座按数据流向分为五个处理层级，各层级职责独立，数据单向流转：

```
原始数据源
    │
    ▼
[数据源接入层]
静态文件接入 / 爬虫动态接入 / 接入状态管理
    │
    ▼
[文档解析与清洗层]
MinerU 解析引擎 / 结构识别 / 内容清洗 / 切片处理 / 元数据抽取
    │
    ▼
[资产治理层]
分类分级 / 标签体系 / 版本管理 / 生命周期管理
    │
    ▼
[权限与访问控制层]
角色权限 / 可见范围 / 鉴权校验 / 审计记录
    │
    ▼
[检索与知识服务层]（由 RAGFlow 支撑）
索引构建 / 检索召回 / 知识组织 / 结果输出
```

底座统一管理以下核心存储：

| 存储类型 | 用途 | 建议选型 |
|---------|------|---------|
| 关系型数据库 | 资产元数据、分类标签、权限配置、审计记录 | PostgreSQL |
| 对象存储 | 原始文件、解析结果文件 | MinIO（私有化）/ 云对象存储 |
| 向量数据库 | 文档切片向量索引（由 RAGFlow 管理） | RAGFlow 内置（基于 Elasticsearch + 向量引擎） |
| 缓存 | 热点检索结果、权限校验缓存 | Redis |
| 消息队列 | 文档处理任务异步调度 | Redis Streams / RabbitMQ |

### 3.2 数据源接入

平台统一接入两类数据源，通过标准化接入链路完成数据汇聚。

#### 静态数据源

| 数据类型 | 接入方式 |
|---------|---------|
| 制度文件 | 文件上传（单文件 / 批量上传） |
| 行业研究资料 | 文件上传 / 目录批量导入 |
| 项目交付文档 | 文件上传 / 外部存储同步 |
| 历史知识材料 | 批量导入 / 历史迁移 |
| 业务专题文档 | 文件上传 / 关联数据集 |

支持的文件格式：PDF、Word（.docx/.doc）、PPT（.pptx/.ppt）、Excel（.xlsx/.xls）、纯文本（.txt/.md）、图片（.jpg/.png，含扫描件）。

#### 实时动态数据源

由爬虫系统通过以下方式推送至底座：

1. 增量推送接口：爬虫系统按周期将新增数据通过 HTTP 接口推送至底座接入层
2. 数据格式要求：JSON 结构，包含原始内容、来源 URL、采集时间、数据类型等标准字段
3. 接入后进入与静态数据相同的解析清洗流程

#### 接入状态管理

每条接入记录维护以下状态：

```
待处理 → 解析中 → 清洗中 → 治理中 → 已入库
                                        ↓
                               解析失败 / 清洗失败（人工介入）
```

### 3.3 文档解析与清洗

文档解析是整个底座数据质量的入口，采用 MinerU 作为核心解析引擎。

#### 3.3.1 MinerU 解析引擎

MinerU 是面向 AI 场景的高精度文档解析工具，支持将 PDF、Word、图片等多格式文件转换为结构化的 Markdown 和 JSON 输出。

**核心解析能力：**

| 能力 | 说明 |
|------|------|
| 多格式解析 | 支持 PDF（文本型 / 扫描型）、Word、图片等输入 |
| 版面分析 | 识别正文、标题、表格、图片、公式等区域，还原阅读顺序 |
| 表格提取 | 将表格结构转换为 HTML 格式，保留行列关系 |
| 公式识别 | 将数学公式转换为 LaTeX 格式 |
| OCR 支持 | 支持扫描件和图片类文档，覆盖中英文及多语言场景 |
| 噪声过滤 | 自动去除页眉、页脚、页码、水印等无效内容 |
| 输出格式 | 结构化 Markdown（含标题层级）和 JSON（含元数据） |

**解析后端策略：**

| 场景 | 推荐后端 | 说明 |
|------|---------|------|
| 标准文本类 PDF | Pipeline（CPU） | 速度快，适合大批量处理 |
| 复杂排版 / 图文混排 | Hybrid-engine | 结合原生文本提取与 VLM，减少幻觉 |
| 扫描件 / 低质量图片 | VLM-engine | 精度最高，需要 GPU 支持 |

生产环境建议：标准文档默认使用 Pipeline 后端，对解析质量抽样评分低于阈值的文档自动触发 Hybrid-engine 重新处理。

**MinerU 部署方式：**

以 FastAPI 服务形式独立部署，底座通过内部 HTTP 接口调用，支持任务排队和异步回调：

```
底座任务调度器 → POST /parse（文件 + 参数）→ MinerU 服务
                ← 异步回调（解析结果 JSON + Markdown）
```

#### 3.3.2 标准化处理流程

MinerU 完成解析后，进入底座标准清洗流程：

**步骤一：结构识别**
- 提取文档标题层级（H1/H2/H3）
- 识别段落、列表、表格、图片说明
- 标注章节边界，建立文档大纲

**步骤二：格式标准化**
- 统一字符编码（UTF-8）
- 统一时间格式（ISO 8601）
- 清理多余空白字符、异常符号

**步骤三：内容去噪**
- 去除模板语、声明语、版权页等无业务价值内容
- 识别并标记低质量段落（字符密度异常、乱码）

**步骤四：重复检测**
- 文档级：基于文件哈希检测完全重复文件
- 段落级：基于 MinHash / SimHash 检测高相似段落，标记但不强制删除

**步骤五：内容切片**
切片是检索质量的关键环节，采用以下分层切片策略：

| 切片类型 | 策略 | 适用场景 |
|---------|------|---------|
| 段落切片 | 按自然段落边界切分 | 叙述类文档 |
| 语义切片 | 按语义相关性聚合分段 | 长文档、研究报告 |
| 章节切片 | 按标题层级切分，保留上下文 | 制度文件、操作手册 |
| 固定窗口切片 | 固定 token 数 + 滑动窗口重叠 | 无明显结构的文档 |

每个切片记录以下字段：切片 ID、原始文档 ID、切片内容、切片类型、所属章节路径、切片序号、字符数。

**步骤六：元数据抽取**
结合 MinerU 解析结果和内容语义分析，抽取以下元数据：

| 字段类型 | 字段示例 |
|---------|---------|
| 来源信息 | 来源系统、上传部门、原始文件名、文件格式 |
| 时间信息 | 创建时间、发布时间、最后更新时间 |
| 内容信息 | 主题领域、关键词（自动提取）、语言类型 |
| 质量信息 | 解析质量评分、切片数量、有效字符数 |

**步骤七：质量校验**
每批处理完成后执行校验：
- 解析完整性：正文是否为空、标题是否提取成功
- 字段完整性：必填元数据字段是否缺失
- 切片合理性：切片数量是否在合理范围（过少可能解析失败，过多可能过度切分）
- 质量评分：综合以上指标生成 0-100 分的质量评分，低于 60 分标记为"待人工复核"

**步骤八：结果入库**
- 解析结果（Markdown / JSON）写入对象存储
- 切片结果写入关系型数据库（切片记录表）
- 向量化后的切片推送至 RAGFlow 构建索引
- 元数据写入资产元数据表
- 处理日志写入任务记录表

### 3.4 分类分级与标签治理

#### 分类体系

采用三级分类结构：

```
一级分类（资产类型）
├── 制度与规范类
│   ├── 企业制度
│   ├── 操作规程
│   └── 合规要求
├── 研究与报告类
│   ├── 行业研究
│   ├── 产业政策
│   └── 人才需求报告
├── 项目与交付类
│   ├── 项目方案
│   ├── 交付报告
│   └── 技术文档
└── 知识与经验类
    ├── 业务知识
    ├── 经验总结
    └── 培训材料
```

分类体系在阶段 0 由业务专家与数据管理员联合制定，写入系统后作为治理规则基准，后续调整需经审批。

#### 数据分级规则

| 级别 | 定义 | 可见范围 |
|------|------|---------|
| L1 公开 | 可对外公开的信息 | 全员 + 授权外部用户 |
| L2 内部 | 企业内部通用信息 | 全员 |
| L3 受限 | 特定部门或角色可见 | 按部门 / 角色授权 |
| L4 机密 | 核心敏感数据 | 指定人员，需审批 |

#### 标签体系

平台维护以下标签维度，每个资产可挂载多个标签：

| 标签维度 | 示例 |
|---------|------|
| 主题标签 | 产业政策、人才发展、项目管理 |
| 来源标签 | 政府文件、内部制定、外部采购 |
| 部门标签 | 研发部、运营部、人力资源部 |
| 时效标签 | 现行有效、历史存档、待更新 |
| 场景标签 | 新员工培训、客户交付、合规审查 |

**自动标签辅助**：解析完成后，系统基于内容关键词和分类规则自动推荐标签，管理员确认或修订后生效。

#### 数据资产与知识资产的关系模型

这是平台数据一致性的核心设计，明确两类资产的衍生关系：

```
原始文件（数据资产）
    │  1:N
    ▼
文档解析结果（解析版本）
    │  1:N
    ▼
内容切片（知识资产单元）
    │  N:1（聚合至 RAGFlow）
    ▼
检索索引（RAGFlow 管理）
```

**版本联动机制**：
- 原始文件更新（重新上传或版本迭代）时，系统创建新的解析版本
- 新解析版本完成后，自动触发切片重建和 RAGFlow 索引更新
- 旧版本切片标记为"历史版本"，不参与检索，但保留供审计查询
- 版本对应关系在元数据中完整记录，确保任意检索结果均可追溯至原始文件版本

### 3.5 权限与可见范围控制

平台采用 RBAC（基于角色的访问控制）+ 资产分级过滤的复合权限模型。

#### 权限层级

| 权限层级 | 控制粒度 | 执行位置 |
|---------|---------|---------|
| 终端访问权限 | 用户是否可登录 nexus-console / 调用 nexus-api | 认证层 |
| 功能权限 | 用户在控制台可执行哪些操作 | 应用层 |
| 资产访问权限 | 用户可访问哪些数据集、知识库 | 数据层 |
| 内容可见范围 | 检索结果中哪些切片对该用户可见 | 检索层（RAGFlow 过滤） |

#### 权限配置维度

1. **按角色配置**：管理员 / 普通用户 / 只读用户 / API 调用方
2. **按部门配置**：部门数据默认仅本部门可见，跨部门访问需授权
3. **按资产分级**：L3 / L4 级资产需显式授权，未授权用户在检索结果中不可见
4. **按终端类型**：控制台具有管理权限，API 端仅具有查询和调用权限

#### 权限校验流程

```
请求进入（Console 操作 / API 调用）
    │
    ▼
身份认证（Token 校验）
    │
    ▼
功能权限校验（是否有操作权限）
    │
    ▼
资产权限校验（是否有目标资产的访问权）
    │
    ▼
结果过滤（检索结果按可见范围过滤）
    │
    ▼
审计记录（操作人、时间、访问资产、结果条数）
```

### 3.6 检索、召回与知识组织

本模块由 RAGFlow 系统承载，底座负责数据准备和接入协调。

#### RAGFlow 在底座中的角色

RAGFlow 作为独立部署的知识服务系统，与底座的交互关系如下：

| 方向 | 内容 |
|------|------|
| 底座 → RAGFlow | 切片内容 + 元数据推送，触发索引构建 |
| 底座 → RAGFlow | 权限过滤条件，在检索时传入 |
| RAGFlow → 底座 | 检索结果（含切片 ID、来源文档、相关度评分） |
| nexus-api → RAGFlow | 检索召回请求、问答请求（通过底座服务层转发） |

#### 检索能力

RAGFlow 提供三种检索模式，可按场景配置：

| 检索模式 | 原理 | 适用场景 |
|---------|------|---------|
| 关键词检索 | 基于全文索引（Elasticsearch） | 精确词汇查找、编号查询 |
| 语义检索 | 基于向量相似度（嵌入模型） | 意图理解、相似内容发现 |
| 混合检索 | 关键词 + 语义融合，RRF 重排 | 通用业务问答，推荐默认配置 |

#### 召回与重排

1. 初始召回：按检索模式返回 Top-K 候选切片
2. 过滤：按权限可见范围、分类、标签、时间过滤
3. 重排：基于 Re-ranker 模型对候选结果重新排序
4. 知识组织：将相关切片聚合，构建结构化上下文供问答使用

#### 知识资产质量基线

建立以下评估指标，在阶段 1 完成后建立基线，作为后续优化的参照：

| 指标 | 定义 | 目标基线 |
|------|------|---------|
| 解析成功率 | 成功完成解析的文档占比 | ≥ 95% |
| 切片质量评分 | 平均切片质量评分 | ≥ 70 分 |
| 检索召回率 | 试点场景测试集的召回率 | ≥ 80%（Top-5） |
| 答案可追溯率 | 问答结果中有明确来源引用的比例 | 100% |

## 四、两类终端职责

### nexus-console

面向数据管理员，承担资产管理与治理入口职责。

**核心功能模块：**

| 模块 | 主要功能 |
|------|---------|
| 数据源管理 | 静态数据源上传 / 批量导入；爬虫数据源配置；接入状态查看 |
| 解析清洗管理 | 解析后端配置；清洗规则配置；处理任务监控；失败任务重处理 |
| 分类分级管理 | 分类体系维护；分级规则配置；标签体系管理；自动标签审核 |
| 资产目录管理 | 资产列表查看；元数据编辑；版本历史查看；资产状态管理 |
| 权限管理 | 角色管理；用户授权；部门权限配置；审批流配置 |
| 运营与审计 | 处理任务统计；访问统计；调用分析；操作审计日志 |

### nexus-api

面向上层业务系统，以标准化接口开放底座能力。

**API 接口分类：**

API 接口按调用性质分为两类，采用不同的架构设计：

**同步查询类（低延迟，有 SLA 要求）：**

| 接口 | 说明 | 目标响应时间 |
|------|------|------------|
| `GET /assets` | 资产列表查询，支持分类、标签、时间过滤 | < 200ms |
| `GET /assets/{id}` | 资产详情及元数据查询 | < 100ms |
| `POST /search` | 检索召回（关键词 / 语义 / 混合） | < 1s |
| `POST /qa` | 知识问答（RAGFlow 驱动） | < 5s |
| `POST /auth/verify` | 权限校验 | < 50ms |

**异步处理类（高吞吐，队列调度）：**

| 接口 | 说明 |
|------|------|
| `POST /ingest` | 提交文档解析入库任务，返回任务 ID |
| `GET /tasks/{id}` | 查询任务处理状态和结果 |
| `POST /tasks/{id}/retry` | 重新触发失败任务 |

**通用能力：**
- 鉴权：所有接口需携带 API Key 或 JWT Token
- 限流：按调用方独立限流（默认 100 QPS / 调用方）
- 审计：每次调用记录调用方、接口、参数摘要、响应时间、结果条数
- 监控：接口可用性、响应时间 P99 纳入运营统计面板

---

## 五、核心价值

1. 将分散数据沉淀为企业可持续复用的资产，减少重复加工和重复建设。
2. 为 FLOW 及后续业务系统提供统一的数据与知识底座。
3. 为上层业务应用提供统一的标准化能力输出接口。
4. 为知识问答、智能检索、智能助手等 AI 场景提供统一支撑。
5. 推动数据管理从项目式交付转向平台化、标准化、服务化运营。

---

## 六、项目实施计划

项目实施周期 6 周，**前两周优先夯实底座数据管道**，后四周在可靠底座上构建终端能力，避免"框架建完但管道不通"的风险。

| 周次 | 工作重点 | 主要内容 | 输出成果 |
|------|---------|---------|---------|
| 第 1 周 | 前置准备与底座设计 | 确定试点场景和首批数据清单；制定分类分级规则初版和标签体系；完成资产模型、元数据字段、数据库表结构设计；完成 MinerU 服务部署验证 | 治理规则初版；资产数据模型；数据库设计；MinerU 服务可用 |
| 第 2 周 | 解析清洗管道打通 | 完成文件上传接入链路；接入 MinerU 解析服务（Pipeline 后端）；完成清洗流程（去噪、去重、切片、元数据抽取）；完成质量校验和结果入库；建立解析质量基线 | 解析清洗管道端到端打通；首批文档完成处理；质量基线确立 |
| 第 3 周 | 治理能力与权限建设 | 完成分类分级、标签配置能力；完成权限模型实现（RBAC + 分级过滤）；完成 RAGFlow 对接（索引构建 + 基础检索验证）；完成版本联动机制 | 治理配置可用；权限模型可运行；RAGFlow 检索链路打通 |
| 第 4 周 | 控制台基础版建设 | 完成数据源管理页面；完成解析任务监控页面；完成分类标签管理页面；完成资产目录浏览页面；完成权限管理基础配置 | nexus-console 基础版可用；主流程可操作 |
| 第 5 周 | API 服务建设与联调 | 完成资产查询 API；完成检索召回 API；完成知识问答 API；完成鉴权、限流、审计能力；完成同步 / 异步接口分离；底座 + Console + API 三层联调 | nexus-api 基础版可调用；三层联调完成 |
| 第 6 周 | 试点验证与收口 | 开展 1-2 个试点场景完整验证；优化解析规则、治理规则、权限配置、接口稳定性；评估质量基线指标达成情况；整理交付材料 | 试点验证报告；问题优化清单；一期交付版本；交付文档 |

**关键里程碑：**
- 第 1 周末：治理规则确定，数据模型设计完成——后续所有工作量以此为基准
- 第 2 周末：解析管道端到端打通——这是整个平台的数据质量入口，必须优先验证
- 第 5 周末：三层联调完成——进入试点验证的前提条件
