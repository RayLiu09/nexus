## 一、文档说明

本文档基于 [企业数据与知识资产平台nexus_v7.0.md](/home/bjbodao/projects/nexus/docs/企业数据与知识资产平台nexus_v7.0.md) 和 [企业数据与知识资产平台技术选型和架构nexus_v1.2.md](/home/bjbodao/projects/nexus/docs/企业数据与知识资产平台技术选型和架构nexus_v1.2.md) 输出平台需求规格说明，定义一期建设范围、角色、功能需求、非功能需求、数据要求、接口要求和验收口径。

本文档用于：

1. 统一业务、产品、研发、测试、实施对平台范围的理解。
2. 作为研发实施、测试验收和交付评审的依据。
3. 为后续版本迭代提供需求基线。
4. 为试点上线提供可验证的业务成功标准。

### 1.1 v1.2 需求优化原则

1. 需求必须与 v7.0 的“统一数据资产底座 + 两类终端”产品结构一致。
2. 需求必须区分 NEXUS、MinerU、RAGFlow、爬虫系统和上层业务系统的职责边界。
3. 需求必须覆盖原始留存、标准化契约、作业状态、权限审计、质量复核和检索引用追溯。
4. 需求必须明确 P0、P1、P2 优先级，避免把长期规划全部压入一期。
5. 角色模型必须收敛，平台管理与数据治理由统一角色承担，授权业务访问统一纳入 API 调用方模型。
6. 验收标准必须可测试、可演示、可追踪，避免只停留在能力描述。

## 二、资深产品经理 Review 结论与 v1.2 优化

### 2.1 Review 发现

| 序号 | Review 发现 | 风险 | v1.2 优化动作 |
|------|-------------|------|--------------|
| P-01 | v1.1 建设目标偏能力化，缺少业务成功指标 | 验收时只能看功能是否存在，难以判断是否真正解决业务问题 | 增加业务目标、试点 KPI 和上线 Go/No-Go 标准。 |
| P-02 | 功能需求都是“一期必须/应具备”，缺少更细的 MVP 优先级 | 研发排期时难以取舍，延期风险高 | 增加 P0/P1/P2 优先级与 MVP 边界。 |
| P-03 | 角色已经收敛，但用户旅程和控制台入口不够清晰 | 研发可实现功能，但用户操作流可能割裂 | 增加角色旅程、控制台信息架构和端到端流程。 |
| P-04 | API 调用方范围较宽，缺少消费场景拆分 | 上层系统、智能应用、授权业务访问的权限和 SLA 容易混淆 | 将 API 调用方拆成消费子类型，但不新增平台角色。 |
| P-05 | 验收指标有结果指标，缺少样本、测试用例和数据准备标准 | 试点验收可能争议样本是否代表真实场景 | 增加试点数据包、验收用例、验收样本和通过标准。 |
| P-06 | 知识资产加工范围仍容易被理解为一期全部生产化 | 造成问答语料、流程语料、图谱、评价标准库范围膨胀 | 明确一期只做检索知识库 P0，问答草稿和流程模型为 P1，图谱和评价为 P2 预留。 |
| P-07 | 运维和运营要求已有基础，但缺少上线后运营指标 | 平台上线后缺少持续迭代依据 | 增加运营看板、质量看板、资产增长、检索效果和问题闭环指标。 |

### 2.2 v1.2 产品优化结论

1. 一期产品目标不是“完成所有知识资产加工”，而是打通 D1-D4 数据从接入、治理、索引到检索开放的最小可用闭环。
2. P0 必须保障资产可入库、可治理、可检索、可追溯、可控权、可审计。
3. P1 聚焦试点效率和质量提升，包括问答草稿、知识资产审核基础版、观测页、API Key 额度配置和批量运营能力。
4. P2 只做预留，不作为一期上线阻塞项，包括 D5/D6 全量接入、知识图谱生产化、SFT 大规模量产、评价标准库生产化和高可用升级。
5. 需求验收必须采用“功能演示 + 数据样本 + 指标验证 + 权限用例 + 故障用例 + 文档交付”的组合方式。

## 三、项目背景、目标与成功指标

### 3.1 背景问题

当前企业内外部非结构化数据存在以下共性问题：

1. 数据分散在网盘、NAS、系统库表、爬虫结果和业务文件中，缺少统一入口。
2. 相同文档在不同项目中重复处理，缺少原始留存和标准化资产沉淀。
3. 文档解析结果、标签规则、检索索引和权限控制之间没有统一治理模型。
4. 上层业务系统和 AI 应用需要重复建设知识库、问答语料和检索服务。
5. 数据资产和知识资产缺少版本、质量、权限、审计和来源引用的一致口径。

### 3.2 建设目标

平台建设目标如下：

1. 构建统一的数据源接入层、原始数据持久化层和作业编排处理层。
2. 构建统一的标准化资产模型，形成数据资产与知识资产的共同输入件。
3. 建立分类、分级、标签、版本、权限、审计等治理体系。
4. 对外提供统一的资产查询、检索、问答、作业和权限校验接口。
5. 为 AI 助教、知识检索、教材生成、图谱构建、评价标准化等场景提供底座支撑。
6. 通过 `nexus-console` 承载控制面治理入口，通过 `nexus-api` 承载统一服务开放入口。

### 3.3 一期业务成功指标

| 指标 | 目标 | 说明 |
|------|------|------|
| 首批数据域覆盖 | D1-D4 均至少完成一类样本数据接入 | 验证分类、分级、标签、索引链路适配不同业务域。 |
| 原始对象落库成功率 | 不低于 95% | 按试点接入样本统计。 |
| 标准化资产可追溯率 | 100% | 任一标准化资产可回溯到原始对象和资产版本。 |
| Top-5 召回率 | 联调不低于 60%，试点后不低于 80% | 基于业务专家确认的检索问题集。 |
| 问答结果引用率 | 100% | 问答结果必须带来源引用。 |
| 权限误放行率 | 0 | 任一未授权内容返回均为阻塞问题。 |
| 作业失败可定位率 | 100% | 失败必须能定位阶段、原因、重试次数和关联对象。 |
| 试点用户满意度 | 业务专家评审通过 | 以试点验收会结论为准。 |

## 四、术语定义

| 术语 | 定义 |
|------|------|
| 数据资产 | 经过接入、登记、留存和治理后，可被统一管理和复用的数据对象。 |
| 知识资产 | 在数据资产基础上经过切片、抽取、结构化或审核加工后形成的可直接消费对象。 |
| 原始对象 `raw_object` | 原始文件、原始 JSON 包、原始媒体引用等可信留存对象。 |
| 文档资产 `document_asset` | 平台侧文档主实体，承载分类、分级、权限和生命周期。 |
| 资产版本 `document_version` | 文档资产的版本实例，关联解析产物和标准化产物。 |
| 标准化文档 `normalized_document` | 面向 PDF、Word、PPT、图片等文档输入的统一标准对象。 |
| 标准化记录 `normalized_record` | 面向 JSON、Excel、数据库同步数据、Webhook 事件的统一标准对象。 |
| 知识切片 `knowledge_chunk` | 用于检索、问答和知识加工的标准切片对象。 |
| 索引清单 `index_manifest` | 记录索引分区、同步状态、失败原因与同步时间的清单对象。 |
| 质量报告 `quality_report` | 记录解析、标准化、元数据、切片质量与复核状态的质量对象。 |
| 知识资产版本 `knowledge_asset_version` | 问答语料、流程语料、图谱、评价标准等加工产物的版本对象。 |
| API 调用方 | 上层业务系统、智能应用、集成方和授权业务访问入口的统一平台消费角色。 |
| MinerU | 非结构化文档解析执行引擎。 |
| RAGFlow | 切片、索引与检索执行引擎。 |
| 控制面 | 接入、元数据、作业、权限、审计、运营和管理能力集合。 |
| 执行面 | 解析、标准化、标签抽取、索引同步、检索执行和知识加工能力集合。 |

## 五、角色与使用方

### 5.1 平台角色

v1.2 延续 v1.1 的角色收敛规则：原“平台管理员”和“数据管理员”合并为“平台/数据管理员”；授权业务访问不再单列为独立角色，统一合入“API 调用方”。

| 角色 | 说明 | 主要职责 |
|------|------|---------|
| 平台/数据管理员 | 平台级管理与数据治理合并角色 | 账号与角色管理、组织范围配置、系统配置、数据源注册、资产审核、分类分级、标签确认、版本管理、权限配置、审批处理、审计查看。 |
| 业务专家 | 业务审核角色 | 标签修订、分类规则确认、知识资产审核、质量抽检、试点场景定义、业务验收。 |
| 运维人员 | 平台运维角色 | 发布、监控、告警、容量管理、备份恢复、故障处理、运行报告输出。 |
| API 调用方 | 能力消费角色 | 上层业务系统、智能应用、集成方和授权业务访问入口，通过 API Key、JWT 或上层应用代理调用资产、检索、问答、作业接口。 |

### 5.2 API 调用方子类型

API 调用方不拆成平台角色，但产品和权限配置需识别以下子类型：

| 子类型 | 典型对象 | 主要诉求 | 权限特点 |
|--------|----------|----------|----------|
| 业务系统调用方 | 教学平台、实训平台、题库系统、学情分析系统 | 稳定检索、资产查询、问答引用 | 按系统、组织、数据域和接口授权。 |
| 智能应用调用方 | AI 助教、教材生成、智能体搭建平台 | 检索上下文、问答、知识资产消费 | 需要更严格的引用、脱敏和频控。 |
| 数据接入调用方 | 爬虫系统、同步连接器、批量导入工具 | 批量推送、接入状态、作业回查 | 只允许访问接入和作业接口。 |
| 授权业务访问入口 | 上层应用中的教师、教研人员、项目成员 | 查询授权资产、查看引用、反馈问题 | 由上层应用代理认证，NEXUS 按调用方和用户上下文联合鉴权。 |

### 5.3 非人工技术主体

| 主体 | 说明 | 权限要求 |
|------|------|---------|
| 系统连接器 | NAS 同步、爬虫推送、数据库同步、Webhook 等接入主体 | 必须绑定来源系统、组织范围、数据域和最小权限。 |
| 后台作业账号 | Worker、定时任务、索引同步、回调任务等内部主体 | 必须使用内部凭据，权限仅限必要的作业和数据范围。 |

### 5.4 外部系统

| 系统 | 与平台关系 |
|------|-----------|
| 爬虫系统 | 作为动态数据源推送产业政策、岗位招聘、人才需求等数据。 |
| MinerU | 作为文档解析执行引擎。 |
| RAGFlow | 作为切片、索引与检索执行引擎。 |
| 上层业务系统 | 通过 `nexus-api` 消费平台能力。 |
| 企业 IAM / SSO | 为控制台与 API 提供统一身份认证。 |

### 5.5 角色权限矩阵

| 能力 | 平台/数据管理员 | 业务专家 | 运维人员 | API 调用方 |
|------|----------------|---------|---------|-----------|
| 数据源注册与配置 | 可管理 | 可查看规则 | 不可操作 | 不可操作 |
| 原始对象台账查看 | 可查看 | 按授权查看 | 按故障处理需要查看元信息 | 不可直接查看 |
| 分类分级配置 | 可管理 | 可建议 / 审核 | 不可操作 | 不可操作 |
| 标签草稿审核 | 可确认 / 修订 / 驳回 | 可修订 / 建议 | 不可操作 | 不可操作 |
| 作业重试与重处理 | 可触发 | 可申请 | 可触发故障恢复类操作 | 可按授权触发指定作业 |
| 知识资产审核发布 | 可发布 | 可审核 | 不可操作 | 不可操作 |
| 资产查询与检索 | 按授权访问 | 按授权访问 | 按运维授权访问元信息 | 按 API 权限访问 |
| 权限配置与审批 | 可管理 | 可参与业务审批 | 不可操作 | 可发起授权申请 |
| 审计查询 | 可查看 | 按需查看自身审核记录 | 可查看运维相关日志 | 可查看自身调用记录 |
| 系统监控与告警 | 可查看 | 不可操作 | 可管理 | 不可操作 |

## 六、用户旅程与产品信息架构

### 6.1 平台/数据管理员旅程

| 阶段 | 操作 | 系统输出 |
|------|------|----------|
| 配置数据源 | 注册上传入口、NAS 目录、爬虫来源、默认分类与组织范围 | `data_source`、接入策略、默认治理规则。 |
| 接入数据 | 上传文件或查看爬虫批次 | `ingest_batch`、`raw_object`、接入状态。 |
| 处理监控 | 查看作业进度、失败原因、重试和复核队列 | 作业状态、错误详情、重试结果。 |

| 发布开放 | 将资产版本发布到检索知识库并配置权限 | `published` 版本、`index_manifest`、权限策略。 |

| 阶段 | 操作 | 系统输出 |
|------|------|----------|
| 规则确认 | 审核分类树、标签维度、质量门槛 | 已确认治理规则。 |

| 检索试测 | 使用业务问题集验证检索和问答结果 | 召回率、引用准确性、问题清单。 |

### 6.3 API 调用方旅程

| 阶段 | 操作 | 系统输出 |

## 七、范围定义与优先级

### 7.1 优先级定义

| 优先级 | 定义 | 上线约束 |
|--------|------|----------|
| P0 | MVP 必须完成，缺失则一期不可上线 | 阻塞上线。 |
| P1 | 一期应具备或试点增强能力，可基础版交付 | 不阻塞最小上线，但影响试点效果。 |
| P2 | 二期预留或后续演进能力 | 不纳入一期上线验收。 |

## 八、业务场景

### 8.1 场景一：静态文档入库

平台/数据管理员或系统连接器提交教材、产业报告、人才培养方案等文档，平台完成原始留存、解析、标准化、索引和资产入库。

验收输出：资产版本、原始对象、解析产物、标准化对象、知识切片、索引清单均可查询。

### 8.2 场景二：爬虫动态数据接入

爬虫系统按批次推送产业政策、岗位招聘和人才需求数据，平台完成批次登记、原始包留存、标准化处理与索引更新。

验收输出：批次号、作业号、原始 JSON 包、结构化记录、索引状态可回查。

### 8.3 场景三：资产治理

平台/数据管理员对新入库资产执行分类分级、标签确认、版本状态管理和人工复核；业务专家参与业务规则与知识质量审核。

验收输出：分类、分级、标签、版本状态、复核动作均有审计记录。

### 8.4 场景四：知识检索与问答

API 调用方通过 `nexus-api` 或上层应用入口执行检索和问答，平台返回带引用、带权限过滤、带脱敏处理的结果。

验收输出：结果包含来源文档、版本、切片 ID、引用位置、权限说明和请求 ID。

### 8.5 场景五：重处理与版本切换

当规则升级、解析失败、索引失效或人工复核发现问题时，平台触发重处理并生成新版本资产。

验收输出：旧版本归档，新版本按审核结果发布，旧索引投影失效或重建。

### 8.6 场景六：知识资产加工

基于标准化资产和知识切片，平台生成问答语料、流程语料、图谱结构和评价标准库，并经过审核后发布。

一期边界：检索知识库为 P0，问答草稿和流程模型为 P1，图谱和评价标准库为 P2。

### 8.7 场景七：运维监控与故障恢复

运维人员通过观测能力查看 API、作业、解析、索引、存储和安全审计指标，对失败任务、队列积压、索引异常和容量风险进行处理。

验收输出：告警、Runbook、故障恢复记录和恢复后验证结果。

## 十三、验收口径

### 13.1 功能验收

一期验收通过需同时满足以下条件：

1. 文档可统一接入、留存、解析、标准化、索引和检索。
2. 爬虫数据可批量推送、原始包留存、标准化处理和索引更新。
3. 控制台可完成数据源管理、作业查询、标签审核、资产查看、权限管理和审计查询。
4. API 可完成资产查询、检索、问答、作业查询、重处理触发和权限校验。
5. 权限与审计链路可运行，L4 数据默认脱敏，未授权内容不返回。
6. RAGFlow 集成链路完整，`index_manifest` 可回查。
7. 任一检索和问答结果都能追溯到资产版本、知识切片和原始对象。
8. 平台角色配置符合 v1.2 收敛规则，不再拆分平台管理与数据治理角色，不再单列授权业务访问角色。
9. 至少完成一次作业失败重试、索引失败重建和权限隔离测试。

### 15.2 关键假设

1. 一期试点数据以 D1-D4 为主，D5/D6 只做模型和适配器预留。
2. RAGFlow 可以满足一期切片、索引和检索执行需求。
3. 业务专家可在第 1 周完成分类、分级和标签规则确认。
4. 单节点或三节点环境可提供所需 GPU、对象存储和数据库资源。
5. 上层业务系统愿意通过 `nexus-api` 接入，不直接依赖 RAGFlow 原生接口。

### 15.3 产品风险与缓解

| 风险 | 表现 | 缓解策略 |
|------|------|----------|
| 范围膨胀 | 图谱、SFT、评价标准库被要求一期全部生产化 | 以 P0/P1/P2 控制范围，P2 不作为上线阻塞。 |
| 治理规则滞后 | 分类分级未确认导致数据无法发布 | 第 1 周设置规则确认 Go/No-Go。 |

| 权限模型复杂 | API 调用方、上层用户、组织范围混淆 | 强制所有调用携带调用方与用户上下文，统一由 NEXUS 求值。 |
| 运维能力不足 | 队列积压、索引失败无人处理 | P1 交付基础观测和 Runbook，上线前演练。 |
| 数据质量不稳定 | 文档格式复杂、扫描件质量差 | 质量评分、人工复核、解析策略升级和重处理。 |

## 十六、版本演进说明

### 16.1 v1.2 范围结论

v1.2 的核心定位是：

1. 夯实非结构化数据处理底座。
2. 打通接入、解析、标准化、索引和检索主链路。
3. 完成企业级治理与服务开放的最小可用闭环。
4. 明确 NEXUS、MinerU、RAGFlow 和上层业务系统边界。
5. 收敛角色模型，降低权限配置和验收复杂度。
6. 用 P0/P1/P2、业务 KPI、端到端用例和 Go/No-Go 标准提升需求可执行性。

### 16.2 后续版本方向

后续版本沿以下方向演进：
