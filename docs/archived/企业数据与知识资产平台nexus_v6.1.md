# 企业数据与知识资产平台 v6.2 — NEXUS

## 一、项目定位

NEXUS 面向企业现有非结构化数据，建设统一的数据沉淀、治理、复用与开放平台，解决企业数据分散、处理重复、管理粗放、使用门槛高的问题。

平台沉淀两类核心资产：

1. 面向业务应用的数据资产
2. 面向大模型和智能应用的知识资产

平台作为企业级数据与知识资产基础设施，统一支撑资产管理和能力开放两类核心场景。

---

## 二、产品结构

平台采用"一个统一数据资产底座 + 两类终端"的产品结构，并集成两个外部系统。

```
┌─────────────────────────────────────────────────────────────────┐
│                      外部系统集成层                               │
│   ┌─────────────────────┐   ┌─────────────────────────────┐    │
│   │   爬虫系统           │   │   RAGFlow 系统               │    │
│   │ 产业政策 / 人才需求  │   │ 检索 / 召回 / 知识组织        │    │
│   └──────────┬──────────┘   └──────────────┬──────────────┘    │
└──────────────┼───────────────────────────── ┼───────────────────┘
               ▼                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                    统一数据资产底座                                │
│                                                                  │
│  数据源接入 → 文档解析与清洗 → 分类分级与标签治理 →              │
│  权限与可见范围控制 → 检索、召回与知识组织                        │
│                                                                  │
└────────────────────────┬────────────────────────────────────────┘
                         │
          ┌──────────────┴───────────────┐
          ▼                              ▼
┌──────────────────┐          ┌──────────────────────┐
│  nexus-console   │          │     nexus-api         │
│  数据管理员终端   │          │  业务系统 API 终端     │
└──────────────────┘          └──────────────────────┘
```

### 外部系统集成

**爬虫系统**
定期爬取产业政策数据和行业人才需求数据，作为平台实时动态数据源的输入通道，通过标准数据接口推送至底座接入层。

**RAGFlow 系统**
提供检索、召回和知识组织能力，支撑平台统一的知识服务链路。底座完成文档解析与切片后，将结构化结果推送至 RAGFlow 构建检索索引，nexus-api 通过 RAGFlow 标准接口获取检索与问答结果。

## 三、统一数据资产底座

统一数据资产底座承载平台核心能力，是整个平台的数据处理与服务基础。

### 3.1 底座整体架构

底座按数据流向分为五个处理层级，各层级职责独立，数据单向流转：

```
原始数据源
    │
    ▼
[数据源接入层]
静态文件接入 / 爬虫动态接入 / 平台数据同步 / 接入状态管理
    │
    ▼
[文档解析与清洗层]
MinerU 解析引擎 / 结构识别 / 内容清洗 / 切片处理 / 元数据抽取
    │
    ▼
[资产治理层]
分类分级 / 标签体系 / 版本管理 / 生命周期管理
    │
    ▼
[权限与访问控制层]
角色权限 / 可见范围 / 鉴权校验 / 审计记录
    │
    ▼
[检索与知识服务层]（由 RAGFlow 支撑）
索引构建 / 检索召回 / 知识组织 / 结果输出
```

底座统一管理以下核心存储：

| 存储类型 | 用途 | 建议选型 |
|---------|------|---------|
| 关系型数据库 | 资产元数据、分类标签、权限配置、审计记录 | PostgreSQL |
| 对象存储 | 原始文件、解析结果文件 | MinIO（私有化）/ 云对象存储 |
| 向量数据库 | 文档切片向量索引（由 RAGFlow 管理） | RAGFlow 内置（基于 Elasticsearch + 向量引擎） |
| 缓存 | 热点检索结果、权限校验缓存 | Redis |
| 消息队列 | 文档处理任务异步调度 | Redis Streams / RabbitMQ |

### 3.2 数据源接入

平台统一接入三类数据源，覆盖外部知识文档、动态爬取数据和内部平台业务数据。

#### 静态知识文档（NAS / 手动上传）

| 数据域 | 典型数据内容 | 格式 | 接入方式 |
|--------|------------|------|---------|
| 产业行业数据 | 产业信息、行业发展报告 | PDF、Word | 文件上传 / 目录批量导入 |
| 岗位职业数据 | 职业能力分析表、技能等级证书、职业大典 | Excel、PDF | 文件上传 / 批量导入 |
| 专业建设数据 | 专业教学标准、人才培养方案、专业简介 | PDF、Word | 文件上传 / 目录批量导入 |
| 课程教材数据 | 教材、课程标准、课程大纲、教案、案例、习题、实训指导书 | Word、Excel、PDF | NAS 目录同步 / 批量导入 |
| 课程媒体资源 | 微课视频、动画、虚拟仿真软件 | MP4、应用包 | NAS 目录同步（仅元数据入库，文件原位存储） |

支持的文件格式：PDF、Word（.docx/.doc）、PPT（.pptx/.ppt）、Excel（.xlsx/.xls）、纯文本（.txt/.md）、图片（.jpg/.png，含扫描件）、视频（.mp4，仅元数据接入）。

#### 实时动态数据源（爬虫系统推送）

由爬虫系统按周期推送至底座，数据类型包括：

| 数据内容 | 更新周期 | 推送格式 |
|---------|---------|---------|
| 产业政策数据 | 每季度 | JSON（含原文、来源 URL、发布时间、地区） |
| 行业人才需求数据 | 每季度 | JSON（含岗位名称、技能要求、地区、薪资区间） |
| 岗位招聘数据 | 每季度 | Excel / JSON（250 万+ 条，按专业大类分批推送） |
| 院校专业布点数据 | 每年 | Excel（院校名称、专业名称、层次） |

推送接口约定：爬虫系统调用 `POST /ingest/batch` 接口，携带数据类型标识和分页批次号，底座完成接收后返回任务 ID 并异步处理。

#### 平台业务数据（I 博导 / 实训产品同步）

I 博导和实训教学产品的结构化业务数据通过数据库同步方式接入，不经过解析流程，直接写入底座元数据库：

| 系统 | 数据类型 | 同步方式 | 时效性 |
|------|---------|---------|-------|
| I 博导 | 用户信息、精品课 / 慕课 / 班级元数据 | 定时全量 + 增量同步 | T+1 |
| I 博导 | 学生学习行为、学业表现、学习路径 | 实时推送（Webhook） | 实时 |
| I 博导 | 教师教学行为、互动数据 | 实时推送（Webhook） | 实时 |
| 实训产品 | 用户信息、实训任务数据、学生操作行为 | 院校私有环境数据，按院校隔离同步 | T+1 |
| 实训产品 | 学生能力评估结果、职业匹配度 | 定时同步 | T+1 |

#### 接入状态管理

```
待处理 → 解析中 → 清洗中 → 治理中 → 已入库
                                        ↓
                               解析失败 / 清洗失败（人工介入）
```

### 3.3 文档解析与清洗

文档解析是整个底座数据质量的入口，采用 MinerU 作为核心解析引擎。结构化业务数据（I 博导、实训产品）不经过本流程，直接进入资产治理层。

#### 3.3.1 MinerU 解析引擎

MinerU 是面向 AI 场景的高精度文档解析工具，支持将 PDF、Word、图片等多格式文件转换为结构化的 Markdown 和 JSON 输出。

**核心解析能力：**

| 能力 | 说明 |
|------|------|
| 多格式解析 | 支持 PDF（文本型 / 扫描型）、Word、图片等输入 |
| 版面分析 | 识别正文、标题、表格、图片、公式等区域，还原阅读顺序 |
| 表格提取 | 将表格结构转换为 HTML 格式，保留行列关系 |
| 公式识别 | 将数学公式转换为 LaTeX 格式 |
| OCR 支持 | 支持扫描件和图片类文档，覆盖中英文及多语言场景 |
| 噪声过滤 | 自动去除页眉、页脚、页码、水印等无效内容 |
| 输出格式 | 结构化 Markdown（含标题层级）和 JSON（含元数据） |

**解析后端策略：**

| 场景 | 推荐后端 | 说明 |
|------|---------|------|
| 标准文本类 PDF（制度文件、标准规范） | Pipeline（CPU） | 速度快，适合大批量处理 |
| 复杂排版 / 图文混排（教材、研究报告） | Hybrid-engine | 结合原生文本提取与 VLM，减少幻觉 |
| 扫描件 / 低质量图片（历史材料、证书） | VLM-engine | 精度最高，需要 GPU 支持 |

生产环境建议：标准文档默认使用 Pipeline 后端，对解析质量抽样评分低于阈值的文档自动触发 Hybrid-engine 重新处理。

**MinerU 部署方式：**

以 FastAPI 服务形式独立部署，底座通过内部 HTTP 接口调用，支持任务排队和异步回调：

```
底座任务调度器 → POST /parse（文件 + 参数）→ MinerU 服务
                ← 异步回调（解析结果 JSON + Markdown）
```

#### 3.3.2 标准化处理流程

MinerU 完成解析后，进入底座标准清洗流程：

**步骤一：结构识别**
- 提取文档标题层级（H1/H2/H3）
- 识别段落、列表、表格、图片说明
- 标注章节边界，建立文档大纲

**步骤二：格式标准化**
- 统一字符编码（UTF-8）
- 统一时间格式（ISO 8601）
- 清理多余空白字符、异常符号

**步骤三：内容去噪**
- 去除模板语、声明语、版权页等无业务价值内容
- 识别并标记低质量段落（字符密度异常、乱码）

**步骤四：重复检测**
- 文档级：基于文件哈希检测完全重复文件
- 段落级：基于 MinHash / SimHash 检测高相似段落，标记但不强制删除

**步骤五：内容切片**
切片是检索质量的关键环节，采用以下分层切片策略：

| 切片类型 | 策略 | 适用数据 |
|---------|------|---------|
| 章节切片 | 按标题层级切分，保留上下文 | 制度文件、教学标准、职业大典 |
| 语义切片 | 按语义相关性聚合分段 | 行业研究报告、产业政策文件 |
| 段落切片 | 按自然段落边界切分 | 人才培养方案、专业简介 |
| 知识点切片 | 按课程知识图谱节点切分 | 教材、课程大纲、教案、习题 |
| 固定窗口切片 | 固定 token 数 + 滑动窗口重叠 | 无明显结构的历史材料 |

每个切片记录以下字段：切片 ID、原始文档 ID、切片内容、切片类型、所属章节路径、切片序号、字符数。

**步骤六：元数据抽取**
结合 MinerU 解析结果和内容语义分析，抽取以下元数据：

| 字段分组 | 字段 |
|---------|------|
| 来源信息 | 来源系统、上传部门 / 上传人、原始文件名、文件格式、来源 URL（爬虫数据） |
| 时间信息 | 创建时间、发布时间、采集时间、最后更新时间 |
| 内容信息 | 数据域（一级分类）、资产类型（二级分类）、关键词（自动提取）、所涉专业、所涉岗位 |
| 质量信息 | 解析质量评分、切片数量、有效字符数、是否含表格、是否含图片 |
| 业务信息 | 适用学历层次、适用地区、时效标签（初始值）、更新周期建议 |

**步骤七：质量校验**
每批处理完成后执行校验：
- 解析完整性：正文是否为空、标题是否提取成功
- 字段完整性：必填元数据字段是否缺失
- 切片合理性：切片数量是否在合理范围（过少可能解析失败，过多可能过度切分）
- 质量评分：综合以上指标生成 0-100 分的质量评分，低于 60 分标记为"待人工复核"

**步骤八：结果入库**
- 解析结果（Markdown / JSON）写入对象存储
- 切片结果写入关系型数据库（切片记录表）
- 向量化后的切片推送至 RAGFlow 构建索引
- 元数据写入资产元数据表
- 处理日志写入任务记录表

### 3.4 分类分级与标签治理

本节基于公司实际数据资产清单，建立与业务价值链对齐的分类体系、差异化分级规则和多维标签体系，形成可落地的治理方案。

#### 3.4.1 数据资产全景与业务价值链

公司数据资产沿以下业务价值链流转，各环节数据具有明确的依赖关系：

```
产业行业数据
    │ 确定区域产业需求与政策方向
    ▼
岗位职业数据
    │ 明确岗位工作任务与能力要求
    ▼
专业建设数据
    │ 将岗位能力转化为专业培养目标
    ▼
课程与教材数据
    │ 支撑课程内容开发与教学实施
    ▼
教学行为与实训数据
    │ 记录真实教学过程与学习轨迹
    ▼
能力评价与就业匹配
```

分类体系以此价值链为主轴设计，确保分类结构与业务使用逻辑一致。

---

#### 3.4.2 分类体系

采用"数据域（一级）→ 资产类型（二级）→ 具体数据集（三级）"三级分类结构。

```
数据资产目录
│
├── D1 产业行业知识域

├── D2 岗位职业知识域
│   ├── D2-1 岗位招聘数据
│   ├── D2-2 职业能力分析
│   ├── D2-3 技能等级证书
│   └── D2-4 国家职业大典
│
├── D3 专业建设知识域
│   ├── D3-1 国家专业教学标准
│   ├── D3-2 院校专业布点数据
│   ├── D3-3 人才需求调研报告
│   ├── D3-4 人才培养方案
│   └── D3-5 专业简介
│
├── D4 课程教学知识域
│   ├── D4-1 教材与参考资料
│   ├── D4-2 课程标准与大纲
│   ├── D4-3 教学设计与课件
│   ├── D4-4 教学案例
│   ├── D4-5 习题与试卷
│   ├── D4-6 实训指导书
│   └── D4-7 媒体资源（微课 / 动画 / 虚拟仿真）
│
├── D5 教学行为与平台数据域
│   ├── D5-1 用户与机构信息
│   ├── D5-2 课程与班级信息
│   ├── D5-3 学生学习行为数据
│   ├── D5-4 学生学业表现数据

    ├── D6-1 能力评估结果
    └── D6-2 职业匹配度数据

**分类维护规则：**
- 分类体系在第 1 周由业务专家与数据管理员联合制定，写入系统后作为治理基准
- 新增数据集默认归入最近匹配的二级分类，三级分类可按需扩展
- 一级、二级分类调整需经数据管理员审批，三级分类调整由上传部门提出、管理员确认

#### 3.4.3 数据分级规则

平台数据按开放程度和敏感性分为四级，各级对应不同的可见范围与访问控制策略。

| 级别 | 定义 | 典型数据 | 可见范围 | 访问控制 |
|------|------|---------|---------|---------|
| **L1 公开** | 官方发布、可公开引用的外部数据 | 国家职业大典、国家专业教学标准、政策文件 | 全员 + 授权外部用户 | 无需审批，直接访问 |
| **L2 内部** | 企业内部通用，不涉及个人或院校隐私 | 行业研究报告、岗位招聘数据、内部教材、课程资料 | 全员 | 登录后直接访问 |
| **L3 受限** | 特定部门或院校范围内使用，含院校私有数据 | 各院校人才培养方案、院校实训数据、院校专业布点 | 按院校 / 部门授权 | 需角色 / 院校权限，跨院校访问需审批 |
| **L4 机密** | 含个人身份信息或高度敏感行为数据 | 用户个人信息（姓名 / 手机号 / 邮箱）、学生学习行为明细、教师行为明细 | 指定角色（数据管理员 / 系统管理员） | 需明确授权，操作全程审计 |

**分级赋值说明：**
- 数据入库时，由系统根据数据域自动赋初始分级（见下表），管理员可在 nexus-console 中调整
- 个人数据字段（姓名、手机号等）无论所在数据集分级如何，在 API 输出时默认脱敏

| 数据域 | 默认初始分级 |
|--------|-----------|
| D1 产业行业知识域 | L1 / L2（官方发布 → L1，内部积累 → L2） |
| D2 岗位职业知识域 | L1 / L2（官方发布 → L1，招聘爬取 → L2） |
| D3 专业建设知识域 | L1 / L3（官方标准 → L1，院校私有方案 → L3） |
| D4 课程教学知识域 | L2 / L3（内部教材 → L2，院校私有资料 → L3） |
| D5 教学行为与平台数据域 | L3 / L4（课程信息 → L3，个人行为明细 → L4） |
| D6 能力评价与就业数据域 | L3 / L4（汇总报告 → L3，个人评估结果 → L4） |

#### 3.4.4 标签体系

标签是资产精确检索和场景化调用的核心工具，平台维护 7 个标签维度，每条资产可挂载多个标签。

**维度一：专业领域标签**（对应资产所涉专业方向）

| 标签值 | 说明 |
|--------|------|
| 电子商务 | 电商运营、直播带货、跨境电商等方向 |
| 大数据技术 | 数据分析、数据处理、BI 应用等方向 |
| 人工智能 | AI 应用开发、机器学习应用等方向 |
| 市场营销 | 品牌推广、数字营销等方向 |
| 物流管理 | 供应链、仓储物流等方向 |
| 通用 / 跨专业 | 不限专业的通用内容 |

#### 3.4.6 自动标签与人工校正机制

**自动标签规则**

文档解析完成后，系统基于以下规则自动推荐标签，推荐结果写入标签草稿区，等待管理员确认：

| 标签维度 | 自动识别规则 |
|---------|------------|
| 专业领域 | 文档标题 / 内容关键词匹配（如含"电子商务"→ 电子商务标签） |
| 学历层次 | 文档标题 / 来源路径关键词匹配（如含"高职"→ 高职专科标签） |
| 地域范围 | 内容中省市名称识别 |
| 数据来源 | 接入方式自动赋值（上传 → 内部积累；爬虫推送 → 爬虫采集；官方 URL → 官方发布） |
| 时效状态 | 入库时默认赋值"现行有效" |
| 更新周期 | 按数据域默认值自动赋值（见 3.4.5 速查表） |

**人工校正流程**

```
文档入库
    │
    ▼
自动标签生成（草稿状态）
    │
    ▼
管理员在 nexus-console 标签审核页查看推荐
    │
    ├── 确认：标签生效，资产进入可检索状态
    │
    ├── 修订：调整标签值后确认
    │
    └── 驳回：标记为"待复核"，返回人工重新处理
```

批量入库场景支持批量确认，管理员可按数据域或接入批次统一审核。

#### 3.4.7 数据资产与知识资产的关系模型

这是平台数据一致性的核心设计，明确两类资产的衍生关系：

```
原始文件（数据资产）
    │  1:N
    ▼
文档解析结果（解析版本）
    │  1:N
    ▼
内容切片（知识资产单元）——携带所属文档的分类、分级、标签
    │  N:1（聚合至 RAGFlow）
    ▼
检索索引（RAGFlow 管理）——按分级和标签构建独立索引分区
```

**版本联动机制**：
- 原始文件更新（重新上传或版本迭代）时，系统创建新的解析版本
- 新解析版本完成后，自动触发切片重建和 RAGFlow 索引更新
- 旧版本切片标记为"历史版本"，时效状态自动变更为"历史存档"，不参与检索，但保留供审计查询
- 版本对应关系在元数据中完整记录，确保任意检索结果均可追溯至原始文件版本

### 3.5 知识资产精细化加工

分类分级完成后，原始数据资产仅处于"可管理"状态，尚不能直接支撑 AI 应用和上层业务系统的使用需求。本节描述如何借助 RAGFlow 及 LLM 加工管道，将分类治理后的原始资产精细化为**可直接消费的知识数据资产**。

#### 3.5.1 加工定位：从数据资产到知识资产

原始数据资产经过分类分级治理后，进入知识资产加工层。加工层的目标是将非结构化、半结构化的原始内容，按不同应用场景所需的知识形态进行再加工：

```
原始数据资产（分类分级完成）
    │
    ▼
┌───────────────────────────────────────────────────┐
│              知识资产精细化加工层                    │
│                                                   │
│  RAGFlow（分块 / 向量化 / 检索）                   │
│       +                                           │
│  LLM 加工管道（抽取 / 生成 / 结构化）              │
│       +                                           │
│  人工校验（教学服务中心 / 教育研究院）               │
└───────────────────────────────────────────────────┘
    │
    ▼
知识数据资产（可直接消费）
├── RAG 检索知识库        → AI 助教、智能检索
├── 问答语料（SFT）        → 模型微调、客服替代
├── 流程语料（Agent）      → AI Agent 编排、自动生成
├── 知识图谱              → 课程推荐、专业规划
└── 评价标准库            → 自动评分、学情分析
```

#### 3.5.3 五类加工管道设计

##### 管道一：RAG 检索知识库构建（A 类资产）

适用资产：教材拆解知识块（#3）、人才培养方案结构化（#2）

**加工流程：**

```
原始文档（D4 教材 / D3 人培方案）
    │ MinerU 解析
    ▼
结构化 Markdown（含标题层级、表格）
    │ 知识点切片（章节切片 + 知识点切片）
    ▼
切片单元（携带：所属课程、知识点标签、学历层次、专业领域）
    │ 推送至 RAGFlow
    ▼
RAGFlow 向量化 + 索引构建
    │ 按分类（D4 分类树）+ 分级建立独立知识库分区
    ▼
可检索知识库（支持混合检索 + 权限过滤）
```

**RAGFlow 配置要点：**

| 配置项 | 推荐值 | 说明 |
|--------|-------|------|
| 分块模板 | 按标题层级（Manual） | 保留教材的知识层次结构 |
| 嵌入模型 | bge-large-zh-v1.5 | 中文教育场景适配好 |
| 检索模式 | 混合检索（融合 + RRF 重排） | 兼顾关键词精确和语义理解 |
| Top-K | 5-8 | 教材知识点密度适中 |
| 知识库分区 | 按 D4 二级分类 + 专业领域标签分区 | 避免不同专业知识互相污染检索结果 |

---

##### 管道二：问答语料生成（SFT 训练数据，A 类资产）

适用资产：教学问答语料（#1）、实训评分样例（#9）

**加工流程：**

```
原始文档（教材正文、教案、习题、实训指导书）
    │ MinerU 解析 + 知识点切片
    ▼
切片单元（知识表达段落）
    │ LLM 调用：基于切片内容生成问答对
    │ Prompt 模板：
    │   "根据以下教材内容，生成 3-5 个教学问答对，
    │    要求：问题贴近学生实际疑惑，答案准确、简洁"
    ▼
问答对草稿（JSON：{question, answer, source_chunk_id, course, topic}）
    │ 推送至 RAGFlow 知识库（用于检索验证）
    │ + 同步输出至人工校验队列
    ▼
教学服务中心人工校验
    │ 校验维度：准确性、教学适用性、表述规范性
    ▼
审核通过的问答对 → SFT 语料库（标准格式：instruction / input / output）
```

**质量控制：**

| 控制点 | 规则 |
|--------|------|
| 生成数量控制 | 每个知识点切片生成 3-5 对，避免过度生成重复 |
| 自动过滤 | 问题长度 < 10 字或答案 < 20 字的条目自动丢弃 |
| 人工抽检比例 | 首批 100%，稳定后按 20% 抽检 |
| 来源可追溯 | 每条语料保留 source_chunk_id，可回溯至原始教材段落 |

---

##### 管道三：流程语料结构化（Agent 编排语料，B 类资产）

适用资产：教材编写流程（#4）、课程标准编写流程（#6）、教学设计流程（#7）、电商项目全过程（#10）等

**加工流程：**

```
原始文档（教材样本、教案样本、实训指导书、电商案例）
    │ MinerU 解析（重点提取：步骤列表、流程说明、操作规程）
    ▼
流程片段识别（LLM 调用）
    │ Prompt 模板：
    │   "识别以下内容中的操作流程，提取为：

### 3.6 权限与可见范围控制

平台采用 RBAC（基于角色的访问控制）+ 资产分级过滤的复合权限模型。

#### 权限层级

| 权限层级 | 控制粒度 | 执行位置 |
|---------|---------|---------|
| 终端访问权限 | 用户是否可登录 nexus-console / 调用 nexus-api | 认证层 |
| 功能权限 | 用户在控制台可执行哪些操作 | 应用层 |
| 资产访问权限 | 用户可访问哪些数据集、知识库 | 数据层 |
| 内容可见范围 | 检索结果中哪些切片对该用户可见 | 检索层（RAGFlow 过滤） |

#### 权限配置维度

1. **按角色配置**：管理员 / 普通用户 / 只读用户 / API 调用方
2. **按院校隔离**：L3 级院校私有数据默认仅该院校授权用户可见，跨院校访问需审批
3. **按资产分级**：L3 / L4 级资产需显式授权，未授权用户在检索结果中不可见
4. **个人数据保护**：L4 数据的个人标识字段（姓名、手机号等）在 API 输出时默认脱敏，仅数据管理员可查看明文

#### 权限校验流程

```
请求进入（Console 操作 / API 调用）
    │
    ▼
身份认证（Token 校验）
    │
    ▼
功能权限校验（是否有操作权限）
    │
    ▼
资产权限校验（是否有目标资产的访问权 + 院校隔离校验）
    │
    ▼
结果过滤（检索结果按分级 + 可见范围过滤；L4 字段脱敏处理）
    │
    ▼
审计记录（操作人、时间、访问资产、结果条数）
```

### 3.7 检索、召回与知识组织

本模块由 RAGFlow 系统承载，底座负责数据准备和接入协调。

#### RAGFlow 在底座中的角色

RAGFlow 作为独立部署的知识服务系统，与底座的交互关系如下：

| 方向 | 内容 |
|------|------|
| 底座 → RAGFlow | 切片内容 + 元数据（含分类、分级、标签）推送，触发索引构建 |
| 底座 → RAGFlow | 权限过滤条件（用户分级、院校 ID），在检索时传入 |
| RAGFlow → 底座 | 检索结果（含切片 ID、来源文档、相关度评分） |
| nexus-api → RAGFlow | 检索召回请求、问答请求（通过底座服务层转发） |

#### 检索能力

RAGFlow 提供三种检索模式，可按场景配置：

| 检索模式 | 原理 | 适用场景 |
|---------|------|---------|
| 关键词检索 | 基于全文索引（Elasticsearch） | 精确词汇查找、编号查询 |
| 语义检索 | 基于向量相似度（嵌入模型） | 意图理解、相似内容发现 |
| 混合检索 | 关键词 + 语义融合，RRF 重排 | 通用业务问答，推荐默认配置 |

#### 召回与重排

1. 初始召回：按检索模式返回 Top-K 候选切片
2. 过滤：按权限可见范围、数据分级、分类、标签、时间过滤
3. 重排：基于 Re-ranker 模型对候选结果重新排序
4. 知识组织：将相关切片聚合，构建结构化上下文供问答使用

#### 知识资产质量基线

建立以下评估指标，在阶段 1 完成后建立基线，作为后续优化的参照：

| 指标 | 定义 | 目标基线 |
|------|------|---------|
| 解析成功率 | 成功完成解析的文档占比 | ≥ 95% |
| 切片质量评分 | 平均切片质量评分 | ≥ 70 分 |
| 检索召回率 | 试点场景测试集的召回率 | ≥ 80%（Top-5） |
| 答案可追溯率 | 问答结果中有明确来源引用的比例 | 100% |

## 四、两类终端职责

### nexus-console

面向数据管理员，承担资产管理与治理入口职责。

**核心功能模块：**

| 模块 | 主要功能 |
|------|---------|
| 数据源管理 | 静态数据源上传 / 批量导入；爬虫数据源配置；平台数据同步配置；接入状态查看 |
| 解析清洗管理 | 解析后端配置；清洗规则配置；处理任务监控；失败任务重处理 |
| 分类分级管理 | 分类体系维护（D1-D6）；分级规则配置；标签审核（草稿确认 / 修订 / 驳回）；批量标注 |
| 资产目录管理 | 资产列表查看；元数据编辑；版本历史查看；时效状态管理 |
| 权限管理 | 角色管理；用户授权；院校隔离配置；审批流配置 |
| 运营与审计 | 处理任务统计；访问统计；调用分析；操作审计日志；L4 数据访问专项审计 |

### nexus-api

面向上层业务系统，以标准化接口开放底座能力。

**API 接口分类：**

API 接口按调用性质分为两类，采用不同的架构设计：

**同步查询类（低延迟，有 SLA 要求）：**

| 接口 | 说明 | 目标响应时间 |
|------|------|------------|
| `GET /assets` | 资产列表查询，支持分类（D1-D6）、标签、时间过滤 | < 200ms |
| `GET /assets/{id}` | 资产详情及元数据查询 | < 100ms |
| `POST /search` | 检索召回（关键词 / 语义 / 混合），权限过滤自动生效 | < 1s |
| `POST /qa` | 知识问答（RAGFlow 驱动） | < 5s |
| `POST /auth/verify` | 权限校验 | < 50ms |

**异步处理类（高吞吐，队列调度）：**

| 接口 | 说明 |
|------|------|
| `POST /ingest` | 提交文档解析入库任务，返回任务 ID |
| `POST /ingest/batch` | 爬虫系统批量推送接口 |
| `GET /tasks/{id}` | 查询任务处理状态和结果 |
| `POST /tasks/{id}/retry` | 重新触发失败任务 |

**通用能力：**
- 鉴权：所有接口需携带 API Key 或 JWT Token
- 限流：按调用方独立限流（默认 100 QPS / 调用方）
- 审计：每次调用记录调用方、接口、参数摘要、响应时间、结果条数
- 监控：接口可用性、响应时间 P99 纳入运营统计面板

## 七、架构评审意见

> 评审视角：资深数据资产架构师
> 评审版本：v6.1 → v6.2 调整依据

### 总体评价

v6.1 在数据治理体系设计上具有较高完整度，分类体系、标签维度和检索架构的设计思路清晰，但**交付边界模糊、工作量估算严重失真**，按 v6.1 计划执行存在较高的里程碑失守风险。

---

### 显著亮点

**1. 五层处理架构分层清晰**
接入层 → 解析层 → 治理层 → 权限层 → 服务层的单向流转设计符合数据资产平台工程规范，各层职责边界明确，后续扩展不需要推倒重来。

**2. D1-D6 六域 + 7 维标签治理体系完整**
分类体系覆盖了从原始文档到结构化知识的全域，7 维标签（来源、专业领域、学历层次、地域、时效、应用场景、更新周期）能够支撑后续精细化权限控制和检索过滤，是方案中设计最扎实的部分，且直接与真实数据清单对齐。

**3. RAGFlow 选型务实**
RAGFlow 原生支持多格式文档解析和向量检索，与 MinerU 在解析层形成互补，避免了自研检索引擎的高成本，对当前团队规模是合理选择。

**4. 双终端分离（nexus-console / nexus-api）**
管理面与服务面分离，治理操作与业务调用不共用接口，避免了后期权限管理和流量隔离的混乱，是正确的架构决策。

---

### 核心问题与风险

**问题 1：3.5 节"知识资产精细化加工"被置于底座章节，交付边界严重模糊**

16 项知识资产中，知识图谱（#13、#14、#15）依赖多源数据融合、实体对齐和人工审核标注，问答语料（#1）、流程语料（#4-#8）依赖 LLM 加工管道稳定运行，这些能力的交付周期以"月"为单位而非"周"。将其写入底座章节会让评审方误认为一期 6 周内须全部就绪，实际上一期能完成的只有管道一（基础文档分片入库）的 MVP。

**问题 2：LLM 加工管道缺少前置依赖说明，是隐藏的阻塞风险**

管道二（问答语料生成）和管道三（流程语料结构化）均依赖 LLM 推理服务。方案中没有关于 LLM 服务部署方式（私有化部署还是 API 接入）、模型选型、token 成本预算、企业网络访问策略的任何说明。如果 LLM 服务未就绪，管道二、三全部无法启动，而原计划对此没有 Go/No-Go 节点保护。

**问题 3：第 2 周将"解析管道打通"与"I 博导实时 Webhook 同步"并列，工作量严重超载**

MinerU 接入需要完成文件上传 → 解析触发 → 结果回写 → 质量校验的完整链路，任何一环的调试都可能消耗数天。I 博导 Webhook 同步则是完全独立的实时数据接入通道，需要鉴权设计、幂等处理和失败重试机制。两件事任意单独做都是合理的一周工作量，并列在同一周没有可行性支撑。

**问题 4：第 3 周四项并行，是计划中最危险的一周**

分类治理配置、RBAC 权限模型、RAGFlow 对接、版本联动机制四件事存在明显的前置依赖关系：RBAC 未完成则 Console 页面功能边界无法确定；RAGFlow 对接需向量化任务稳定后才能验证效果；版本联动依赖数据模型收敛。四件事压缩在一周内，实际上是把风险推迟到第 4 周集中爆发。

**问题 5：250 万条岗位招聘数据（D2-1）没有独立接入方案**

250 万条结构化数据的接入涉及批量写入性能（与单文档解析完全不同的量级）、字段映射与清洗规则、去重和增量同步策略、存储层容量规划。方案中将其与普通文档混同处理，没有任何针对性说明。按原计划执行，D2-1 极可能在第 2 周成为未预期的技术阻塞点。

**问题 6：知识图谱（#13、#14、#15）标注为 ⭐⭐⭐⭐⭐ 最高优先级但无法在 6 周内交付**

知识图谱需要实体识别、关系抽取、多源融合、图数据库选型与部署、人工审核流程设计，任何单项都不是 6 周内能稳定交付的内容。最高优先级标注会误导资源分配，导致团队在一期就对图谱投入时间，而真正应优先交付的底座能力（稳定的文档解析链路、可用的 RAG 检索）反而可能受到挤压。
