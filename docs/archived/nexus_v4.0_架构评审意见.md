# NEXUS v4.0 架构评审意见

> 评审视角：企业数据架构师
> 评审对象：企业数据与知识资产平台 v4.0 方案文档

---

## 总体评价

**方向正确，定位清晰，但工程深度不足，现阶段更接近产品规划文档，尚不足以驱动工程落地。**

"统一底座 + 三类终端"的架构结构合理，用户分层清晰，一期最小闭环的设计思路值得肯定。主要问题集中在：底座技术决策缺失、交付周期被严重压缩、数据治理的人与规则未纳入计划、以及两类核心资产的关系模型不清晰。

---

## 亮点

**1. 架构分层意识正确**
能力下沉至统一底座，终端只做适配，符合平台化演进原则，从根本上避免了终端膨胀和重复建设。

**2. 用户分层边界清晰**
三类用户（管理员 / 业务人员 / 系统集成方）与三类终端的映射关系对应明确，没有犯"大而全统一界面"的常见错误。

**3. 资产分类有前瞻性**
早期就区分"面向业务应用的数据资产"与"面向大模型的知识资产"，两者在处理管道、索引策略和消费方式上本质不同，这个区分越早做越好。

**4. 一期有闭环交付意识**
"资产沉淀、业务可用、系统可接"三件事同步交付，优于"先建底座、再谈应用"的割裂式推进，能有效规避平台建完无人使用的风险。

---

## 主要问题

### 问题一：统一底座是整个方案的最大风险点

底座是平台的核心，但方案对其描述最为薄弱，仅列出 8 项能力清单，技术实现路径、组件选型、数据模型均未涉及。

| 能力项 | 未决策的关键问题 |
|--------|----------------|
| 检索与召回 | 向量检索、关键词检索还是混合检索？影响知识质量的核心决策 |
| 文档解析 | 对 PDF / Word / 扫描件的处理方案？解析质量是全平台数据质量的入口 |
| 权限控制 | 行级 / 资产级 / 部门级的复合权限模型？模型不统一则三端权限行为不一致 |

**结论**：底座在进入阶段 1 之前，必须有一份独立的技术架构文档，明确核心组件选型、数据模型和 API 契约。

---

### 问题二：阶段 1 交付边界过宽，工程可行性存疑

阶段 1 计划在 3-4 周内，由 2 名开发同时完成底座框架 + Console 基础版 + Chat MVP + API 基础版，这在工程上难以保障质量。

常见的两类失败模式：
- 底座"框架"建成，但数据管道实际跑不通
- 三个终端各交付了界面，底层能力未形成真正闭环

**建议调整**：阶段 1 聚焦底座核心管道 + Console 数据接入，Chat 和 API 推至阶段 2，先用真实数据验证底座稳定性，再扩展终端。

---

### 问题三：数据治理中"人"与"规则"的缺失

方案列举了分类分级、标签、版本、生命周期等治理功能，但企业数据治理的真正难点不在系统功能，而在于：

- 谁来定标签规则？
- 分类体系如何与各业务部门对齐？
- 数据所有权归属争议如何解决？

方案中"业务专家 1 名"承担的工作量严重低估了治理协调的实际复杂度。若治理规则没有在阶段 0 确定，阶段 1 的数据入库将是无序积累，后续治理代价成倍放大。

---

### 问题四：数据资产与知识资产的关系模型未定义

两类资产在生命周期管理上存在本质差异：数据资产管的是原始文件，知识资产管的是经过切片处理后的碎片。两者之间缺少"衍生关系链路"设计，会导致：

> 原始文档已更新，但知识库中仍在使用旧版本内容进行问答。

这是知识类平台最常见的数据一致性问题，必须在底座设计阶段解决。

---

### 问题五：nexus-chat 的质量风险未做预期管理

Chat 终端的可用性高度依赖检索质量和大模型表现，方案将"答案可追溯"列为基础能力，但在 RAG 场景下这是有一定工程复杂度的问题。若一期 Chat 答案质量不稳定，业务用户信任难以建立，将直接影响平台从"后台系统"向"业务平台"的转型目标。

---

## 优化建议

### 建议一（P0）：补充底座技术架构文档

在进入阶段 0 之前，输出一份覆盖以下内容的技术决策文档：
- 向量数据库 / 检索引擎选型
- 文档解析方案（含扫描件处理策略）
- 权限模型设计（与企业现有 IdP 的集成方式）
- 数据资产与知识资产的关系模型及版本联动机制

---

### 建议二（P0）：重新规划阶段 1 交付边界

| 调整前 | 调整后 |
|--------|--------|
| 底座 + Console + Chat + API 同步交付 | 底座核心管道 + Console 数据接入 + API 基础 |
| Chat MVP 同步上线 | Chat MVP 推至阶段 2 |
| 3-4 周内形成三端闭环 | 3-4 周内验证底座可靠性和 1 个试点场景 |

---

### 建议三（P1）：将治理规则制定纳入阶段 0 强制输出

阶段 0 增加以下交付物：
- 分类分级规则（初版）
- 标签体系框架
- 数据所有权责任矩阵（明确各类数据的归属部门）

这三项不完成，阶段 1 的数据接入工作缺乏基础。

---

### 建议四（P1）：建立数据质量基线评估机制

当前方案无任何质量度量机制。建议在阶段 1 同步建立：
- 文档解析准确率的抽样基线
- 基于试点场景构造的检索召回评估集

有了基线，阶段 2 的优化工作才有明确方向，平台质量演进才可量化。

---

### 建议五（P2）：nexus-chat 一期限定场景，建立信任再扩展

一期 Chat 不建议做"全知识库通问"，建议：
- 限定 1-2 个具体场景（如制度查询、交付物检索）
- 在窄场景内建立答案质量和可追溯性的信任基础
- 二期再基于用户反馈扩展覆盖范围

---

### 建议六（P2）：nexus-api 区分同步与异步两类接口

当前 API 端混合了性质不同的两类请求：

| 类型 | 典型接口 | 设计诉求 |
|------|---------|---------|
| 同步查询类 | 检索、问答、资产查询 | 低延迟、有 SLA 约束 |
| 异步处理类 | 文档解析入库 | 高吞吐、适合队列异步处理 |

两类接口混为一类会导致 API 设计和运维策略冲突，建议在 API 设计阶段就做架构分离。

---

## 行动建议

在正式进入阶段 0 前，建议先完成以下两件事：

1. **技术团队输出底座技术架构文档**，覆盖选型决策和核心数据模型，这是后续所有工作量估算和风险评估的基础。

2. **与业务部门确认治理规则的制定责任人**，数据治理的人的问题不解决，系统功能再完善也无法落地。

这两件事不需要额外资源，但如果跳过它们直接进入建设，阶段 1 大概率会在"底座功能可演示但数据管道不稳定"的状态下结束，并将问题传导至后续每个阶段。
