## 一、结论：Spec 是否足以直接指导 Prototype

《企业数据与知识资产平台需求 Spec v1.2 — NEXUS》已经具备原型设计的上游输入价值，能够明确产品范围、角色、P0/P1/P2 优先级、控制台信息架构、用户旅程、功能需求和验收口径。

但 Spec 不能直接替代 Prototype 设计文档，原因如下：

| 缺口 | Spec v1.2 已覆盖 | Prototype 仍需补充 |
|------|-----------------|------------------|
| 页面结构 | 已定义一级菜单和能力范围 | 需要明确页面清单、布局、主次区域、列表/详情/抽屉/弹窗关系。 |
| 操作流 | 已定义用户旅程和业务场景 | 需要明确每一步页面跳转、按钮、状态变化和异常分支。 |
| 字段展示 | 已定义数据对象和元数据要求 | 需要明确每个页面展示字段、筛选字段、表单字段和字段优先级。 |
| 状态表达 | 已定义作业、索引、资产状态 | 需要明确状态标签、颜色语义、可点击动作和禁用条件。 |
| 交互细节 | 已定义功能需求 | 需要明确批量操作、确认弹窗、表单校验、空态、错误态、加载态。 |
| 原型验收 | 已定义系统验收 | 需要定义原型评审检查项、页面覆盖率和端到端可演示路径。 |

因此，原型设计应以 Spec v1.2 为需求基线，以本文档作为页面级设计输入。

## 二、Prototype v1.0 目标与范围

### 2.1 原型目标

Prototype v1.0 的目标不是呈现最终视觉 UI，而是验证产品信息架构、核心流程、页面字段、状态表达和角色权限是否能够支撑一期 P0 上线闭环。

原型需要回答以下问题：

1. 平台/数据管理员能否完成数据源配置、接入、作业追踪、资产治理、发布和权限配置。
2. 业务专家能否完成分类标签确认、质量复核、检索试测和知识资产审核。
3. API 调用方能否完成 API Key 申请、联调、调用记录查看和问题反馈。
4. 运维人员能否看到作业积压、索引失败、容量风险和告警处理入口。
5. 任一检索、问答、作业、资产、索引和审计结果是否具备可追溯入口。

### 2.2 原型范围

Prototype v1.0 覆盖 P0 页面完整闭环，并为 P1 页面提供基础框架。

| 范围 | 页面 / 能力 | 原型要求 |
|------|-------------|----------|
| P0 | 工作台、数据源管理、原始数据台账、作业中心、资产目录、治理中心、权限与审计 | 必须绘制页面结构、关键字段、核心操作和状态流。 |
| P0 | 静态文档入库、爬虫批量接入、资产治理发布、权限隔离、重处理 | 必须能串成端到端演示路径。 |
| P1 | 检索测试、知识资产、运维观测、API Key 运营 | 绘制基础框架和主要交互入口。 |
| P2 | D5/D6 正式接入、知识图谱生产化、SFT 语料量产、评价标准库 | 不进入 Prototype v1.0，可在导航或空态中标注“后续版本”。 |

### 2.3 原型保真度

| 阶段 | 保真度 | 目标 |
|------|--------|------|
| 第一轮 | 低保真线框 | 验证菜单、页面结构、核心流程和字段。 |
| 第二轮 | 中保真交互 | 验证筛选、表单、状态、弹窗、抽屉和端到端路径。 |
| 第三轮 | 视觉样稿 | 仅对工作台、资产目录、作业中心、资产详情、治理中心、检索测试做视觉深化。 |

## 三、用户角色与原型入口

### 3.1 原型角色

| 角色 | 原型入口 | 主要关注页面 |
|------|----------|--------------|
| 平台/数据管理员 | `nexus-console` | 工作台、数据源管理、原始数据台账、作业中心、资产目录、治理中心、权限与审计。 |
| 业务专家 | `nexus-console` | 工作台、治理中心、质量复核、检索测试、知识资产。 |
| 运维人员 | `nexus-console` | 工作台、作业中心、运维观测、审计日志、告警详情。 |
| API 调用方 | `nexus-console` 的 API 管理视图或上层系统联调入口 | API Key、接口权限、调用记录、作业查询、检索测试。 |

### 3.2 权限差异在原型中的体现

| 能力 | 平台/数据管理员 | 业务专家 | 运维人员 | API 调用方 |
|------|----------------|---------|---------|-----------|
| 数据源配置 | 可见可编辑 | 只读或不可见 | 不可见 | 不可见 |
| 标签审核 | 可确认 / 修订 / 驳回 | 可建议 / 修订 / 审核 | 不可见 | 不可见 |
| 作业重试 | 可操作 | 可申请 | 可操作 | 按授权可操作 |
| 权限配置 | 可操作 | 不可操作 | 只读审计 | 仅查看自身权限 |
| 审计日志 | 全量按权限查看 | 查看本人审核记录 | 查看运维相关日志 | 查看自身调用记录 |
| 运维观测 | 可查看 | 不可见 | 可查看可处理 | 不可见 |

原型不需要绘制四套完全不同 UI，但需要在关键按钮上标注角色可见性和禁用规则。

### 4.3 全局组件

| 组件 | 用途 | 原型要求 |
|------|------|----------|
| 状态标签 | 表示资产、作业、索引、审核、分级状态 | 必须统一颜色和文案。 |
| 筛选栏 | 数据源、批次、作业、资产、审计通用筛选 | 支持收起 / 展开高级筛选。 |
| 数据表格 | 列表主形态 | 支持列配置、分页、批量选择、排序。 |
| 详情抽屉 | 快速查看对象详情 | 不离开列表上下文，适合原始对象、作业、标签草稿。 |
| 详情页 | 承载复杂对象 | 资产详情、数据源详情、知识资产详情使用详情页。 |
| 操作确认弹窗 | 高风险操作确认 | 重处理、下线、发布、删除 API Key、批量驳回需确认。 |
| 审计时间线 | 展示操作历史 | 资产、作业、权限、API Key 都需可复用。 |
| 来源引用卡片 | 展示文档、版本、切片、页码 | 检索测试、问答结果、知识资产详情复用。 |
| 空态 | 无数据引导 | 提供创建入口或说明权限不足。 |
| 错误态 | 作业、索引、API 错误 | 显示错误码、原因、建议操作、请求 ID。 |

### 4.4 状态标签规范

| 类型 | 状态 | 展示文案 | 原型语义 |
|------|------|----------|----------|
| 接入 | `registered` | 已登记 | 灰色，等待校验。 |
| 接入 | `raw_persisted` | 已落原始库 | 蓝色，可进入作业。 |
| 接入 | `failed` | 接入失败 | 红色，可查看原因。 |
| 作业 | `running` | 处理中 | 蓝色，显示进度。 |
| 作业 | `retryable_failed` | 可重试失败 | 橙色，显示重试按钮。 |
| 作业 | `manual_review` | 待人工复核 | 紫色，进入复核队列。 |
| 资产版本 | `published` | 现行有效 | 绿色，可检索。 |
| 资产版本 | `pending_review` | 待复核 | 橙色，不可发布。 |
| 资产版本 | `archived` | 历史存档 | 灰色，默认不参与检索。 |
| 索引 | `indexed` | 已索引 | 绿色。 |
| 索引 | `stale` | 待重建 | 橙色，提示权限或治理变更。 |
| 索引 | `failed` | 索引失败 | 红色，可重试。 |
| 分级 | `L1` | 公开 | 绿色。 |
| 分级 | `L2` | 内部 | 蓝色。 |
| 分级 | `L3` | 受限 | 橙色。 |
| 分级 | `L4` | 机密 | 红色，强调脱敏和审计。 |

## 五、核心端到端原型流程

### 5.1 流程 A：静态文档入库到可检索

```
数据源管理
  → 新建 / 选择上传入口
  → 上传文件 / 批量上传
  → 生成接入批次
  → 原始数据台账查看 raw_object
  → 作业中心查看解析 / 标准化 / 索引进度
  → 资产目录查看资产详情
  → 治理中心确认分类、分级、标签
  → 发布资产版本
  → 检索测试验证召回与引用
```

关键原型校验：

1. 每一步都能看到对象 ID 和状态。
2. 失败时能回到作业中心查看原因。
3. 发布前必须检查治理必填字段。
4. 发布后可看到索引状态和检索测试入口。

### 5.2 流程 B：爬虫批量推送到资产化

```
数据源管理
  → 爬虫推送配置
  → 查看批次推送记录
  → 原始数据台账查看原始 JSON 包
  → 作业中心查看 structured_sync / normalize_record / index_build
  → 资产目录按 D1/D2 过滤
  → 治理中心批量确认标签
  → 检索测试验证动态数据结果
```

关键原型校验：

1. 爬虫批次必须展示来源系统、抓取时间、规则版本、原始 URL。
2. JSON 原始包和标准化记录要能切换查看。
3. 批量标签确认需有确认弹窗和审计记录。

### 5.3 流程 C：资产治理与发布

```
治理中心
  → 待审核标签列表
  → 打开标签审核抽屉
  → 查看自动标签依据
  → 确认 / 修订 / 驳回
  → 质量复核
  → 发布资产版本
  → index_manifest 更新
```

关键原型校验：

1. 标签草稿必须显示生成依据和置信度。
2. 驳回需填写原因。
3. L3/L4 资产发布前必须确认组织范围。
4. 发布动作需二次确认。

### 5.4 流程 D：权限隔离与 API 调用方联调

```
权限与审计
  → 新建 API Key
  → 配置接口范围、数据域、组织范围、分级上限、QPS
  → API 调用方联调 search / qa
  → 检索测试模拟调用方权限
  → 审计日志查看调用记录和脱敏动作
```

关键原型校验：

1. API Key 创建后只展示一次明文。
2. 权限范围必须可读、可复核。
3. L4 数据默认不返回明文字段。
4. 审计日志能按调用方、接口、请求 ID 查询。

### 5.5 流程 E：失败作业重试与重处理

```
工作台告警 / 作业中心
  → 查看失败作业
  → 打开作业详情
  → 查看阶段、错误、关联对象、日志摘要
  → 点击重试 / 触发重处理
  → 查看新作业状态
  → 资产详情查看新版本与旧版本关系
```

关键原型校验：

1. 失败原因必须可见。
2. 重试与重处理要区分。
3. 高风险重处理需确认影响范围。
4. 新旧版本关系必须可追溯。

## 六、页面级 Prototype 说明

### 6.1 NX-00 登录与入口

优先级：P0

目标用户：平台/数据管理员、业务专家、运维人员。

页面目的：通过企业 SSO 或本地登录进入 `nexus-console`。

页面结构：

```
┌──────────────────────────────┐
│ NEXUS                         │
│ 企业数据与知识资产平台          │
│                                │
│ [ 使用企业 SSO 登录 ]           │
│ [ API 调用方联调入口 ]          │
│                                │
│ 环境：试点 / 生产               │
└──────────────────────────────┘
```

关键交互：

| 操作 | 结果 |
|------|------|
| 使用企业 SSO 登录 | 进入工作台。 |
| 无权限用户登录 | 显示无权限空态和申请入口。 |
| API 调用方联调入口 | 进入受限的 API 文档 / 调用记录视图。 |

### 6.2 NX-01 工作台

优先级：P0

目标用户：全部控制台角色。

页面目的：提供平台运行总览和待处理入口。

低保真布局：

```
┌ 工作台 ─────────────────────────────────────────────┐
│ 今日概览：接入批次 / 成功率 / 失败作业 / 待复核 / 告警 │
├─────────────────────────────────────────────────────┤
│ [接入趋势图]        [作业状态分布]       [索引状态]     │
├─────────────────────────────────────────────────────┤
│ 待处理事项                                            │
│ - 待标签审核  23  [去处理]                            │
│ - 索引失败    5   [去处理]                            │
│ - L4 访问异常 1   [查看审计]                          │
├─────────────────────────────────────────────────────┤
│ 最近批次 / 最近失败作业 / 最近发布资产                 │
└─────────────────────────────────────────────────────┘
```

核心字段：

| 模块 | 字段 |
|------|------|
| 今日概览 | 接入批次数、原始对象数、标准化成功率、解析成功率、索引成功率、权限拒绝次数。 |
| 待处理事项 | 类型、数量、最高优先级、最早产生时间、处理入口。 |
| 最近批次 | 批次号、来源、对象数、成功数、失败数、创建时间、状态。 |
| 最近失败作业 | 作业 ID、作业类型、关联资产、失败阶段、失败原因、重试次数。 |

关键交互：

| 操作 | 结果 |
|------|------|
| 点击待标签审核 | 跳转治理中心并带入筛选 `待审核`。 |
| 点击索引失败 | 跳转作业中心并筛选 `index_build failed`。 |
| 点击 L4 访问异常 | 跳转审计日志并筛选高敏访问。 |
| 点击批次 | 打开批次详情。 |

空态 / 错误态：

| 状态 | 展示 |
|------|------|
| 无待处理 | 显示“当前无待处理事项”。 |
| 指标加载失败 | 显示局部错误卡片和刷新按钮。 |

### 6.3 NX-02 数据源管理

优先级：P0

目标用户：平台/数据管理员。

页面目的：管理上传入口、NAS 同步、爬虫推送和预留同步源。

低保真布局：

```
┌ 数据源管理 ─────────────────────────────── [新建数据源] ┐
│ Tab：全部 / 文件上传 / NAS / 爬虫 / Webhook / 数据库预留 │

│ 表格：                                                   │
│ 名称 | 类型 | 业务域 | 默认分类 | 默认分级 | 最近同步 | 状态 | 操作 │
└─────────────────────────────────────────────────────────┘
```

列表字段：

| 字段 | 说明 |
|------|------|
| 数据源名称 | 可点击进入详情。 |
| 来源类型 | 文件上传、NAS、爬虫、Webhook、数据库。 |

| 默认分类 | 一级 / 二级分类。 |

| 原文片段 | 根据权限展示，L4 默认脱敏。 |
| 原始对象 | 有权限时可查看元信息。 |

## 八、表单校验与交互规则

### 8.1 通用表单校验

| 字段类型 | 校验规则 |
|----------|----------|
| 名称 | 必填，最长 100 字，不允许仅空格。 |
| 描述 | 最长 500 字。 |
| 数据域 | 必填，D1-D6 枚举。 |
| 分级 | 必填，L1-L4 枚举。 |
| 组织范围 | L3/L4 必填。 |
| API Key 额度 | 正整数，需有默认值。 |
| URL | 必须符合 URL 格式。 |
| 文件 | 必须符合格式白名单和大小限制。 |
| 驳回原因 | 必填，最长 500 字。 |

### 8.2 批量操作规则

| 操作 | 规则 |
|------|------|
| 批量确认标签 | 仅允许同数据域、同标签维度批量处理。 |
| 批量下线资产 | 必须二次确认，并展示影响检索范围。 |
| 批量重建索引 | 必须展示预计对象数和影响范围。 |
| 批量重试作业 | 仅支持可重试失败状态。 |

### 8.3 高风险操作规则

以下操作必须二次确认：

1. 发布资产。
2. 下线资产。
3. 禁用 API Key。
4. 触发重处理。
5. 批量驳回标签。
6. 重建索引。
7. 导出审计日志。

### 9.1 空态

| 表单校验失败 | 字段下方显示错误原因。 |

## 十一、原型评审口径

### 11.1 页面覆盖检查

| 检查项 | 通过标准 |
|--------|----------|
| P0 页面覆盖 | NX-00 至 NX-09 必须完整覆盖。 |
| P0 弹窗 / 抽屉覆盖 | 发布、重处理、标签审核、作业详情、来源引用必须覆盖。 |
| P1 页面框架 | 检索测试、知识资产、运维观测需具备基础结构。 |
| 角色权限标注 | 关键按钮需标注可见角色或禁用条件。 |
| 状态表达 | 作业、资产、索引、分级状态需统一表达。 |

### 11.2 端到端流程检查

| 流程 | 通过标准 |
|------|----------|
| 静态文档入库 | 能从上传走到资产详情和检索测试。 |
| 爬虫批量接入 | 能从数据源配置走到批次和标准化记录。 |
| 资产治理发布 | 能从标签审核走到发布和索引更新。 |
| 权限隔离 | 能创建 API Key 并模拟权限过滤结果。 |
| 失败重试 | 能从工作台告警走到作业详情并重试。 |

### 11.3 原型评审问题清单

评审时必须确认以下问题：

1. 平台/数据管理员是否能理解从“接入”到“发布”的主路径。
2. 业务专家是否能快速定位待复核、标签草稿和检索问题。
3. 运维人员是否能定位失败作业、索引失败和告警详情。
4. API 调用方权限范围是否表达清楚。
5. 资产详情是否足够支撑可追溯要求。
6. L3/L4 权限和脱敏是否在界面中有明确提示。
7. 批量操作和高风险操作是否有足够防误触设计。

## 十三、后续原型迭代建议

Prototype v1.0 完成后建议按以下顺序迭代：

1. 先评审 P0 端到端流程，不先纠结视觉风格。
2. 第二轮补齐字段级交互、批量操作和异常态。
3. 第三轮选择工作台、作业中心、资产目录、资产详情、治理中心、检索测试做中高保真视觉稿。
4. API 调用方视图应在接口文档确定后补充更细的联调台原型。
5. 运维观测页应在实际监控指标确定后再深化图表。
