# 企业数据与知识资产平台 v3.0 调整方案 - NEXUS

## 一、调整说明

基于现有产品构思，NEXUS 不再仅定义为“管理控制台 + API 服务”的双层产品，而应升级为一套面向不同业务角色的三终端统一平台。平台围绕企业非结构化数据的沉淀、治理、复用与开放，统一形成数据资产与知识资产底座，并通过不同终端将能力交付给不同类型用户。

建议形成以下三类产品终端：

1. `nexus-console`：面向数据管理员的管理控制台
2. `nexus-chat`：面向业务人员的自然语言对话式数据资产应用终端
3. `nexus-api`：面向上层业务系统的 API 服务端

这三类终端共享同一套数据接入、解析处理、治理管理、检索问答、权限控制和资产服务能力，避免重复建设。

## 二、产品定位

围绕公司现有非结构化数据，建设统一的数据沉淀、治理与复用平台，系统解决以下核心问题：

1. 数据怎么积累
2. 数据怎么处理
3. 数据怎么管理
4. 数据怎么使用

平台最终沉淀两类核心资产：

1. 面向业务应用的数据资产
2. 面向大模型和智能应用的知识资产

NEXUS 的目标不是建设单点工具，而是建设企业级的数据与知识资产基础平台，为管理、应用和开放三类场景提供统一底座。

## 三、总体产品形态

NEXUS 建议采用“一个统一底座 + 三类终端”的产品结构：

1. 统一资产底座
2. 管理控制台终端
3. 对话应用终端
4. API 服务终端

其中，统一资产底座负责沉淀平台核心能力：

1. 数据源接入
2. 文档解析与清洗
3. 分类分级与标签治理
4. 目录、版本、生命周期管理
5. 权限与可见范围控制
6. 检索、召回、知识组织
7. 问答与智能应用支撑
8. 审计、统计与运营分析

三类终端均基于统一底座构建，但服务对象、交互方式和输出能力不同。

## 四、三类终端设计

### 4.1 面向数据管理员的管理控制台 - `nexus-console`

目标用户：

1. 数据管理员
2. 平台运营人员
3. 部门数据负责人

核心定位：

提供企业数据资产和知识资产的全生命周期管理能力，是平台治理与运营入口。

核心能力建议包括：

1. 数据源接入管理
2. 数据集、知识库、文档库的创建与维护
3. 分类分级、标签体系、元数据管理
4. 清洗规则、切片规则、解析策略配置
5. 资产目录管理与版本管理
6. 权限控制、共享范围、审批流配置
7. 生命周期管理，包括上架、更新、归档、下线
8. 检索配置、召回策略、知识构建参数配置
9. 数据处理任务管理与运行监控
10. 使用统计、调用分析、审计追踪

价值重点：

1. 让数据资产“可接入、可管理、可治理、可追踪”
2. 让知识资产“可构建、可维护、可运营、可复用”
3. 为对话终端和 API 服务端提供可信资产底座

### 4.2 面向业务人员的自然语言对话终端 - `nexus-chat`

目标用户：

1. 业务人员
2. 运营人员
3. 销售、交付、支持等非技术用户

核心定位：

提供基于自然语言对话的数据资产应用入口，让业务人员不依赖复杂检索和后台操作，也能直接使用平台沉淀的数据与知识资产。

核心能力建议包括：

1. 自然语言问答
2. 面向资产目录的语义检索与推荐
3. 基于权限的数据内容查询与知识检索
4. 多轮对话与上下文理解
5. 答案引用来源展示与可追溯
6. 支持按部门、主题、场景进行知识导航
7. 支持常见业务任务的快捷入口，如制度查询、材料查找、经验问答
8. 支持将结果导出、分享或转交给业务流程

设计原则：

1. 以“问问题”和“找资料”为第一入口
2. 结果必须可追溯、可解释、可控权限
3. 不直接暴露复杂治理概念，强调低门槛和高效率

价值重点：

1. 降低企业内部数据与知识使用门槛
2. 提升已有资产的实际使用率
3. 让资产平台从“后台管理系统”升级为“业务使用平台”

### 4.3 面向上层业务应用的 API 服务端 - `nexus-api`

目标用户：

1. FLOW 及其他业务系统开发团队
2. 智能应用建设团队
3. 内部集成平台与服务编排团队

核心定位：

将平台沉淀的数据与知识能力以标准化服务形式开放给上层系统，成为企业统一的数据资产与知识服务基础设施。

核心能力建议包括：

1. 数据资产查询 API
2. 目录与元数据查询 API
3. 检索与召回 API
4. 知识问答 API
5. 文档解析与入库 API
6. 标签、分类、权限校验相关 API
7. 调用鉴权、限流、审计、监控能力
8. 场景化能力封装，如智能检索、知识摘要、内容推荐

设计原则：

1. 对上层系统提供统一接口规范
2. 与控制台配置能力保持一致
3. 保证服务可集成、可扩展、可审计

价值重点：

1. 避免各业务系统重复建设知识检索和问答能力
2. 提升平台能力复用率
3. 为后续 AI 场景提供标准化服务能力输出

## 五、核心价值调整

在引入三类终端后，平台核心价值建议调整为以下五点：

1. 将分散数据沉淀为企业可持续复用的资产，减少重复加工和重复建设。
2. 为 FLOW 及后续业务系统提供统一的数据资产与知识资产底座。
3. 为业务人员提供自然语言驱动的数据与知识使用入口，降低使用门槛。
4. 为知识问答、智能检索、智能助手等 AI 场景提供统一支撑。
5. 推动数据管理从项目式交付转向平台化、标准化、服务化运营。

## 六、目标用户与角色分层

建议平台从一开始就按角色分层设计，而不是按功能模块孤立设计。

### 6.1 数据管理员角色

关注点：

1. 数据是否接得进来
2. 数据是否治理得住
3. 权限是否可控
4. 资产是否可运营

对应终端：

1. `nexus-console`

### 6.2 业务使用者角色

关注点：

1. 能否快速找到需要的信息
2. 是否能通过自然语言直接提问
3. 结果是否可信、可追溯
4. 是否贴近日常业务场景

对应终端：

1. `nexus-chat`

### 6.3 系统集成方角色

关注点：

1. 能否通过标准 API 接入
2. 是否稳定、统一、易集成
3. 权限和审计是否完善
4. 是否支持后续能力扩展

对应终端：

1. `nexus-api`

## 七、实施计划调整

在原有阶段划分基础上，实施计划建议显式纳入三类终端建设目标，避免后续出现“底座建完但终端缺失”或“终端先行但底座薄弱”的问题。

### 阶段0：前置准备

周期：2 周

重点内容：

1. 明确一期试点场景和建设目标
2. 梳理首批数据源、知识资产范围和业务使用对象
3. 明确三类终端的一期边界与优先级
4. 形成总体方案、信息架构和初步排期评估

阶段成果：

1. 明确试点范围和建设边界
2. 形成首批数据清单和资产范围清单
3. 明确 `nexus-console`、`nexus-chat`、`nexus-api` 的一期能力范围
4. 输出一期建设方案和初步排期

### 阶段1：基础平台建设

周期：3 到 4 周

重点内容：

1. 接入首批知识类和重点非结构化数据
2. 建设基础分类、标签、清洗、检索能力
3. 搭建统一资产底座，完成目录、权限、版本、生命周期等基础管理能力
4. 上线 `nexus-console` 基础版
5. 上线 `nexus-api` 基础能力版
6. 上线 `nexus-chat` 最小可用版本，支持基础问答与语义检索

阶段成果：

1. 形成数据资产平台基础框架
2. 实现首批数据资产的统一接入和基础管理
3. 三类终端具备一期最小可用能力
4. 支撑 1 到 2 个试点场景验证使用

### 阶段2：数据资产扩展与治理建设

周期：1 到 2 周

重点内容：

1. 扩展更多数据源和业务部门接入
2. 推进跨部门数据资产复用
3. 建设标准化治理机制，包括分类分级、标签规则和管理规范
4. 提升 `nexus-console` 的治理与运营能力
5. 提升 `nexus-chat` 的场景化使用能力，如专题问答、资料定位、权限内检索
6. 完善 `nexus-api` 的标准接口、鉴权和服务治理能力

阶段成果：

1. 扩大数据资产覆盖范围
2. 初步形成跨部门复用机制
3. 建立标准化治理框架和管理规则
4. 提升三类终端在多场景下的复用能力和管理能力

### 阶段3：能力开放与赋能

周期：1 周

重点内容：

1. 面向上层业务系统和智能应用开展接入赋能
2. 明确 API 使用规范、接入规范和输出标准
3. 沉淀 `nexus-chat` 的典型业务场景模板
4. 建立平台运营、反馈和持续迭代机制

阶段成果：

1. 完成上层场景接入和能力输出闭环
2. 形成统一的能力开放标准
3. 形成业务用户使用路径与推广机制
4. 建立平台持续运营、反馈和迭代机制

## 八、一期建设范围建议

如果一期资源有限，建议聚焦“最小闭环”而不是追求功能铺满。

一期最小闭环建议为：

1. `nexus-console`：完成数据接入、目录管理、标签管理、权限管理、版本管理、任务监控
2. `nexus-chat`：完成自然语言问答、语义检索、结果引用、权限内访问
3. `nexus-api`：完成检索 API、问答 API、资产查询 API、鉴权与审计

一期试点优先验证三件事：

1. 数据能否稳定沉淀为资产
2. 业务人员能否真正使用起来
3. 上层系统能否通过统一 API 完成集成

## 九、资源诉求调整

原方案中的“1 名负责人 + 2 名开发 + 1 名业务专家”可以作为最低配置，但在同时建设三类终端的情况下，存在排期紧、并行度不足、终端体验不完整的风险。

### 9.1 最低配置

1. 负责人 1 名，负责整体规划、推进和跨部门协调
2. 开发 2 名，负责控制台、底座能力、API 与对话能力建设
3. 业务专家 1 名，负责试点场景梳理、数据协调和业务验收

### 9.2 建议配置

1. 产品/项目负责人 1 名
2. 后端开发 2 名，负责数据接入、治理、检索、API 服务
3. 前端开发 1 名，负责控制台和对话终端建设
4. 业务专家 1 名，负责试点场景定义、验收与推广
5. 如涉及模型编排、提示词、检索增强策略，可补充 AI 应用工程支持 0.5 到 1 人

建议理由：

1. `nexus-console` 与 `nexus-chat` 的交互形态差异较大，不适合完全混合开发
2. `nexus-api` 需要稳定性、接口规范和审计能力，后端工作量独立且持续
3. 对话终端是否真正可用，将直接决定平台是否能被业务端接受

## 十、总结

NEXUS v3.0 建议从“数据资产管理平台”升级为“统一资产底座 + 三类终端”的企业级平台方案。

整体推进路径建议为：

1. 先明确试点场景和首批资产范围
2. 先建设统一底座与管理控制能力
3. 同步交付面向业务的自然语言对话终端最小版本
4. 同步开放标准 API，支撑上层业务系统接入
5. 通过试点验证资产沉淀、业务使用和系统集成三类价值闭环

这样可以确保平台不是只停留在“后台可管理”，而是真正实现：

1. 资产可治理
2. 业务可使用
3. 能力可开放
4. 平台可持续运营
