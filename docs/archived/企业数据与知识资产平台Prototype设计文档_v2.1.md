## 一、结论：Spec v2.1 是否足以指导 Prototype

《企业数据与知识资产平台需求 Spec v2.1 — NEXUS》已经可以作为 Prototype v2.1 的上游需求基线。它明确了角色、P0/P1/P2 优先级、控制台信息架构、功能需求、数据对象、接口要求、端到端验收和架构 v2.1 的关键约束。

但 Spec 仍不能直接替代 Prototype 设计文档，Prototype v2.1 需要进一步补齐页面级设计输入：

| 缺口 | Spec v2.1 已覆盖 | Prototype v2.1 补充 |
|------|-----------------|---------------------|
| 页面结构 | 已定义一级菜单和能力范围 | 明确页面清单、Tab、列表、详情页、抽屉、弹窗关系。 |
| 操作流 | 已定义业务场景和验收用例 | 明确页面跳转、按钮、状态变化、异常分支和禁用条件。 |
| 字段展示 | 已定义主数据和元数据要求 | 明确列表字段、筛选字段、表单字段、详情字段和字段优先级。 |
| 状态表达 | 已定义资产版本、作业、索引、治理状态 | 统一状态标签、颜色语义、可点击动作和不可操作原因。 |
| 规则治理 | 已定义规则集、规则、决策日志 | 明确规则配置页、规则发布确认、规则命中追踪和重跑治理入口。 |
| AI 治理 | 已定义 AI 建议、模型配置、质量评分、证据引用和采纳状态 | 明确 AI 建议卡片、质量评分解释、证据引用、人工覆盖、反馈标签和 AI 重评分入口。 |
| 追溯链路 | 已定义 raw、version、ref、chunk、audit | 明确资产详情、引用卡片、决策日志和血缘 Tab 的跳转关系。 |
| 原型验收 | 已定义系统验收 | 定义页面覆盖率、端到端可演示路径和架构一致性检查项。 |

因此，Prototype v2.1 以 Spec v2.1 和架构 v2.1 为基线，作为低保真 / 中保真原型设计、评审和迭代的直接输入。

## 二、Prototype v2.1 目标与范围

### 2.1 原型目标

Prototype v2.1 的目标不是呈现最终视觉 UI，而是验证产品信息架构、核心流程、页面字段、状态表达、AI 治理、质量评分、规则护栏和角色权限是否能够支撑一期 P0 上线闭环。

原型需要回答以下问题：

1. 平台/数据管理员能否完成本地组织用户维护、数据源配置、接入、作业追踪、规则配置、治理复核、权限配置和审计查看。
2. 业务专家能否完成规则效果评审、标签和分级修订、质量复核、检索试测和知识资产审核。
3. API 调用方能否完成 API Key 申请、权限范围查看、接口联调、调用记录查看和问题反馈。
4. 运维人员能否基于工作台、作业中心、审计日志和基础运行状态定位失败作业和请求链路。
5. 任一检索、问答、作业、资产、索引、规则治理和审计结果是否具备可追溯入口。
6. 原型是否避免把全部资产推入人工发布流，是否体现自动治理优先、人工复核兜底。
7. 原型是否能说明 AI 为什么给出某个分类、分级、标签、组织范围和质量评分，以及规则为何采纳或拦截。
8. 人工覆盖 AI 建议后，是否能形成反馈回灌和重评分闭环。

### 2.2 原型范围

| 范围 | 页面 / 能力 | 原型要求 |
|------|-------------|----------|
| P0 | 工作台、数据源管理、数据接入、原始数据台账、作业中心、资产目录、资产详情、治理中心、规则配置、AI 治理与质量评分、权限与审计 | 必须绘制页面结构、关键字段、核心操作、状态流和追溯入口。 |
| P0 | 静态文档入库、爬虫批量接入、AI 治理自动可用、AI 质量评分解释、人工复核、规则配置发布、权限隔离、重处理 / 重治理 / AI 重评分 | 必须能串成端到端演示路径。 |
| P1 | 检索测试、知识资产、API Key 运营、规则效果分析、AI 效果分析、钉钉同步配置 | 绘制基础框架和主要交互入口。 |
| P2 | D5/D6 正式接入、知识图谱生产化、SFT 语料量产、评价标准库、独立运维观测中心 | 不进入 Prototype v2.1 主路径，可在空态或后续版本说明中标注。 |

### 2.3 原型保真度

| 阶段 | 保真度 | 目标 |
|------|--------|------|
| 第一轮 | 低保真线框 | 验证菜单、页面结构、核心流程、字段和状态口径。 |
| 第二轮 | 中保真交互 | 验证筛选、表单、规则编辑、状态、弹窗、抽屉和端到端路径。 |
| 第三轮 | 视觉样稿 | 对工作台、资产目录、资产详情、治理中心、规则配置、作业中心、权限与审计做视觉深化。 |

## 三、用户角色与原型入口

### 3.1 原型角色

| 角色 | 原型入口 | 主要关注页面 |
|------|----------|--------------|
| 平台/数据管理员 | `nexus-console` | 工作台、数据源管理、数据接入、原始数据台账、作业中心、资产目录、治理中心、AI 治理与质量评分、规则配置、权限与审计。 |
| 业务专家 | `nexus-console` | 工作台、治理中心、AI 建议抽检、质量评分校准、规则决策、质量复核、检索测试、知识资产。 |
| 运维人员 | `nexus-console` | 工作台、作业中心、资产详情元信息、审计日志、基础运行状态。 |
| API 调用方 | API 管理视图或上层系统联调入口 | API Key、接口权限、调用记录、作业查询、检索联调。 |

### 3.2 权限差异在原型中的体现

| 能力 | 平台/数据管理员 | 业务专家 | 运维人员 | API 调用方 |
|------|----------------|---------|---------|-----------|
| 本地组织用户维护 | 可见可编辑 | 不可见 | 不可见 | 不可见 |
| 数据源配置 | 可见可编辑 | 只读或不可见 | 不可见 | 不可见 |
| 治理规则配置 | 可创建 / 校验 / 发布 / 回滚 | 可建议 / 可评审 | 不可见 | 不可见 |
| AI 治理建议 | 可采纳 / 修订 / 驳回 / 重评分 | 可抽检 / 校准 / 反馈 | 只读失败摘要 | 不可见 |
| 治理待办 | 可确认 / 修订 / 驳回 | 可建议 / 修订 / 审核 | 不可见 | 不可见 |
| 决策追踪 | 可查看授权范围 | 可查看参与审核范围 | 只读故障摘要 | 不可见 |
| 作业重试 | 可操作 | 可申请 | 可操作 | 按授权可操作 |
| 权限配置 | 可操作 | 不可操作 | 只读审计 | 仅查看自身权限 |
| 审计日志 | 全量按权限查看 | 查看本人审核记录 | 查看维护相关日志 | 查看自身调用记录 |
| 基础运行状态 | 可查看 | 不可见 | 可查看 | 不可见 |

原型不需要绘制四套完全不同 UI，但必须在关键按钮上标注角色可见性、禁用规则和禁用原因。

## 五、核心端到端原型流程

### 5.1 流程 A：静态文档入库到自动可用

```text
数据源管理
  → 新建 / 选择上传入口
  → 数据接入上传文件 / 批量上传
  → 生成接入批次
  → 原始数据台账查看 raw_object
  → 作业中心查看解析 / 标准化 / AI 治理评分 / 规则护栏 / 索引进度
  → AI 生成治理建议和质量评分
  → 规则护栏自动采纳并生成治理结果
  → 资产版本进入 available
  → 资产目录查看当前版本
  → 检索测试验证召回与引用
```

关键原型校验：

1. 每一步都能看到对象 ID、状态和追溯入口。
2. 标准化完成后能看到 `normalized_asset_ref`。
3. 自动治理通过后能查看 AI 建议、质量维度分、证据引用、命中规则和决策日志。
4. 版本进入 `available` 后资产目录显示“当前可用版本”。
5. 资产进入可检索索引前必须满足 AI 质量评分、治理结果、质量报告和索引准入。

### 5.2 流程 B：爬虫批量推送到资产化

```text
数据源管理
  → 爬虫推送配置
  → 查看批次推送记录
  → 原始数据台账查看原始 JSON 包
  → 作业中心查看 structured_sync / normalize_record / ai_governance_score / governance_evaluate / index_build
  → 资产目录按 D1/D2 过滤
  → 决策追踪查看 AI 分类、分级、标签、组织范围建议和规则采纳状态
  → 检索测试验证动态数据结果
```

关键原型校验：

1. 爬虫批次必须展示来源系统、抓取时间、来源 URL、原始包摘要。
2. JSON 原始包和标准化记录要能切换查看。
3. 治理结果必须展示 AI 模型配置、提示词版本、规则集版本和命中规则。

### 5.3 流程 C：AI 治理建议与质量评分

```text
治理中心
  → AI 治理 Tab
  → 查看资产的 AI 分类 / 分级 / 标签 / 组织范围建议
  → 查看 AI 质量维度分、证据引用、置信度和阻断原因
  → 规则护栏采纳 / 部分采纳 / 进入复核
  → 人工抽检并提交反馈标签
  → 必要时触发 AI 重评分
```

关键原型校验：

1. AI 建议必须展示“建议值、理由、证据、置信度、采纳状态”。
2. AI 质量评分必须展示维度分、综合分、阻断原因和修复建议。
3. AI 建议不得直接表现为最终结果，界面必须区分“AI 建议”和“正式治理结果”。
4. 人工覆盖 AI 结论必须填写原因和反馈标签。

### 5.4 流程 D：规则配置、发布与自动治理

```text
治理中心
  → 规则配置 Tab
  → 新建规则集草稿
  → 配置分类 / 分级 / 标签 / 组织范围 / AI 输出护栏 / 质量准入 / 复核触发 / 索引准入规则
  → 规则校验
  → 发布规则集
  → 新接入资产先生成 AI 建议，再使用 active 规则集执行护栏治理
  → 决策追踪查看 AI 建议和规则命中
```

关键原型校验：

1. 规则编辑器必须展示规则类型、优先级、条件、动作、置信度阈值和执行模式。
2. 规则发布前必须展示校验结果、影响范围和审计提示。
3. active 规则集只读，修改必须创建新版本草稿。
4. 规则命中必须能在治理决策日志中回溯，并能说明是否覆盖了 AI 建议。

### 5.5 流程 E：资产进入人工复核

```text
工作台待复核
  → 治理中心治理待办
  → 打开复核抽屉
  → 查看 AI 建议 / 质量评分 / 证据引用 / 命中规则 / 冲突原因 / 候选值
  → 人工确认或修订分类、分级、标签、组织范围
  → 复核通过后版本进入 available 或继续 review_required
  → 索引投影更新或保持禁用
```

关键原型校验：

1. 复核项必须说明为何进入 `review_required`。
2. 人工覆盖必须展示 AI 建议值、修改前后值，并要求填写原因和反馈标签。
3. L3/L4 资产复核时必须强调组织范围、脱敏策略和索引准入。
4. 复核通过不等同于无条件可检索，仍需满足索引准入。

### 5.6 流程 F：权限隔离与 API 调用方联调

```text
权限与审计
  → 新建 API Key
  → 配置接口范围、数据域、组织范围、分级上限、QPS
  → API 调用方联调 search / qa
  → 检索测试模拟调用方权限
  → 审计日志查看调用记录和脱敏动作
```

关键原型校验：

1. API Key 创建后只展示一次明文。
2. 权限范围必须可读、可复核。
3. L4 数据默认不返回明文字段。
4. 审计日志能按调用方、接口、请求 ID 查询。

### 5.7 流程 G：失败作业重试、重处理、重治理与 AI 重评分

```text
工作台基础运行状态 / 作业中心
  → 查看失败作业
  → 打开作业详情
  → 查看阶段、错误、关联对象、日志摘要
  → 点击重试 / 触发重处理 / 触发重治理 / 触发 AI 重评分
  → 查看新作业状态
  → 资产详情查看新版本与旧版本关系
```

关键原型校验：

1. 失败原因必须可见。
2. 重试、重处理、重治理要区分。
3. 高风险重处理需确认影响范围。
4. 新旧版本关系必须可追溯。
5. 当前版本不能由手工字段指定，只能由唯一 `available` 版本派生展示。

## 八、表单校验与交互规则

### 8.1 通用表单校验

| 字段类型 | 校验规则 |
|----------|----------|

| 组织范围 | L3/L4 或规则判断需要组织范围时必填。 |
| API Key 额度 | 正整数，需有默认值。 |
| URL | 必须符合 URL 格式。 |
| 文件 | 必须符合格式白名单和大小限制。 |
| 驳回原因 | 必填，最长 500 字。 |
| 规则条件 | 只能使用白名单字段和受限操作符。 |
| 规则动作 | 必须符合规则类型允许的动作集合。 |
| AI 反馈标签 | 人工覆盖、评分偏差、证据不足、低置信、敏感风险等枚举必选。 |
| 发布说明 | 规则发布、回滚、人工覆盖时必填。 |

### 8.2 批量操作规则

| 操作 | 规则 |
|------|------|
| 批量确认治理结果 | 仅允许同数据域、同规则类型、同复核原因的低风险样本。 |
| 批量下线资产 | 必须二次确认，并展示影响检索范围。 |
| 批量重建索引 | 必须展示预计对象数和影响范围。 |
| 批量重试作业 | 仅支持可重试失败状态。 |
| 批量重治理 | 必须选择规则集版本，并展示影响资产数量。 |
| 批量 AI 重评分 | 必须选择模型配置版本，并展示预计资产数量和外部模型调用风险。 |

### 8.3 高风险操作规则

以下操作必须二次确认：

1. 人工复核通过并使版本进入 `available`。
2. 停用资产或版本。
3. 禁用 API Key。
4. 触发重处理。
5. 触发批量重治理。
6. 触发 AI 重评分或批量 AI 重评分。
7. 采纳或驳回 AI 对 L3/L4 资产的治理建议。
8. 发布或回滚规则集。
9. 重建索引。
10. 导出审计日志。

### 8.4 禁用条件

| 操作 | 禁用条件 |
|------|----------|
| 人工复核通过 | 缺少标准化引用、AI 建议未处理、质量报告不达标、治理必填缺失、索引准入为拒绝。 |
| 进入可检索索引 | 版本不是 `available`、索引准入非 `allow`、权限范围不明。 |
| 编辑规则 | 规则集不是 `draft`。 |
| 发布规则集 | 规则校验存在错误或发布说明为空。 |
| 重治理 | 没有可用规则集或目标资产无标准化引用。 |
| AI 重评分 | 没有可用 AI 模型配置、目标资产无标准化引用或调用策略禁止。 |
| 查看原文片段 | 用户无权限或资产为 L4 且未显式授权。 |

## 十一、原型评审口径

### 11.1 页面覆盖检查

| 检查项 | 通过标准 |
|--------|----------|
| P0 页面覆盖 | NX-00 至 NX-10 必须完整覆盖。 |
| P0 弹窗 / 抽屉覆盖 | 人工复核、规则发布、重处理 / 重治理 / AI 重评分、API Key、治理待办、作业详情、来源引用、决策追踪、规则编辑、AI 建议与质量评分必须覆盖。 |
| P1 页面框架 | 检索测试、知识资产需具备基础结构。 |
| 角色权限标注 | 关键按钮需标注可见角色或禁用条件。 |
| 状态表达 | 作业、资产版本、索引、规则集、治理决策、AI 采纳、分级状态需统一表达。 |
| 架构边界 | 不出现企业统一身份平台强依赖，不出现独立运维观测中心主路径。 |

### 11.2 端到端流程检查

| 流程 | 通过标准 |
|------|----------|
| 静态文档入库 | 能从上传走到资产详情、AI 治理建议、质量评分、自动治理、当前可用版本和检索测试。 |
| 爬虫批量接入 | 能从数据源配置走到批次、标准化记录、AI 建议、治理决策和检索。 |
| AI 治理评分 | 能查看 AI 建议、质量维度分、证据引用、置信度、采纳状态和人工反馈。 |
| 规则配置发布 | 能从规则草稿走到校验、发布、AI 输出护栏命中和决策日志。 |
| 人工复核 | 能从 `review_required` 走到 AI 建议、决策查看、人工覆盖、反馈标签、复核通过或继续复核。 |
| 权限隔离 | 能创建 API Key 并模拟权限过滤结果。 |
| 失败重试 | 能从工作台基础状态走到作业详情并重试。 |
| 重治理 | 能选择规则集版本并触发重治理，治理变化后索引投影更新或标记 `stale`。 |
| AI 重评分 | 能选择模型配置版本并触发重评分，评分变化后生成新的质量报告和决策日志。 |

### 11.3 原型评审问题清单

评审时必须确认以下问题：

1. 平台/数据管理员是否能理解从“接入”到“自动治理可用”的主路径。
2. 业务专家是否能快速定位待复核、AI 低置信、规则冲突、标签草稿和检索问题。
3. AI 建议与质量评分是否能说明“建议值、理由、证据、置信度和采纳状态”。
4. 规则配置是否足够指导研发实现规则集、AI 输出护栏、规则编辑、校验、发布和回滚。
5. 治理决策追踪是否能解释 AI 自动通过、需复核和人工覆盖。
6. API 调用方权限范围是否表达清楚。
7. 资产详情是否足够支撑 raw、version、standardized ref、ai governance、quality、governance、chunk、index 的全链路追溯。
8. L3/L4 权限和脱敏是否在界面中有明确提示。
9. 批量操作和高风险操作是否有足够防误触设计。
10. 是否避免把所有资产都推入人工复核或人工开放流程。

## 十三、后续原型迭代建议

Prototype v2.1 完成后建议按以下顺序迭代：

1. 先评审 P0 端到端流程，不先纠结视觉风格。
2. 第二轮补齐字段级交互、AI 建议解释、质量评分证据、规则表达式编辑、批量操作和异常态。
3. 第三轮选择工作台、作业中心、资产目录、资产详情、治理中心、规则配置、权限与审计做中高保真视觉稿。
4. API 调用方视图应在接口文档确定后补充更细的联调台原型。
5. 规则配置页应在首批真实规则确认后补充规则模拟和规则效果分析。
6. AI 治理页应在首批真实样本确认后补充模型效果分析、提示词版本对比和人工反馈统计。
7. 独立运维观测中心应在后续运维业务设计明确后再单独出原型，不进入一期主路径。
