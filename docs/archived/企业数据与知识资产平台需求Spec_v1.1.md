## 一、文档说明

本文档基于 [企业数据与知识资产平台nexus_v7.0.md](/home/bjbodao/projects/nexus/docs/企业数据与知识资产平台nexus_v7.0.md) 和 [企业数据与知识资产平台技术选型和架构nexus_v1.1.md](/home/bjbodao/projects/nexus/docs/企业数据与知识资产平台技术选型和架构nexus_v1.1.md) 输出平台需求规格说明，定义一期建设范围、角色、功能需求、非功能需求、数据要求、接口要求和验收口径。

本文档用于：

1. 统一业务、产品、研发、测试、实施对平台范围的理解。
2. 作为研发实施、测试验收和交付评审的依据。
3. 为后续版本迭代提供需求基线。

### 1.1 v1.1 需求优化原则

1. 需求必须与 v7.0 的“统一数据资产底座 + 两类终端”产品结构一致。
2. 需求必须区分 NEXUS、MinerU、RAGFlow、爬虫系统和上层业务系统的职责边界。
3. 需求必须覆盖原始留存、标准化契约、作业状态、权限审计、质量复核和检索引用追溯。
4. 需求必须明确一期必做、一期应具备、二期预留和后续演进，避免把长期规划全部压入一期。
5. 角色模型必须收敛，平台管理与数据治理由统一角色承担，授权业务访问统一纳入 API 调用方模型。

## 二、项目背景与建设目标

### 2.1 背景问题

当前企业内外部非结构化数据存在以下共性问题：

1. 数据分散在网盘、NAS、系统库表、爬虫结果和业务文件中，缺少统一入口。
2. 相同文档在不同项目中重复处理，缺少原始留存和标准化资产沉淀。
3. 文档解析结果、标签规则、检索索引和权限控制之间没有统一治理模型。
4. 上层业务系统和 AI 应用需要重复建设知识库、问答语料和检索服务。
5. 数据资产和知识资产缺少版本、质量、权限、审计和来源引用的一致口径。

### 2.2 建设目标

平台建设目标如下：

1. 构建统一的数据源接入层、原始数据持久化层和作业编排处理层。
2. 构建统一的标准化资产模型，形成数据资产与知识资产的共同输入件。
3. 建立分类、分级、标签、版本、权限、审计等治理体系。
4. 对外提供统一的资产查询、检索、问答、作业和权限校验接口。
5. 为 AI 助教、知识检索、教材生成、图谱构建、评价标准化等场景提供底座支撑。
6. 通过 `nexus-console` 承载控制面治理入口，通过 `nexus-api` 承载统一服务开放入口。

## 三、术语定义

| 术语 | 定义 |
|------|------|
| 数据资产 | 经过接入、登记、留存和治理后，可被统一管理和复用的数据对象。 |
| 知识资产 | 在数据资产基础上经过切片、抽取、结构化或审核加工后形成的可直接消费对象。 |
| 原始对象 `raw_object` | 原始文件、原始 JSON 包、原始媒体引用等可信留存对象。 |
| 文档资产 `document_asset` | 平台侧文档主实体，承载分类、分级、权限和生命周期。 |
| 资产版本 `document_version` | 文档资产的版本实例，关联解析产物和标准化产物。 |
| 标准化文档 `normalized_document` | 面向 PDF、Word、PPT、图片等文档输入的统一标准对象。 |
| 标准化记录 `normalized_record` | 面向 JSON、Excel、数据库同步数据、Webhook 事件的统一标准对象。 |
| 知识切片 `knowledge_chunk` | 用于检索、问答和知识加工的标准切片对象。 |
| 索引清单 `index_manifest` | 记录索引分区、同步状态、失败原因与同步时间的清单对象。 |
| 质量报告 `quality_report` | 记录解析、标准化、元数据、切片质量与复核状态的质量对象。 |
| 知识资产版本 `knowledge_asset_version` | 问答语料、流程语料、图谱、评价标准等加工产物的版本对象。 |
| MinerU | 非结构化文档解析执行引擎。 |
| RAGFlow | 切片、索引与检索执行引擎。 |
| 控制面 | 接入、元数据、作业、权限、审计、运营和管理能力集合。 |
| 执行面 | 解析、标准化、标签抽取、索引同步、检索执行和知识加工能力集合。 |

## 四、角色与使用方

### 4.1 平台角色

v1.1 对平台角色进行收敛：原“平台管理员”和“数据管理员”合并为“平台/数据管理员”；授权业务访问不再单列为独立角色，统一合入“API 调用方”。

| 角色 | 说明 | 主要职责 |
|------|------|---------|
| 平台/数据管理员 | 平台级管理与数据治理合并角色 | 账号与角色管理、组织范围配置、系统配置、数据源注册、资产审核、分类分级、标签确认、版本管理、权限配置、审批处理、审计查看。 |
| 业务专家 | 业务审核角色 | 标签修订、分类规则确认、知识资产审核、质量抽检、试点场景定义、业务验收。 |
| 运维人员 | 平台运维角色 | 发布、监控、告警、容量管理、备份恢复、故障处理、运行报告输出。 |
| API 调用方 | 能力消费角色 | 上层业务系统、智能应用、集成方和授权业务访问入口，通过 API Key、JWT 或上层应用代理调用资产、检索、问答、作业接口。 |

### 4.2 非人工技术主体

| 主体 | 说明 | 权限要求 |
|------|------|---------|
| 系统连接器 | NAS 同步、爬虫推送、数据库同步、Webhook 等接入主体 | 必须绑定来源系统、组织范围、数据域和最小权限。 |
| 后台作业账号 | Worker、定时任务、索引同步、回调任务等内部主体 | 必须使用内部凭据，权限仅限必要的作业和数据范围。 |

### 4.3 外部系统

| 系统 | 与平台关系 |
|------|-----------|
| 爬虫系统 | 作为动态数据源推送产业政策、岗位招聘、人才需求等数据。 |
| MinerU | 作为文档解析执行引擎。 |
| RAGFlow | 作为切片、索引与检索执行引擎。 |
| 上层业务系统 | 通过 `nexus-api` 消费平台能力。 |
| 企业 IAM / SSO | 为控制台与 API 提供统一身份认证。 |

### 4.4 角色权限矩阵

| 能力 | 平台/数据管理员 | 业务专家 | 运维人员 | API 调用方 |
|------|----------------|---------|---------|-----------|
| 数据源注册与配置 | 可管理 | 可查看规则 | 不可操作 | 不可操作 |
| 原始对象台账查看 | 可查看 | 按授权查看 | 按故障处理需要查看元信息 | 不可直接查看 |
| 分类分级配置 | 可管理 | 可建议 / 审核 | 不可操作 | 不可操作 |
| 标签草稿审核 | 可确认 / 修订 / 驳回 | 可修订 / 建议 | 不可操作 | 不可操作 |
| 作业重试与重处理 | 可触发 | 可申请 | 可触发故障恢复类操作 | 可按授权触发指定作业 |
| 知识资产审核发布 | 可发布 | 可审核 | 不可操作 | 不可操作 |
| 资产查询与检索 | 按授权访问 | 按授权访问 | 按运维授权访问元信息 | 按 API 权限访问 |

| 系统监控与告警 | 可查看 | 不可操作 | 可管理 | 不可操作 |

## 五、范围定义

### 5.1 一期范围

一期建设范围固定如下：

1. 数据源接入：文件上传、目录导入、NAS 同步、爬虫推送。
2. 原始数据持久化：原始对象留存、校验摘要、版本快照、接入台账。
3. 作业编排：持久化作业中心、重试补偿、失败回查、人工复核。
4. 文档解析：基于 MinerU 的 PDF、Word、PPT、图片、扫描件解析。
5. 标准化处理：生成 `normalized_document`、`normalized_record`。
6. RAGFlow 集成：数据集映射、切片画像映射、索引构建、检索执行。
7. 治理能力：分类、分级、标签、版本、生命周期、质量复核。
8. 权限审计：RBAC、ABAC、组织范围、字段脱敏、审计日志。
9. 服务开放：`nexus-console`、`nexus-api`。
10. 运维观测：基础日志、指标、链路追踪、告警接入点。
11. 部署交付：单节点与三节点集群方案。

### 5.2 一期数据域范围

| 数据域 | 范围级别 | 说明 |
|------|---------|------|
| D1 产业行业知识域 | 一期纳入 | 静态文档与动态数据均纳入。 |
| D2 岗位职业知识域 | 一期纳入 | 招聘数据、职业能力分析、证书、职业大典。 |
| D3 专业建设知识域 | 一期纳入 | 教学标准、专业布点、人才培养方案。 |
| D4 课程教学知识域 | 一期纳入 | 教材、教案、课件、习题、实训指导书、媒体元数据。 |
| D5 教学行为与平台数据域 | 二期预留 | 平台行为数据、用户与课程信息。 |
| D6 能力评价与就业数据域 | 二期预留 | 能力评估结果、职业匹配度数据。 |

### 5.3 一期不纳入范围

以下内容不在 v1.1 的一期交付范围内：

1. D5/D6 全量正式接入与实时流处理。
2. 知识图谱完整生产化建设。
3. SFT 训练语料大规模量产。
4. 独立的运维观测中心完整产品化实现。
5. 全链路双活和跨机房容灾架构。
6. 面向最终 C 端用户的独立门户产品。

### 5.4 范围级别定义

| 范围级别 | 定义 |
|----------|------|
| 一期必须 | 一期验收必须完成，否则不满足上线基线。 |
| 一期应具备 | 一期需要完成最小闭环或预留入口，允许以基础版交付。 |
| 二期预留 | 一期完成模型、接口或扩展点预留，不要求生产化。 |
| 后续演进 | 不纳入一期和二期明确交付，仅作为架构演进方向。 |

## 六、业务场景

### 6.1 场景一：静态文档入库

平台/数据管理员或系统连接器提交教材、产业报告、人才培养方案等文档，平台完成原始留存、解析、标准化、索引和资产入库。

### 6.2 场景二：爬虫动态数据接入

爬虫系统按批次推送产业政策、岗位招聘和人才需求数据，平台完成批次登记、原始包留存、标准化处理与索引更新。

### 6.3 场景三：资产治理

平台/数据管理员对新入库资产执行分类分级、标签确认、版本状态管理和人工复核；业务专家参与业务规则与知识质量审核。

### 6.4 场景四：知识检索与问答

API 调用方通过 `nexus-api` 或上层应用入口执行检索和问答，平台返回带引用、带权限过滤、带脱敏处理的结果。

### 6.5 场景五：重处理与版本切换

当规则升级、解析失败、索引失效或人工复核发现问题时，平台触发重处理并生成新版本资产。

### 6.6 场景六：知识资产加工

基于标准化资产和知识切片，平台生成问答语料、流程语料、图谱结构和评价标准库，并经过审核后发布。

### 6.7 场景七：运维监控与故障恢复

运维人员通过观测能力查看 API、作业、解析、索引、存储和安全审计指标，对失败任务、队列积压、索引异常和容量风险进行处理。

## 七、功能需求

## 十二、版本演进说明

### 12.1 v1.1 范围结论

v1.1 的核心定位是：

1. 夯实非结构化数据处理底座。
2. 打通接入、解析、标准化、索引和检索主链路。
3. 完成企业级治理与服务开放的最小可用闭环。
4. 明确 NEXUS、MinerU、RAGFlow 和上层业务系统边界。
5. 收敛角色模型，降低权限配置和验收复杂度。

### 12.2 后续版本方向

后续版本沿以下方向演进：

1. D5 / D6 业务数据正式接入。
2. 问答语料、流程语料、图谱和评价标准库规模化加工。
3. `ops-observability` 独立产品化。
4. 数据库、检索和观测体系高可用升级。
5. 引入更完整的知识图谱 API、Agent 流程模板服务和评价标准服务。
