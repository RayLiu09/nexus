# 2026.05.11 关于Pipeline处理流程的补充意见
- 作业编排中心，资源接入后ingest_validate处理似乎是缺失的，也为记录该处理阶段的审计事件日志；
- 数据资产的assetize与normalize是不同的阶段，各自的职责，承载的服务组件以及处理规则需要明确化体现在“企业数据与资产平台nexus”的版本文档中，目前3.3.3节关于这一方面的呈现还是比较模糊；而软件实现框架上需要体现相应的处理组件；
- 目前MinerU的处理是单节点包含(CPU和GPU)，架构上需要预留扩展即适应MinerU集群化处理架构，可以按照CPU和GPU不同资源调度处理流量；minerU解析结果parsed中间结果.json中原文档中的图片数据发生丢失(提取的images存储跟.json结果一期存储，保证后期可以渲染出来)，调用mineru接口进行文档解析时需要根据内容类型需求自动开启OCR解析识别能力；mineru模型版本，三个选项:pipeline、vlm、MinerU-HTML，默认pipeline。如果解析的是HTML文件，model_version需明确指定为MinerU-HTML，如果是非HTML文件，可选择pipeline或vlm；请根据待解析文件的类型选择合适的model_version参数；
- 数据的平台资产标准化，即normalize化，由normalize_service承载，normalize处理规则(由业务专家定义)，处理过程采用基于LLM模型进行内容理解抽取，由业务规则做保底验证；
- normalized_asset_ref目前的实现结构与“企业数据与资产平台nexus”设计文档不匹配，存在结构缺失，需要优化重构；
- normalized标准化后，由RAGFlow建立数据资产检索索引，切片化；而不同数据资产类型的切片化/索引化处理逻辑不同，即每种normalized资产有其对应的chunk_profile类别，chunk_profile的规则由业务专家定义；即“企业数据与资产平台nexus”的“六、切片规范”的具体业务规则待业务专家定义清楚后文档需要优化重构；可标记为后续待优化处理；
- “七、元数据抽取规范”，目前代码实现中不同阶段关于metadata的处理都比较简陋，需要根据具体的业务规则要求在不同的处理阶段采集不同程度的metadata信息；
- “十、资产化入库规则”缺失了assetize阶段的入库规则；
- 数据分级仅作用于资产对象本身没有问题，后续加工版本知识等一路继承即可，目前数据分解不作为核心目标进行建设，数据资产默认配置L1/L2，数据的访问都需要登录凭证；
- 标签挂载在资产级asset没有问题，版本级和切片级继承自资产的处理机制是OK的；
- “自动化标签与人工校正机制”章节，“标准化处理和 metadata_enrich 作业完成后，系统基于接入契约、标准化文档内容和知识切片语义自动生成标签草稿，结果写入标签草稿区，等待管理员确认：”，(1). metadata_enrich服务进行标签化加工的对象是标准化数据资产normalized资产，此时还没有知识切片对象；(2). 流程设计上应尽量减轻人工审核负担，仅对不确定的资产标签进入人工审核状态，其他暂无异议的直接落库进入可用状态(需要记录审计日志)，流程上支持人工review已经处理完成的资产标签记录并进行修订驳回；
- “数据资产与知识资产的关系模型”中，knowledge_chunk切片, governece_result治理的对象是normalized对象，不是asset, asset_version对象，此处衍生关系图不准确；
- 技术架构上数据资产的标准化nornalize和知识化“知识资产精细化加工”划分为不同阶段的pipeline似乎更加合理。原因有2：(1).从数据接入到知识化流程步骤繁多，完全串行化时间周期过长；(2). 数据资产知识化的目标是明确的业务场景，即从简单的知识问答检索， 知识图谱/岗位图谱构建，到复杂的课程标准编写流程等，每种业务场景的知识加工标准和规则完全不同，且他们各自之间没有相互依赖关系；完全采用一个完整的pipeline将导致知识化流程与上游过于耦合，不利于后期的扩展维护；
- 再次声明一点，知识化加工环节，一期项目仅需要覆盖“简单知识问答检索”，即“RAG检索知识库”一个场景；
