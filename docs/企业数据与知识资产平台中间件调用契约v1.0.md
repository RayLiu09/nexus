## 版本记录

| 版本 | 日期 | 说明 |
|------|------|------|
| v1.0 | 2026-04-30 | 基于《企业数据与知识资产平台技术选型和架构 nexus_v2.2》梳理 MinIO、Redis、RabbitMQ 在 NEXUS P0 中的职责边界、命名规范、调用示例和禁止事项。 |

## 一、文档定位

本文档是 NEXUS v2.2 的中间件调用契约，约束后端 API、作业 Worker、适配器、测试用例和部署配置如何使用 MinIO、Redis、RabbitMQ。

本契约不替代《企业数据与知识资产平台技术选型和架构 nexus_v2.2》，而是对其中的中间件使用方式做工程化细化。所有实现必须继续遵守 v2.2 的核心边界：

1. PostgreSQL 是主数据、作业状态、治理结果、审计记录和幂等记录的最终事实源。
2. MinIO 只保存原始对象、解析产物、标准化产物和必要的执行文件，不承担主数据口径。
3. Redis 只保存可重建的缓存、短期状态、幂等处理中间态和轻量分布式控制信息，不保存不可丢失事实。
4. RabbitMQ 只承担异步任务投递、事件分发、重试和死信隔离，不保存最终业务状态。
5. 作业阶段、失败原因、重试次数、治理决策、版本状态和审计日志必须落 PostgreSQL，并可在控制台作业中心、资产详情、治理中心和权限审计中追溯。
6. 所有调用链必须携带 `trace_id`，对外 API 同时返回 `request_id`。
7. 不允许在日志、消息体、缓存 Key、对象 Key、异常堆栈或审计摘要中暴露 API Key、MinIO Secret、RabbitMQ 密码、L3/L4 明文、大段原文或模型输入全文。

## 二、统一调用原则

### 2.1 中间件职责矩阵

| 中间件 | NEXUS 定位 | 可保存内容 | 不保存内容 | 最终事实源 |
|--------|------------|------------|------------|------------|
| MinIO | 原始与中间产物对象存储 | `raw/`、`staging/`、`parsed/`、`normalized/` 下的对象文件、解析产物、标准化 JSON、必要清单文件 | 资产当前版本、治理结果、权限结果、作业最终状态、审计主记录 | PostgreSQL 台账字段，如 `raw_object.object_uri`、`parse_artifact.artifact_uri`、`normalized_asset_ref.object_uri` |
| Redis | 缓存与短期协同 | 热点元数据、权限判定结果、规则集缓存、Prompt 配置缓存、短期幂等锁、接口缓存、Worker 心跳、轻量运行状态 | 官方资产状态、唯一版本关系、质量报告、治理结果、审计日志、长期任务队列 | PostgreSQL，可按事件或 TTL 重建 |
| RabbitMQ | 异步任务与事件通道 | Celery 任务、领域事件、重试消息、死信消息、轻量上下文 ID | 大文件、完整原文、最终作业状态、最终治理结果、长期待办 | PostgreSQL `job`、`index_manifest`、`governance_decision_log`、`audit_log` |

### 2.2 统一标识与上下文字段

以下字段在跨中间件调用时必须保持一致：

| 字段 | 用途 | 生成方 | 使用要求 |
|------|------|--------|----------|
| `trace_id` | 内部链路追踪 | API 入口或作业编排器 | HTTP、Celery 消息、日志、审计摘要、Worker 阶段记录都必须携带。 |
| `request_id` | 对外请求追踪 | `nexus-api` | 对外响应返回；内部可映射到 `trace_id`。 |
| `job_id` | 作业主记录 | `job-orchestrator` | RabbitMQ 消息只传 `job_id` 和必要上下文，Worker 以数据库作业记录为准。 |
| `batch_id` | 接入批次 | `ingest-gateway` | 用于原始台账、批量作业、对象 Key 和控制台追溯。 |
| `raw_object_id` | 原始对象台账 | `raw-storage` / `ingest-gateway` | MinIO 对象与 PostgreSQL 台账的主关联 ID。 |
| `asset_id` | 长期资产身份 | `metadata-service` | 不得用作对象覆盖写入依据。 |
| `version_id` | 资产版本边界 | `metadata-service` | 解析、标准化、治理、索引和重处理的主要边界。 |
| `ref_id` | 标准化对象引用 | `normalize-service` / `metadata-service` | 指向 `normalized/` 对象，不反写到 `document_version`。 |
| `idempotency_key` | 幂等控制 | API 调用方或接入服务 | PostgreSQL 必须保存最终幂等结果；Redis 只可做短期防并发。 |

### 2.3 环境与命名前缀

除非具体部署另有隔离策略，所有中间件命名应包含以下维度：

```text
nexus.<env>.<domain>.<object_or_purpose>[.<qualifier>]
```

字段说明：

| 字段 | 说明 | 示例 |
|------|------|------|
| `nexus` | 固定系统名前缀 | `nexus` |
| `env` | 环境标识 | `dev`、`test`、`staging`、`prod` |
| `domain` | 业务域或技术域 | `ingest`、`raw`、`parse`、`normalize`、`governance`、`index`、`search`、`auth`、`rule`、`prompt`、`job` |
| `object_or_purpose` | 对象或用途 | `raw_object`、`document_version`、`permission`、`rule_set`、`ai_prompt_profile` |
| `qualifier` | 可选限定 | `v1`、`dlq`、`lock`、`cache`、`retry` |

对象存储 Key 使用 `/` 分隔，Redis Key 使用 `:` 分隔，RabbitMQ exchange / routing key / queue 使用 `.` 分隔。

### 2.4 敏感数据处理

1. L3/L4 原文默认不进入 Redis 和 RabbitMQ。
2. MinIO 可以保存加密或访问受控的原始对象，但对象 Key 不得包含敏感标题、姓名、手机号、证件号、API Key 或业务密级明文。
3. AI 治理输入摘要、质量证据和模型调用摘要只保存字段统计、摘要、证据引用和脱敏说明，不保存大段模型输入或模型原始长响应。
4. Redis 缓存值如包含权限、分级、组织范围或脱敏策略，必须设置 TTL，并在权限、规则、治理结果变化时主动失效。
5. RabbitMQ 消息体只传 ID、状态、版本、幂等键和轻量摘要；Worker 需要正文时必须从 PostgreSQL 台账定位 MinIO 对象后读取。

## 三、MinIO 调用契约

### 3.1 定位与职责

MinIO 是 NEXUS 的对象存储基线，服务于“原始数据先留存后处理”和“标准化后治理”的链路。它承载以下对象：

1. 接入后的原始文件或原始 JSON 包。
2. 解析前后的临时对象和中间文件。
3. MinerU 输出的解析产物。
4. `normalized_document` / `normalized_record` 标准化产物。
5. 必要的导入清单、校验清单和可回放执行输入。

MinIO 不保存资产当前版本、治理正式结果、权限策略、审计主记录和作业最终状态。这些内容必须由 PostgreSQL 表和读取视图表达。

### 3.2 命名规范

#### 3.2.1 Bucket 命名

P0 推荐按环境隔离 Bucket：

| 用途 | Bucket 模板 | 示例 |
|------|-------------|------|
| NEXUS 对象主桶 | `nexus-<env>-objects` | `nexus-prod-objects` |
| 临时导入桶，若需物理隔离 | `nexus-<env>-staging` | `nexus-dev-staging` |
| 备份或归档桶，P0 可预留 | `nexus-<env>-archive` | `nexus-prod-archive` |

单桶方案下必须使用一级目录分区；多桶方案下仍建议保留相同目录结构，降低代码分支。

#### 3.2.2 一级目录分区

MinIO 对象 Key 的一级目录固定为：

| 分区 | 说明 | 典型写入方 | 台账关系 |
|------|------|------------|----------|
| `raw/` | 原始对象留存区，包括上传文件、NAS 文件快照、爬虫 JSON 原始包 | `ingest-gateway`、`source-adapters`、`raw-storage` | `raw_object.object_uri` |
| `staging/` | 临时处理区，包括上传预处理、病毒扫描结果、格式转换临时文件 | `ingest-gateway`、`parse-workers` | 不作为长期事实；必要时由 `job_stage` 记录 |
| `parsed/` | 解析产物区，包括 MinerU 中间 JSON、Markdown、图片、版面信息和解析清单 | `parse-workers` | `parse_artifact.artifact_uri` |
| `normalized/` | 标准化对象区，包括 `normalized_document`、`normalized_record` 和标准化校验结果 | `normalize-service` | `normalized_asset_ref.object_uri` |
| `export/` | 受控导出区，P0 默认不作为主路径 | `nexus-api` / 后续导出任务 | 必须有关联审批和审计 |

#### 3.2.3 Object Key 模板

Object Key 必须稳定、可追溯、避免覆盖，并避免包含敏感业务标题。推荐模板如下：

```text
raw/<source_type>/<source_id>/<yyyy>/<mm>/<dd>/<batch_id>/<raw_object_id>/<checksum_prefix>/<safe_file_id>.<ext>
staging/<job_type>/<job_id>/<stage>/<attempt_no>/<object_id>.<ext>
parsed/<version_id>/<parse_artifact_id>/<artifact_type>/<artifact_name>
normalized/<normalized_type>/<version_id>/<ref_id>/<schema_version>/<checksum_prefix>.json
export/<approval_id>/<job_id>/<artifact_id>.<ext>
```

字段约束：

| 字段 | 约束 |
|------|------|
| `source_type` | 使用架构定义枚举：`file_upload`、`nas_sync`、`crawler_push`、`webhook`、`database`。 |
| `source_id` | 使用 NEXUS 内部 ID，不使用外部系统敏感名称。 |
| `batch_id` | 必须与 `ingest_batch.batch_id` 一致。 |
| `raw_object_id` | 必须与 `raw_object.raw_object_id` 一致。 |
| `version_id` | 必须与 `document_version.version_id` 一致。 |
| `ref_id` | 必须与 `normalized_asset_ref.ref_id` 一致。 |
| `checksum_prefix` | 使用 SHA-256 前 8 到 16 位，只用于定位和排障，完整 checksum 落 PostgreSQL。 |
| `safe_file_id` | 使用内部文件 ID 或清洗后的非敏感名称；不得直接使用包含敏感信息的原始文件名。 |
| `attempt_no` | 从 1 开始递增，用于重试产物隔离。 |

#### 3.2.4 URI 规范

数据库中保存对象引用时使用统一 URI 形式：

```text
s3://<bucket>/<object_key>
```

示例：

```text
s3://nexus-prod-objects/raw/file_upload/ds_001/2026/04/30/bat_9d2f/raw_7c8a/91b4c72a/file_f3e2.pdf
s3://nexus-prod-objects/normalized/document/ver_42/ref_88/schema-v1/9ac2e1d4.json
```

对象 URI 只作为定位引用，不作为权限判定依据。权限判定必须回到 `document_version`、`governance_result`、组织范围、数据分级和调用方上下文。

#### 3.2.5 元数据规范

写入对象时应附加对象元数据，便于排障和离线核验。元数据值不得包含大段原文。

| 元数据 Key | 说明 |
|------------|------|
| `x-amz-meta-nexus-trace-id` | 调用链 ID。 |
| `x-amz-meta-nexus-batch-id` | 接入批次 ID。 |
| `x-amz-meta-nexus-raw-object-id` | 原始对象 ID，适用于 `raw/`。 |
| `x-amz-meta-nexus-version-id` | 资产版本 ID，适用于 `parsed/`、`normalized/`。 |
| `x-amz-meta-nexus-ref-id` | 标准化引用 ID，适用于 `normalized/`。 |
| `x-amz-meta-nexus-checksum-sha256` | 完整 SHA-256。 |
| `x-amz-meta-nexus-content-class` | `raw`、`staging`、`parsed`、`normalized`、`export`。 |
| `x-amz-meta-nexus-schema-version` | 标准化对象 schema 版本，适用于 `normalized/`。 |

### 3.3 使用示例

#### 3.3.1 原始文件写入

```python
from dataclasses import dataclass
from datetime import datetime, timezone


@dataclass(frozen=True)
class RawObjectWriteCommand:
    env: str
    bucket: str
    source_type: str
    source_id: str
    batch_id: str
    raw_object_id: str
    checksum_sha256: str
    safe_file_id: str
    extension: str
    content_type: str
    trace_id: str


def build_raw_object_key(cmd: RawObjectWriteCommand) -> str:
    now = datetime.now(timezone.utc)
    checksum_prefix = cmd.checksum_sha256[:12]
    return (
        f"raw/{cmd.source_type}/{cmd.source_id}/"
        f"{now:%Y/%m/%d}/{cmd.batch_id}/{cmd.raw_object_id}/"
        f"{checksum_prefix}/{cmd.safe_file_id}.{cmd.extension.lstrip('.')}"
    )


def put_raw_object(s3_client, cmd: RawObjectWriteCommand, body: bytes) -> str:
    key = build_raw_object_key(cmd)
    s3_client.put_object(
        Bucket=cmd.bucket,
        Key=key,
        Body=body,
        ContentType=cmd.content_type,
        Metadata={
            "nexus-trace-id": cmd.trace_id,
            "nexus-batch-id": cmd.batch_id,
            "nexus-raw-object-id": cmd.raw_object_id,
            "nexus-checksum-sha256": cmd.checksum_sha256,
            "nexus-content-class": "raw",
        },
    )
    return f"s3://{cmd.bucket}/{key}"
```

落库要求：

1. `put_object` 成功后，必须在同一业务流程中写入或更新 `raw_object` 台账。
2. `raw_object` 至少记录 `object_uri`、`checksum`、`size_bytes`、`mime_type`、`source_id`、`batch_id`、`status`。
3. 如果数据库写入失败，必须通过补偿作业或孤儿对象清理清单处理，不允许静默吞掉。
4. 原始对象只追加不覆盖。重复接入通过 `checksum`、来源主键和幂等键判定，不通过覆盖旧对象解决。

#### 3.3.2 标准化对象写入

```python
import json


def put_normalized_document(
    s3_client,
    *,
    bucket: str,
    version_id: str,
    ref_id: str,
    schema_version: str,
    checksum_sha256: str,
    trace_id: str,
    normalized_document: dict,
) -> str:
    key = (
        f"normalized/document/{version_id}/{ref_id}/"
        f"{schema_version}/{checksum_sha256[:12]}.json"
    )
    payload = json.dumps(normalized_document, ensure_ascii=False).encode("utf-8")
    s3_client.put_object(
        Bucket=bucket,
        Key=key,
        Body=payload,
        ContentType="application/json",
        Metadata={
            "nexus-trace-id": trace_id,
            "nexus-version-id": version_id,
            "nexus-ref-id": ref_id,
            "nexus-schema-version": schema_version,
            "nexus-checksum-sha256": checksum_sha256,
            "nexus-content-class": "normalized",
        },
    )
    return f"s3://{bucket}/{key}"
```

落库要求：

1. 写入 `normalized/` 后，必须创建或更新 `normalized_asset_ref`。
2. 同一 `version_id` 同一时间只能有一个 `status = generated` 的标准化引用。
3. 重新标准化时生成新的 `ref_id` 和对象 Key，旧 `normalized_asset_ref` 标记为 `deprecated`，不覆盖旧对象。
4. 治理、切片、AI 评分和索引只能读取 `normalized_asset_ref` 指向的有效标准化对象。

#### 3.3.3 解析产物写入

```text
parsed/<version_id>/<parse_artifact_id>/mineru-middle-json/result.json
parsed/<version_id>/<parse_artifact_id>/markdown/content.md
parsed/<version_id>/<parse_artifact_id>/images/page_0001_block_0003.png
parsed/<version_id>/<parse_artifact_id>/manifest.json
```

`manifest.json` 建议包含：

| 字段 | 说明 |
|------|------|
| `version_id` | 资产版本 ID。 |
| `raw_object_id` | 来源原始对象 ID。 |
| `parse_artifact_id` | 解析产物 ID。 |
| `mineru_mode` | `pipeline`、`hybrid`、`vlm` 等。 |
| `files` | 解析产物文件列表、checksum、大小和类型。 |
| `created_at` | 生成时间。 |
| `trace_id` | 调用链 ID。 |

#### 3.3.4 读取对象

读取对象必须先查 PostgreSQL 台账，再访问 MinIO：

```text
raw_object_id
  -> query raw_object.object_uri and checksum
  -> permission / operation check
  -> get object from MinIO
  -> verify checksum when used as processing input
```

不得从客户端传入任意 `s3://` URI 后直接读取对象。

### 3.4 禁止事项

1. 禁止将 MinIO 当作主数据库使用；资产状态、版本状态、治理结果、权限策略、审计记录不得只存在对象元数据或 JSON 文件中。
2. 禁止覆盖写入 `raw/`、`parsed/`、`normalized/` 下已形成台账关系的对象；重处理必须创建新对象、新引用或新产物记录。
3. 禁止对象 Key 包含手机号、身份证号、人员姓名、客户名称、密级、API Key、原始敏感文件名或大段业务标题。
4. 禁止绕过 `raw_object` 台账直接把上传文件交给 Worker。
5. 禁止基于 `raw/` 原始对象直接生成正式治理结果；治理正式输入必须是 `normalized_document` 或 `normalized_record`。
6. 禁止将外部可访问的预签名 URL 长期保存到数据库、日志或消息体中；预签名 URL 只可短期下发给已授权主体。
7. 禁止在 `staging/` 对象上建立长期业务引用；`staging/` 只用于可清理临时处理。
8. 禁止在日志中打印对象正文、JSON 原始包全文、MinerU 解析全文或标准化全文。
9. 禁止使用 MinIO bucket policy 代替 NEXUS 权限模型；NEXUS 仍需做 RBAC、ABAC、组织范围、数据分级和脱敏控制。
10. 禁止用删除对象的方式表达资产停用或版本归档；状态必须由 PostgreSQL 主数据表达，对象生命周期另行治理。

---

## 四、Redis 调用契约

### 4.1 定位与职责

Redis 是 NEXUS 的缓存和短期协同组件，用于降低热点查询成本、支撑权限和规则的快速读取、处理短期幂等并发控制，并为工作台展示基础运行状态提供轻量数据来源。

Redis 中的任何数据都必须满足以下条件之一：

1. 可以从 PostgreSQL、MinIO、RabbitMQ 管理状态或服务运行状态重新构建。
2. 过期后不影响最终业务正确性。
3. 丢失后最多造成短期性能下降、重复校验或重新排队，不造成权限泄露、状态丢失或审计缺口。

### 4.2 命名规范

#### 4.2.1 Redis Key 总体格式

```text
nexus:<env>:<domain>:<object_or_purpose>:<id_or_hash>[:<qualifier>]
```

示例：

```text
nexus:prod:auth:permission:caller_1001:user_2001:hash_7a91
nexus:prod:rule:active_set:metadata_governance:v12
nexus:prod:prompt:active_profile:quality_scoring:v4
nexus:prod:asset:current_view:asset_8f3a
nexus:prod:job:heartbeat:worker:parse-01
nexus:prod:ingest:idempotency:caller_1001:idem_9a7c
```

#### 4.2.2 Key 分类

| 类型 | Key 模板 | Value 类型 | 推荐 TTL | 失效触发 |
|------|----------|------------|----------|----------|
| 权限结果缓存 | `nexus:<env>:auth:permission:<caller_id>:<user_id>:<scope_hash>` | JSON | 1 到 5 分钟 | API Key、角色、组织范围、资产分级、权限策略变化 |
| API Key 摘要缓存 | `nexus:<env>:auth:api_key:<key_hash_prefix>` | JSON | 5 到 15 分钟 | API Key 禁用、过期、权限变更 |
| 当前资产读取缓存 | `nexus:<env>:asset:current_view:<asset_id>` | JSON | 1 到 10 分钟 | `VersionAvailable`、`GovernanceChanged`、`IndexProjectionStale` |
| 版本标准化引用缓存 | `nexus:<env>:version:normalized_ref:<version_id>` | JSON | 5 到 30 分钟 | 重新标准化、引用废弃 |
| 规则集缓存 | `nexus:<env>:rule:active_set:<rule_set_type>:v<version>` | JSON | 可长 TTL，建议 1 到 6 小时 | `GovernanceRuleSetPublished` |
| Prompt 配置缓存 | `nexus:<env>:prompt:profile:<profile_id>:v<profile_version>` | JSON | 可长 TTL，建议 1 到 6 小时 | `AIPromptProfilePublished`、禁用配置 |
| 接口响应缓存 | `nexus:<env>:api:response:<route_hash>:<query_hash>` | JSON | 30 秒到 5 分钟 | 相关主数据变更 |
| 幂等处理中间态 | `nexus:<env>:idem:<caller_id>:<idempotency_key>` | JSON / string | 10 分钟到 24 小时 | 数据库幂等结果完成后可缩短 TTL |
| 分布式锁 | `nexus:<env>:lock:<resource_type>:<resource_id>` | string | 5 秒到 5 分钟 | 自动过期；释放时校验 token |
| Worker 心跳 | `nexus:<env>:job:heartbeat:worker:<worker_id>` | JSON | 30 到 120 秒 | Worker 定时刷新 |
| 队列积压快照 | `nexus:<env>:runtime:rabbitmq:queue_depth:<queue_name>` | JSON | 30 到 120 秒 | 运行状态采集器刷新 |

#### 4.2.3 Value 结构规范

Redis Value 推荐使用 JSON，必须包含以下基础字段：

| 字段 | 说明 |
|------|------|
| `schema_version` | 缓存值结构版本，便于灰度升级。 |
| `generated_at` | 缓存生成时间。 |
| `ttl_seconds` | 设计 TTL，便于排查。 |
| `source` | 缓存来源，如 `postgresql`、`rule_publish_event`、`runtime_probe`。 |
| `trace_id` | 生成该缓存的调用链 ID，运行状态类可为空。 |

权限、规则、Prompt 等缓存还应包含版本字段：

| 缓存类型 | 必须字段 |
|----------|----------|
| 权限缓存 | `policy_version`、`caller_id`、`user_id`、`allowed_org_scopes`、`max_level`、`masking_policy_version` |
| 规则集缓存 | `rule_set_id`、`rule_set_type`、`version`、`published_at` |
| Prompt 配置缓存 | `profile_id`、`profile_version`、`prompt_version`、`output_schema_version`、`redaction_policy`、`litellm_model_alias` |

#### 4.2.4 DB 分区规范

生产环境优先通过 ACL 和 Key 前缀隔离，不依赖 Redis DB 编号表达强安全边界。若单实例启用多个 DB，可采用以下约定：

| DB | 用途 |
|----|------|
| 0 | 默认应用缓存。 |
| 1 | 权限、API Key 和组织范围缓存。 |
| 2 | 规则集、Prompt 配置和治理上下文缓存。 |
| 3 | 幂等锁、短期锁和 Worker 心跳。 |
| 4 | 接口响应缓存。 |
| 5 到 15 | 预留。 |

DB 编号不是安全控制。不同服务仍必须使用最小权限 ACL、专用账号和受控网络访问。

### 4.3 使用示例

#### 4.3.1 权限结果缓存

```python
import json
from hashlib import sha256


def permission_cache_key(
    *,
    env: str,
    caller_id: str,
    user_id: str,
    asset_scope_payload: dict,
) -> str:
    scope_hash = sha256(
        json.dumps(asset_scope_payload, sort_keys=True).encode("utf-8")
    ).hexdigest()[:16]
    return f"nexus:{env}:auth:permission:{caller_id}:{user_id}:{scope_hash}"


def get_permission(redis_client, key: str) -> dict | None:
    raw = redis_client.get(key)
    if raw is None:
        return None
    return json.loads(raw)


def set_permission(redis_client, key: str, value: dict) -> None:
    redis_client.set(
        key,
        json.dumps(value, ensure_ascii=False),
        ex=300,
    )
```

缓存值示例：

```json
{
  "schema_version": "permission-cache-v1",
  "generated_at": "2026-04-30T10:15:30Z",
  "ttl_seconds": 300,
  "source": "postgresql",
  "trace_id": "tr_01HY...",
  "caller_id": "caller_1001",
  "user_id": "user_2001",
  "policy_version": "policy-v8",
  "allowed_org_scopes": ["org_001", "org_001.002"],
  "max_level": "L2",
  "masking_policy_version": "mask-v3",
  "decision": "allow_with_masking"
}
```

使用要求：

1. 缓存命中后，返回内容前仍需执行输出字段脱敏。
2. 权限策略、API Key、组织范围、资产分级、治理结果变更后，必须删除相关前缀或提升 `policy_version` 使旧缓存失效。
3. L4 内容不得因为缓存命中而绕过默认脱敏。

#### 4.3.2 规则集缓存

```text
Key:
nexus:prod:rule:active_set:metadata_governance:v12

Value:
{
  "schema_version": "rule-set-cache-v1",
  "generated_at": "2026-04-30T10:20:00Z",
  "ttl_seconds": 21600,
  "source": "GovernanceRuleSetPublished",
  "trace_id": "tr_01HZ...",
  "rule_set_id": "grs_12",
  "rule_set_type": "metadata_governance",
  "version": 12,
  "status": "active",
  "conflict_policy": "highest_level_wins",
  "rules": []
}
```

刷新策略：

```text
GovernanceRuleSetPublished
  -> 删除 nexus:<env>:rule:active_set:<rule_set_type>:*
  -> 加载 PostgreSQL 中最新 active 规则集
  -> 写入新版本缓存
  -> Worker 后续任务按 rule_set_id + version 记录决策日志
```

规则执行不得只依赖 Redis 缓存版本。正式决策日志必须记录 `rule_set_id`、版本、命中规则和冲突原因。

#### 4.3.3 幂等防并发锁

```python
from uuid import uuid4


def acquire_idempotency_lock(
    redis_client,
    *,
    env: str,
    caller_id: str,
    idempotency_key: str,
    ttl_seconds: int = 600,
) -> str | None:
    key = f"nexus:{env}:idem:{caller_id}:{idempotency_key}"
    token = str(uuid4())
    locked = redis_client.set(key, token, nx=True, ex=ttl_seconds)
    return token if locked else None
```

调用要求：

1. Redis 锁只防止短时间并发重复提交。
2. 最终幂等结果必须落 PostgreSQL，例如 `idempotency_key + caller_id + target_version_id`。
3. Redis 锁过期、丢失或 Redis 重启后，系统仍必须能靠数据库唯一约束防止重复有效资产、重复作业或重复治理结果。

#### 4.3.4 Worker 心跳和基础运行状态

```text
Key:
nexus:prod:job:heartbeat:worker:parse-worker-01

Value:
{
  "schema_version": "worker-heartbeat-v1",
  "generated_at": "2026-04-30T10:25:00Z",
  "ttl_seconds": 60,
  "source": "worker",
  "worker_id": "parse-worker-01",
  "worker_type": "parse",
  "hostname": "nexus-worker-1",
  "current_job_id": "job_91",
  "status": "running"
}
```

工作台可以读取此类短期状态展示 Worker 心跳、队列积压和最近错误，但不得把它作为作业成功或失败的最终依据。

### 4.4 禁止事项

1. 禁止把 Redis 作为主状态库；`document_version.version_status`、`job.status`、`governance_result`、`quality_report`、`audit_log` 不得只写 Redis。
2. 禁止缓存 L3/L4 明文、大段标准化正文、完整 Prompt 输入、完整模型响应或 API Key 明文。
3. 禁止使用无 TTL 的业务缓存；规则集和 Prompt 配置即使 TTL 较长，也必须支持发布事件主动失效。
4. 禁止使用 `KEYS nexus:*` 等阻塞命令做线上扫描；需要批量失效时使用版本号、标签集合、`SCAN` 或发布事件驱动。
5. 禁止用 Redis 分布式锁代替数据库唯一约束和事务；当前版本唯一性、标准化引用唯一性、有效质量报告唯一性必须由 PostgreSQL 约束保证。
6. 禁止通过缓存命中绕过权限二次校验和输出脱敏。
7. 禁止在 Redis Key 中放入原始文件名、用户姓名、手机号、证件号、查询原文、问答问题全文或其他敏感内容。
8. 禁止使用 Redis Pub/Sub 承担不可丢失业务事件；可靠任务和关键事件必须走 RabbitMQ + PostgreSQL 状态记录。
9. 禁止生产环境开放无 ACL、无密码或公网可达的 Redis。
10. 禁止将 `FLUSHALL`、`FLUSHDB` 作为常规运维手段；生产配置应禁用或重命名高危命令。

## 五、RabbitMQ 调用契约

### 5.1 定位与职责

RabbitMQ 是 NEXUS 的异步任务代理和事件分发通道，配合 Celery Worker 支撑接入、解析、标准化、AI 治理、规则治理、索引同步、重处理、重治理和 AI 重评分。

RabbitMQ 只负责可靠投递和隔离，不负责保存最终业务结果。Worker 消费消息后必须以 PostgreSQL 中的 `job`、`job_stage`、`document_version`、`normalized_asset_ref`、`governance_decision_log`、`index_manifest` 等记录为准。

P0 部署基线：

1. vhost 使用 `nexus`。
2. 默认任务交换机使用 `nexus.jobs`，类型为 topic。
3. 默认死信交换机使用 `nexus.dlx`，类型为 topic。
4. 队列默认使用 quorum queue。
5. `nexus.*` 队列应用 DLX 策略。
6. 消息体最大大小应控制在 16 MB 以内，业务上应远低于该值；大对象必须通过 MinIO URI 引用。

### 5.2 命名规范

#### 5.2.1 Vhost

| 环境 | vhost |
|------|-------|
| 开发 / 测试 / 生产默认 | `nexus` |

如多环境共用 RabbitMQ 集群，可采用环境后缀：

```text
nexus-dev
nexus-test
nexus-staging
nexus-prod
```

AMQP URL 中 `/nexus` 表示 vhost `nexus`，不是根路径下的目录。

#### 5.2.2 Exchange

| Exchange | 类型 | 用途 |
|----------|------|------|
| `nexus.jobs` | topic | 作业任务、领域事件和可重试任务投递。 |
| `nexus.dlx` | topic | 死信消息投递。 |
| `nexus.audit` | topic，可选 | P1 后如需审计异步化可预留；P0 审计主记录仍应同步或事务内落 PostgreSQL。 |

P0 不建议为每个服务创建独立 exchange。优先通过 topic routing key 和队列绑定区分任务类型。

### 5.3 消息契约

#### 5.3.1 标准消息 Envelope

RabbitMQ 消息体必须使用统一 Envelope：

```json
{
  "schema_version": "nexus-event-v1",
  "event_id": "evt_01J...",
  "event_type": "RawObjectPersisted",
  "occurred_at": "2026-04-30T10:30:00Z",
  "trace_id": "tr_01J...",
  "request_id": "req_01J...",
  "producer": "ingest-gateway",
  "idempotency_key": "idem_8b91",
  "job_id": "job_1001",
  "attempt": 1,
  "payload": {},
  "headers": {
    "tenant_id": "default",
    "env": "prod"
  }
}
```

字段说明：

| 字段 | 必填 | 说明 |
|------|------|------|
| `schema_version` | 是 | 消息结构版本。 |
| `event_id` | 是 | 全局唯一事件 ID，用于去重和审计关联。 |
| `event_type` | 是 | 领域事件或任务类型。 |
| `occurred_at` | 是 | 事件发生时间。 |
| `trace_id` | 是 | 内部链路追踪 ID。 |
| `request_id` | 否 | 外部请求 ID，无外部请求时可为空。 |
| `producer` | 是 | 生产者服务名。 |
| `idempotency_key` | 否 | 幂等键。 |
| `job_id` | 是 | 作业 ID；纯缓存失效事件可为空，但建议仍关联配置变更审计 ID。 |
| `attempt` | 是 | 当前投递或业务重试次数。 |
| `payload` | 是 | 轻量业务载荷。 |
| `headers` | 否 | 环境、租户、优先级等扩展信息。 |

#### 5.3.2 Payload 只传引用

示例：原始对象落库完成事件。

```json
{
  "schema_version": "nexus-event-v1",
  "event_id": "evt_raw_001",
  "event_type": "RawObjectPersisted",
  "occurred_at": "2026-04-30T10:30:00Z",
  "trace_id": "tr_ingest_001",
  "request_id": "req_upload_001",
  "producer": "ingest-gateway",
  "idempotency_key": "caller_1001:upload_20260430_001",
  "job_id": "job_ingest_001",
  "attempt": 1,
  "payload": {
    "batch_id": "bat_001",
    "raw_object_id": "raw_001",
    "source_id": "ds_001",
    "source_type": "file_upload",
    "object_uri": "s3://nexus-prod-objects/raw/file_upload/ds_001/2026/04/30/bat_001/raw_001/91b4c72a/file_001.pdf",
    "checksum_sha256": "91b4c72a...",
    "mime_type": "application/pdf",
    "size_bytes": 1048576
  },
  "headers": {
    "env": "prod",
    "tenant_id": "default"
  }
}
```

禁止将文件二进制、原始 JSON 包全文、标准化正文全文、Prompt 输入全文、模型响应全文放入消息。

#### 5.3.3 消息 Headers

AMQP headers 至少应包含：

| Header | 说明 |
|--------|------|
| `x-nexus-trace-id` | `trace_id`。 |
| `x-nexus-request-id` | `request_id`，如有。 |
| `x-nexus-event-type` | 事件类型。 |
| `x-nexus-schema-version` | 消息 schema 版本。 |
| `x-nexus-idempotency-key` | 幂等键，如有。 |
| `x-nexus-job-id` | 作业 ID。 |

### 5.4 使用示例

#### 5.4.1 生产消息

```python
import json
from datetime import datetime, timezone
from uuid import uuid4


def publish_raw_object_persisted(channel, *, exchange: str, job_id: str, payload: dict, trace_id: str, request_id: str | None):
    event = {
        "schema_version": "nexus-event-v1",
        "event_id": f"evt_{uuid4().hex}",
        "event_type": "RawObjectPersisted",
        "occurred_at": datetime.now(timezone.utc).isoformat(),
        "trace_id": trace_id,
        "request_id": request_id,
        "producer": "ingest-gateway",
        "idempotency_key": payload.get("idempotency_key"),
        "job_id": job_id,
        "attempt": 1,
        "payload": payload,
        "headers": {
            "env": "prod",
            "tenant_id": "default",
        },
    }
    body = json.dumps(event, ensure_ascii=False).encode("utf-8")
    channel.basic_publish(
        exchange=exchange,
        routing_key="nexus.ingest.raw_object_persisted.v1",
        body=body,
        properties={
            "delivery_mode": 2,
            "content_type": "application/json",
            "headers": {
                "x-nexus-trace-id": trace_id,
                "x-nexus-request-id": request_id or "",
                "x-nexus-event-type": "RawObjectPersisted",
                "x-nexus-schema-version": "nexus-event-v1",
                "x-nexus-job-id": job_id,
            },
        },
        mandatory=True,
    )
```

生产者要求：

1. 发布消息前必须先写入业务主记录或作业记录，避免消息先到而消费者查不到事实源。
2. 消息发布失败时，API 不得声称作业已进入异步处理；应返回可重试错误或写入待补偿状态。
3. `event_id`、`job_id`、`idempotency_key` 必须支持消费者去重。

#### 5.4.2 消费消息

```python
def handle_document_parse(event: dict, db_session, minio_client) -> None:
    trace_id = event["trace_id"]
    job_id = event["job_id"]
    payload = event["payload"]

    job = load_job_for_update(db_session, job_id)
    if job.status in {"succeeded", "failed"}:
        return

    mark_job_running(db_session, job_id, trace_id=trace_id)

    raw_object = load_raw_object(db_session, payload["raw_object_id"])
    verify_raw_object_read_allowed(raw_object)

    try:
        artifact_uri = run_mineru_and_put_artifacts(
            minio_client,
            raw_object_uri=raw_object.object_uri,
            version_id=payload["version_id"],
            trace_id=trace_id,
        )
        save_parse_artifact(db_session, job_id=job_id, artifact_uri=artifact_uri)
        mark_job_succeeded(db_session, job_id, trace_id=trace_id)
    except RetryableParseError as exc:
        mark_job_retryable_failed(db_session, job_id, reason=str(exc), trace_id=trace_id)
        raise
    except Exception as exc:
        mark_job_failed(db_session, job_id, reason=str(exc), trace_id=trace_id)
        raise
```

消费者要求：

1. 消费开始、阶段成功、阶段失败、重试和最终结果都必须写入 PostgreSQL 作业记录。
2. 消费成功后再 ACK；可重试异常交给 Celery / RabbitMQ 重试策略；不可重试异常进入失败或死信。
3. 消费者必须具备幂等能力。重复收到同一 `event_id` 或同一 `job_id + task_type` 时，不得重复创建有效版本、有效标准化引用、有效质量报告或重复索引。

#### 5.4.3 重试与死信

推荐策略：

| 错误类型 | 处理方式 |
|----------|----------|
| 瞬时网络错误、LiteLLM 临时超时、RAGFlow 临时不可用 | 自动重试，指数退避，限制最大次数。 |
| 文件格式不支持、checksum 不匹配、Schema 校验失败 | 标记 `failed` 或 `review_required`，不无限重试。 |
| 权限策略阻断、L4 明文调用外部模型被拒绝 | 标记 `review_required` 或 `policy_blocked`，写审计。 |
| 重试耗尽 | 进入 DLQ，并在作业中心展示死信状态、失败阶段和最后错误。 |

DLQ 消息必须能通过 `job_id` 回到作业中心，显示：

1. 失败阶段。
2. 失败原因。
3. 重试次数。
4. 最后一次 `trace_id`。
5. 关联对象：`batch_id`、`raw_object_id`、`asset_id`、`version_id`、`ref_id`。
6. 可执行动作：重试、重处理、重治理、标记需复核。

### 5.5 禁止事项

1. 禁止把 RabbitMQ 当作作业状态库；排队中、处理中、失败、重试、死信、成功等状态必须落 PostgreSQL。
2. 禁止在消息体中传文件二进制、完整原文、完整 JSON 原始包、大段标准化正文、完整 Prompt 输入、完整模型响应或 API Key 明文。
3. 禁止消费者依赖“消息只投递一次”。所有消费者必须按 `event_id`、`job_id`、`idempotency_key` 和数据库唯一约束实现幂等。
4. 禁止无上限重试。每类任务必须定义最大重试次数、退避策略和重试耗尽后的状态。
5. 禁止 Worker 直接把 AI 输出写入正式 `governance_result`；必须经过 Schema 校验、字段白名单、脱敏策略、规则护栏、置信度阈值和状态机判定。
6. 禁止解析 Worker、治理 Worker、索引 Worker 直接切换当前可用版本；`available` 切换必须由 `metadata-service` 在本地事务中完成。
7. 禁止用 RabbitMQ 事件替代审计日志；规则发布、Prompt 发布、权限变更、人工覆盖、版本状态切换等关键动作必须写 `audit_log`。
8. 禁止生产环境使用默认 `guest` 远程访问、共享超级管理员账号或把 RabbitMQ 管理端口暴露到公网。
9. 禁止创建无命名规则的临时队列承载业务任务；临时调试队列不得进入生产配置。
10. 禁止跨服务直接消费对方内部队列并绕过 `nexus-api` 或模块边界；对外业务能力仍由 `nexus-api` 承载。

## 六、跨中间件链路样例

### 6.1 文档接入到标准化

```text
1. API 接收上传请求，生成 request_id / trace_id。
2. ingest-gateway 校验来源、幂等键和文件摘要。
3. MinIO 写入 raw/ 对象，返回 object_uri。
4. PostgreSQL 写入 ingest_batch、raw_object、job。
5. RabbitMQ 发布 nexus.ingest.raw_object_persisted.v1，只传 batch_id、raw_object_id、object_uri、checksum、job_id。
6. parse-worker 消费任务，从 PostgreSQL 查询 raw_object，再读取 MinIO raw/。
7. parse-worker 调用 MinerU，解析产物写 MinIO parsed/，落 parse_artifact。
8. normalize-service 生成 normalized_document，写 MinIO normalized/，落 normalized_asset_ref。
9. Redis 可缓存 version_current_normalized_ref_view，但必须可由 PostgreSQL 重建。
10. 作业中心从 PostgreSQL 查询阶段、状态、失败原因和 trace_id。
```

### 6.2 规则发布到缓存失效

```text
1. 平台/数据管理员发布 governance_rule_set。
2. PostgreSQL 创建新版本规则集，并写 audit_log。
3. RabbitMQ 发布 nexus.rule.rule_set_published.v1。
4. 缓存消费者删除 Redis 中旧 active rule set 缓存。
5. governance-rule 从 PostgreSQL 加载新 active 规则集并写入 Redis 缓存。
6. 后续治理决策记录 rule_set_id、version、matched_rules 和 conflict_reasons。
```

### 6.3 版本可用到索引同步

```text
1. metadata-service 在事务中归档旧 available 版本并激活新 available 版本。
2. PostgreSQL 写 VersionAvailable 事件记录和 audit_log。
3. RabbitMQ 发布 nexus.version.available.v1。
4. ragflow-adapter 消费事件，查询 version_current_normalized_ref_view 和 governance_result。
5. ragflow-adapter 读取 MinIO normalized/ 对象，生成 RAGFlow 投影。
6. 索引状态写入 index_manifest。
7. Redis 删除 asset_current_view、permission filter、search response 等相关缓存。
```

## 七、验收检查清单

| 检查项 | MinIO | Redis | RabbitMQ |
|--------|-------|-------|----------|
| 命名是否符合本契约 | Bucket、Key、URI、元数据 | Key 前缀、TTL、Value schema | vhost、exchange、routing key、queue、task name |
| 是否可追溯 | 对象 URI 可回到台账 | 缓存值有来源和版本 | 消息有 `event_id`、`job_id`、`trace_id` |
| 是否可幂等 | checksum + 台账唯一约束 | 短期锁 + 数据库最终幂等 | 消费者去重 + 数据库唯一约束 |
| 是否保护敏感数据 | Key 不含敏感信息，对象访问受控 | 不缓存明文敏感内容 | 消息只传引用不传正文 |
| 是否符合 v2.2 边界 | 不做主数据事实源 | 不做状态事实源 | 不做作业事实源 |
| 是否支持控制台追踪 | 原始台账、资产详情可定位对象 | 工作台可展示短期运行态 | 作业中心可定位队列、重试、死信 |
| 失败是否可恢复 | 有 checksum、manifest、台账 | 缓存丢失可重建 | 重试耗尽进 DLQ，作业状态可查 |

## 八、实施建议

1. 在 `nexus-api` 和 Worker 共享模块中提供 `ObjectUri`、`RedisKeyBuilder`、`EventEnvelope`、`RoutingKey` 等基础类型，避免各模块手写字符串。
2. 在测试中固定校验对象 Key、Redis Key、Routing Key 和消息 Envelope，防止后续实现漂移。
3. 对 MinIO 写入、Redis 缓存、RabbitMQ 发布分别封装适配器接口，业务代码只依赖 NEXUS 语义方法。
4. 所有适配器都必须接收 `trace_id`，并在日志中输出结构化字段。
5. 首期可先实现最小集合：`raw/`、`parsed/`、`normalized/`，权限缓存，规则缓存，`nexus.jobs`、`nexus.dlx` 和接入到资产化所需队列。
6. 高风险能力，如导出、长期预签名 URL、L4 明文解析、外部模型调用和跨组织访问，必须经过权限与审计门禁后再使用中间件能力。
