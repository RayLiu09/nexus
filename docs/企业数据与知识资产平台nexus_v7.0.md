# 企业数据与知识资产平台 v7.0 — NEXUS

## 一、项目定位

NEXUS 面向企业现有非结构化数据，建设统一的数据沉淀、治理、复用与开放平台，解决企业数据分散、处理重复、管理粗放、使用门槛高的问题。

平台沉淀两类核心资产：

1. 面向业务应用的数据资产
2. 面向大模型和智能应用的知识资产

平台作为企业级数据与知识资产基础设施，统一支撑资产管理和能力开放两类核心场景。

---

## 二、产品结构

平台采用"一个统一数据资产底座 + 两类终端"的产品结构，并集成两个外部系统。

```
┌─────────────────────────────────────────────────────────────────┐
│                      外部系统集成层                               │
│   ┌─────────────────────┐   ┌─────────────────────────────┐    │
│   │   爬虫系统           │   │   RAGFlow 系统               │    │
│   │ 产业政策 / 人才需求  │   │ 检索 / 召回 / 知识组织        │    │
│   └──────────┬──────────┘   └──────────────┬──────────────┘    │
└──────────────┼───────────────────────────── ┼───────────────────┘
               ▼                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                    统一数据资产底座                                │
│                                                                  │
│  数据源接入 → 原始数据持久化 → 作业编排与处理 →                  │
│  资产治理与知识加工 → 权限控制 → 检索与服务开放                   │
│                                                                  │
└────────────────────────┬────────────────────────────────────────┘
                         │
          ┌──────────────┴───────────────┐
          ▼                              ▼
┌──────────────────┐          ┌──────────────────────┐
│  nexus-console   │          │     nexus-api         │
│  数据管理员终端   │          │  业务系统 API 终端     │
└──────────────────┘          └──────────────────────┘
```

### 外部系统集成

**爬虫系统**
定期爬取产业政策数据和行业人才需求数据，作为平台实时动态数据源的输入通道，通过标准数据接口推送至底座接入层。

**RAGFlow 系统**
作为底层切片、索引与检索执行引擎之一，为平台提供数据集管理、Chunking method、子块策略、元数据字段、自动标签与检索执行能力。底座通过 `ragflow-adapter` 向其推送标准化资产和索引字段，通过 `search-service` 调用其检索能力；`nexus-api` 对外开放的仍是 NEXUS 自身的检索与问答接口，而不是直接暴露 RAGFlow 接口。

## 三、统一数据资产底座

统一数据资产底座承载平台核心能力，是整个平台的数据处理与服务基础。

### 3.1 底座整体架构

NEXUS 以 MinerU 作为非结构化文档解析执行引擎，但不将 MinerU 直接等同于整个平台底座，而是采用"控制面 + 执行面"分离的企业级架构：控制面负责数据源接入、元数据、作业、权限、审计与运维；执行面负责解析、标准化、RAGFlow 集成、知识加工和索引检索执行。这样既能发挥 MinerU 在文档解析上的优势，也能补齐企业数据资产平台应具备的持久化、治理与运营能力。

底座按数据流向划分为六个层级：

```
原始数据源
    │
    ▼
[数据源接入层]
上传网关 / NAS 同步 / 爬虫推送 / 数据库同步 / Webhook / 接入校验 / 来源登记
    │
    ▼
[原始数据持久化层]
原始对象存储 / 原始包留存 / 校验摘要 / 版本快照 / 接入台账 / 数据留痕
    │
    ▼
[作业编排与处理层]
作业中心 / 消息队列 / 重试补偿 / MinerU 解析 Worker / 标准化资产生成 / RAGFlow 集成编排
    │
    ▼
[资产治理与知识加工层]
分类分级 / 标签治理 / 版本管理 / 生命周期 / 质量复核 / 知识加工管道
    │
    ▼
[权限、检索与服务开放层]
RBAC / 可见范围 / 审计 / RAGFlow 检索 / nexus-console / nexus-api
```

横向支撑模块统一覆盖以上各层：

- 统一元数据中心：沉淀数据源、文档、版本、作业、产物、索引状态等核心实体
- 安全与审计中心：统一处理身份鉴别、访问控制、脱敏、操作审计与合规留痕
- 运维观测中心 `ops-observability`（待定）：统一采集日志、指标、链路追踪、失败告警、容量与成本数据，一期预留模块边界和扩展点，不把具体实现固化到单一技术栈

#### 控制面与执行面职责

| 平面 | 核心模块 | 职责 |
|------|---------|------|
| 控制面 | 接入网关、数据源注册中心、元数据中心、作业中心、权限服务、审计服务、运维控制台 | 负责接入管理、任务状态持久化、配置下发、权限判定、审计与运营 |
| 执行面 | 接入适配器、预处理 Worker、MinerU 解析 Worker、标准化 Worker、RAGFlow 集成 Worker、索引 Worker | 负责实际的数据搬运、解析、标准化资产生成、RAGFlow 数据集同步和索引构建 |

#### MinerU、NEXUS 与 RAGFlow 职责划分

| 组件 | 职责边界 | 不承担职责 |
|------|---------|-----------|
| MinerU | 文档解析、版面恢复、结构化中间产物输出、质量信号输出 | 资产标准化、跨源统一契约、权限治理、索引管理 |
| NEXUS | 原始留存、元数据治理、标准化资产定义、作业编排、权限审计、知识资产加工 | 文档底层 OCR/版面识别执行 |
| RAGFlow | 标准化资产后的 Chunking method 执行、子块切分、元数据挂载、索引构建、检索执行 | 企业资产主数据、原始留存、资产版本治理、权限主策略 |

#### 一期需落地的工程模块

| 模块 | 作用 | 核心产出 |
|------|------|---------|
| 接入网关 `ingest-gateway` | 接收上传、批量推送、同步任务请求 | 接入任务单、批次号、幂等键 |
| 数据源适配器 `source-adapters` | 适配 NAS、爬虫、数据库、Webhook 等不同来源 | 标准化接入事件 |
| 原始对象存储 `raw-storage` | 保存原文件、原始 JSON 包、原始媒体引用 | 原始对象 URI、校验摘要 |
| 元数据中心 `metadata-service` | 管理文档、版本、来源、产物、索引、质量等实体 | 统一资产主数据 |
| 作业编排中心 `job-orchestrator` | 驱动异步处理、重试、补偿、回调和状态机 | 作业状态、执行日志、失败事件 |
| MinerU 解析集群 `parse-workers` | 负责 PDF/Office/图片等文档解析 | Markdown、middle-json、图片、质量指标 |
| 标准化处理服务 `normalize-service` | 对 MinerU 输出和结构化同步数据进行统一契约转换、清洗与校验 | `normalized_document`、`normalized_record`、清洗结果 |
| RAGFlow 集成服务 `ragflow-adapter` | 将标准化资产映射到 RAGFlow 数据集、Chunking method、子块策略、元数据字段和标签集合 | 数据集映射关系、切片同步清单、`index_manifest` |
| 检索编排服务 `search-service` | 调用 RAGFlow 检索能力，执行权限过滤、重排、引用回写和问答上下文组织 | 检索结果、问答上下文、来源引用 |
| 权限与审计服务 `iam-audit-service` | 统一处理 RBAC、可见范围、审计留痕 | 授权策略、审计记录 |
| 运维观测服务 `ops-observability`（待定） | 指标采集、日志聚合、告警与容量分析 | SLA 监控、错误追踪、成本看板 |

#### 可扩展性设计原则

一期底座建设须遵循以下设计原则，确保二期及后续阶段的能力可平滑叠加，不触发架构重构：

**1. 先落原始库，后进入处理链路**
任何来源的数据必须先完成来源登记、完整性校验和原始留存，再进入解析与治理流程。原始数据只追加、不覆盖，保证审计、回放与重处理能力。

**2. 任务状态外置持久化**
作业生命周期由 NEXUS 作业中心统一持久化管理，不依赖单个解析服务进程内存状态。MinerU 只承担解析执行职责，平台负责任务可恢复、重试与补偿。

**3. 数据源接入插件化**
接入层以"数据源适配器"模式设计，每类数据源对应独立适配器，共享接入状态机、校验规则和幂等逻辑。二期新增平台业务数据接入时，只需增加适配器实现。

**4. 统一结果契约**
底座不让下游系统直接依赖 MinerU 的原始输出格式，而是在解析后统一转换为平台标准文档契约和知识切片契约，避免不同后端和版本升级引发连锁改造。

**5. 存储与计算解耦**
对象存储保存原始与处理产物，关系库保存元数据与作业状态，消息队列驱动异步执行，检索与向量索引由 RAGFlow 管理。存储层与执行层松耦合，便于扩容与替换。

**6. 运维与安全能力内建**
日志、指标、链路追踪、权限、审计、脱敏在架构设计时统一纳入，而不是作为上线前临时补丁补加。

底座统一管理以下核心存储：

| 存储类型 | 用途 | 选型 |
|---------|------|---------|
| 关系型数据库 | 数据源配置、资产元数据、版本记录、作业状态、标签、权限、审计 | PostgreSQL |
| 对象存储 | 原始文件、原始 JSON 包、解析产物、标准化结果、知识加工附件 | MinIO（私有化）/ 云对象存储 |
| 消息队列 | 接入任务、解析任务、切片任务、索引任务、失败重试与死信队列 | Kafka / RabbitMQ / Redis Streams |
| 缓存 | 幂等键、热点元数据、权限结果、检索缓存 | Redis |
| 搜索与向量索引 | 文档切片检索、向量召回、混合召回 | RAGFlow（基于 Elasticsearch + 向量引擎） |
| 监控与日志存储 | 指标、日志、链路追踪、告警与 SLA 统计 | `ops-observability`（待定），预留 Prometheus / Loki / ELK / OpenTelemetry 接入位 |

### 3.2 数据源接入与原始数据持久化

企业级数据资产平台的接入原则不是"收到后直接解析"，而是"统一接入、统一登记、统一留存、统一编排"。因此，3.2 章节同时定义数据源接入层和原始数据持久化层。

#### 3.2.1 接入对象与接入模式

平台统一接入三类数据源，覆盖外部知识文档、动态爬取数据和内部平台业务数据：

| 数据源类型 | 典型数据内容 | 格式 | 接入方式 | 原始持久化策略 | 后续处理路径 |
|-----------|------------|------|---------|---------------|-------------|
| 静态知识文档 | 产业信息、行业报告、教学标准、人才培养方案、教材、教案、习题、实训指导书 | PDF、Word、Excel、PPT、TXT、Markdown、图片 | 文件上传 / 目录批量导入 / NAS 目录同步 | 原文件落 `raw` 区，保存 SHA-256、原始路径、上传主体、批次号 | 进入 MinerU 解析链路 |
| 课程媒体资源 | 微课视频、动画、虚拟仿真软件、应用包 | MP4、应用包、外部链接 | NAS 同步 / 清单导入 | 文件原位托管或对象存储引用，平台保存元数据和访问引用 | 默认不走 MinerU，仅做元数据治理 |
| 实时动态数据源 | 产业政策、行业人才需求、岗位招聘数据、院校专业布点数据 | JSON、Excel | 爬虫系统推送 / 批量接口导入 | 原始 JSON 包或文件落 `raw` 区，保留来源 URL、抓取时间、抓取规则版本 | 根据内容类型进入解析或结构化清洗链路 |
| 平台业务数据（二期） | 用户信息、课程班级信息、学习行为、教学行为、能力评估结果 | 数据库表、CDC 事件、Webhook JSON | 数据库同步 / Webhook / 批量导入 | 保留源表快照标识、增量游标、同步批次和原始事件包 | 不经 MinerU，直接进入结构化标准化链路 |

支持的文档格式：PDF、Word（`.docx/.doc`）、PPT（`.pptx/.ppt`）、Excel（`.xlsx/.xls`）、纯文本（`.txt/.md`）、图片（`.jpg/.png`，含扫描件）。视频、应用包等非文档类资源一期以元数据接入为主。

#### 3.2.2 接入层工程模块

接入层按标准网关 + 适配器模式实现：

| 模块 | 职责 | 说明 |
|------|------|------|
| 上传网关 | 接收单文件、批量文件、目录导入请求 | 负责文件校验、大小限制、格式白名单、幂等键生成 |
| NAS 同步适配器 | 监听指定目录或按计划扫描目录 | 生成增量清单，避免重复导入 |
| 爬虫推送适配器 | 接收外部爬虫系统批次推送 | 支持 `POST /ingest/batch`，返回批次号和处理作业号 |
| 数据库同步适配器 | 二期接入 I 博导、实训产品数据 | 支持全量初始化、增量游标和失败续传 |
| Webhook 适配器 | 接收实时行为和事件推送 | 统一转为标准接入事件 |
| 预校验模块 | 病毒扫描、空文件检测、MIME 检测、编码探测、文件哈希计算 | 不合格对象在接入阶段直接拦截 |
| 来源登记模块 | 记录来源系统、上传组织、提交人、批次号、业务域、默认分级初值 | 形成接入台账与审计起点 |
| 去重判定模块 | 基于文件哈希、来源主键、来源版本号判断是否重复 | 支持"跳过重复"和"新版本入库"两种策略 |

统一接入契约应至少包含以下字段：`source_type`、`source_id`、`batch_id`、`origin_uri`、`file_name`、`mime_type`、`checksum`、`submitted_by`、`org_scope`、`business_domain`、`ingest_time`。所有适配器最终都要转换成这一标准契约后再进入后续处理。

#### 3.2.3 原始数据持久化策略

原始数据持久化层采用"对象存储 + 元数据台账"双轨模式，确保所有输入均可回溯、可重放、可重处理。

**对象存储分区设计**

| 分区 | 内容 | 设计原则 |
|------|------|---------|
| `raw/` | 原始文件、原始 JSON 包、外部系统原始推送内容 | 只追加不覆盖，作为唯一可信原始留存 |
| `staging/` | 解压、格式转换、OCR 前临时预处理产物 | 可按生命周期策略自动清理 |
| `parsed/` | MinerU 输出的 Markdown、middle-json、图片、模型输出等解析产物 | 保留解析版本和后端信息 |
| `normalized/` | 平台统一标准文档对象、切片、质量报告、加工附件 | 作为下游消费的稳定契约层 |

**元数据台账设计**

| 实体 | 作用 |
|------|------|
| `data_source` | 数据源注册信息，维护接入方式、负责人、同步策略 |
| `ingest_batch` | 一次批量导入或推送的批次记录 |
| `raw_object` | 原始对象台账，记录原始对象 URI、校验值、来源主键、版本信息 |
| `document_asset` | 平台侧文档资产主实体，承接后续分类、权限、生命周期 |
| `document_version` | 文档版本实体，关联原始对象和处理结果 |
| `ingest_event_log` | 接入阶段所有关键事件和错误记录 |

**持久化规则**

- 原始对象不可覆盖，只允许新增版本。
- 同一来源对象重复接入时，优先基于来源主键 + 版本号判断；来源主键缺失时退化为文件哈希判重。
- 视频、仿真软件等大文件可原位托管，平台保存原位 URI、校验值和访问策略，不强制复制入库。
- 爬虫 JSON 推送不只保留清洗后的字段，也要保留原始包，以便后续规则升级时可重放解析。
- 二期平台业务数据同步时，除结构化结果外仍需保留源表批次标识、同步窗口和原始事件包摘要。

#### 3.2.4 接入状态管理

接入状态机如下：

```
待接收 → 已登记 → 已校验 → 已落原始库 → 待编排 → 处理中 → 已资产化
                              │
                              ├────────→ 接入失败
                              ├────────→ 校验失败
                              └────────→ 需人工复核
```

说明：

- "已登记"表示来源、批次、责任主体、业务域等基础信息已入台账
- "已落原始库"表示原始对象已持久化，具备可回放能力
- "待编排"表示原始对象已具备进入作业中心的条件
- "需人工复核"主要用于格式异常、重复冲突、权限范围不明等情况

### 3.3 作业编排与处理层

作业编排与处理层负责将已持久化的原始对象转化为可治理、可检索、可服务的数据资产与知识资产。该层是一套企业级异步处理系统，不是单一解析服务。

#### 3.3.1 作业编排中心

平台以持久化作业中心统一驱动处理链路，不把任务状态托管给 MinerU 自身的进程内队列。这样可以避免单点重启导致任务丢失，并支持批量任务拆分、失败重试和人工干预。

**作业类型定义**

| 作业类型 | 说明 |
|---------|------|
| `ingest_validate` | 接入后执行格式校验、病毒扫描、哈希计算、重复判断 |
| `document_parse` | 触发 MinerU 对文档类资产执行解析 |
| `structured_sync` | 同步结构化业务数据，不经过 MinerU |
| `normalize_document` | 将解析结果转换为平台标准文档契约 |
| `rag_sync_prepare` | 生成 RAGFlow 数据集映射、Chunking method 映射和同步包 |
| `metadata_enrich` | 抽取关键词、专业标签、地区、时效、质量评分等元数据 |
| `index_build` | 将标准化资产及其索引字段同步至 RAGFlow 并触发索引构建 |
| `reprocess` | 因规则升级、解析失败或人工复核而触发的重处理任务 |

**作业中心必须具备的能力**

- 持久化队列和状态机：作业状态写入数据库，队列用于异步分发
- 幂等控制：同一原始对象同一版本同一处理模板只允许一个有效作业实例
- 自动重试与死信队列：瞬时失败自动重试，持续失败进入人工处理队列
- 批量拆分与汇聚：大批次导入可拆分为单文档任务，并在索引阶段汇总状态
- 事件通知：作业完成、失败、待复核时可回调外部系统或写入事件总线

**作业状态机**

```
已创建 → 已排队 → 处理中 → 成功
                    │
                    ├────────→ 可重试失败 → 重试中
                    ├────────→ 不可重试失败
                    └────────→ 待人工复核
```

#### 3.3.2 MinerU 解析引擎

MinerU 在本方案中的定位是"非结构化文档解析执行引擎"，负责把 PDF、Office、图片等文档转换为结构化中间产物，而不承担企业平台的任务控制面职责。平台可利用 MinerU 的多后端能力、版面分析能力和中间结构化结果能力，作为整个非结构化处理链路的核心解析基座。

**MinerU 在底座中的职责**

| 职责 | 说明 |
|------|------|
| 文档解析 | 处理 PDF、扫描件、图片、Office 转换后的文档对象 |
| 版面与阅读顺序恢复 | 识别标题、正文、表格、图片、公式、列表等结构，恢复阅读顺序 |
| 结构化中间产物输出 | 输出 Markdown、middle-json、图片等解析产物，供标准化层继续处理 |
| 质量信号输出 | 提供页数、块数量、是否含表格/公式、解析异常等质量信号 |

**解析后端策略**

| 场景 | 解析后端 | 说明 | 触发策略 |
|------|---------|------|---------|
| 标准文本类 PDF（制度文件、标准规范） | Pipeline | 成本低、速度快，适合大批量处理 | 默认首选 |
| 复杂排版 / 图文混排（教材、研究报告） | Hybrid | 同时利用原生文本提取与多模态能力，降低结构误判 | Pipeline 质量不足时升级 |
| 扫描件 / 低质量图片（历史材料、证书） | VLM | 对 OCR 和复杂版面更稳健，但 GPU 成本更高 | 根据文件类型或质量规则直达 |

**部署方式**

- MinerU 以独立解析服务或 Worker 集群方式部署，对外只暴露内网接口
- Worker 池按 CPU 解析与 GPU 解析分组，分别处理 Pipeline 与 Hybrid/VLM 任务
- 可使用 MinerU Router 做 Worker 健康检查与负载分发，但任务 ID、状态、重试和回调必须由 NEXUS 作业中心外置持久化管理
- 解析产物统一写入 `parsed/` 分区，不能只停留在解析服务本地临时目录

**调用关系**

```
作业中心 → 选择解析 Worker → 调用 MinerU → 产物写入 parsed/
                                   │
                                   └── 回传解析状态、后端类型、质量指标
```

#### 3.3.3 标准化处理、RAGFlow 集成与资产化入库

标准化资产是后续知识资产精细化加工、检索服务、权限控制和版本治理的统一输入件。本节定义标准化处理规范、切片规范、元数据规范、RAGFlow 集成方式和资产入库规则。v7.0 采用明确的职责边界：MinerU 只负责文档解析；NEXUS 负责标准化资产契约、治理规则和资产入库；RAGFlow 承载标准化资产后的切片执行、索引构建和检索执行。

**一、处理职责边界**

| 环节 | 承载组件 | 固定职责 |
|------|---------|---------|
| 文档解析 | MinerU | 将 PDF、Office、图片转换为结构化中间产物 |
| 标准化处理 | `normalize-service` | 将多源输入转换为统一标准资产契约 |
| 元数据主治理 | `metadata-service` | 维护分类、分级、标签、版本、状态和血缘 |
| 切片执行与索引构建 | `ragflow-adapter` + RAGFlow | 将标准化资产映射到数据集、Chunking method、子块策略、索引任务 |
| 检索与问答编排 | `search-service` | 权限过滤、检索重排、引用回写和问答上下文组织 |

**二、RAGFlow 承载范围**

RAGFlow 承载切片执行、子块切分、元数据字段挂载、标签增强、索引构建、检索测试和检索执行。标准化处理、资产版本治理、原始数据留存、权限主策略、审计留痕和资产发布状态由 NEXUS 承担。RAGFlow 数据集是检索执行层对象，不是企业资产主数据对象。

**三、标准化资产规范**

平台定义两类标准化对象：

| 对象 | 适用来源 | 说明 |
|------|---------|------|
| `normalized_document` | PDF、Word、PPT、Markdown、HTML、图片、扫描件 | 面向文档类输入的统一标准对象 |
| `normalized_record` | JSON、Excel、数据库同步、Webhook 结构化事件 | 面向结构化或半结构化输入的统一标准对象 |

`normalized_document` 顶层字段规范如下：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `schema_version` | string | 是 | 标准契约版本，v7.0 起固定维护 |
| `asset_id` | string | 是 | `document_asset` 主键 |
| `version_id` | string | 是 | `document_version` 主键 |
| `source_type` | enum | 是 | `file_upload` / `nas_sync` / `crawler_push` / `platform_sync` |
| `source_ref` | object | 是 | 来源主键、来源 URI、批次号、来源系统 |
| `content_type` | enum | 是 | `document` / `slide_deck` / `table_sheet` / `web_record` / `media_meta` |
| `title` | string | 否 | 标题；缺失时由标题抽取或来源文件名补足 |
| `language` | string | 是 | 主语言编码，默认 `zh-CN` 或检测结果 |
| `toc` | array | 否 | 目录树，记录标题层级和顺序 |
| `blocks` | array | 是 | 标准内容块数组 |
| `body_markdown` | string | 否 | 标准化 Markdown 表达，供 RAGFlow 和下游消费 |
| `attachments` | array | 否 | 图片、表格快照、附件引用 |
| `metadata` | object | 是 | 来源、业务、时间、治理等元数据 |
| `governance` | object | 是 | 分类、分级、组织范围、状态 |
| `quality` | object | 是 | 质量评分、异常项、人工复核状态 |
| `lineage` | object | 是 | 原始对象、解析产物、处理链路追踪信息 |

标准内容块 `blocks` 采用统一块模型：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `block_id` | string | 是 | 块唯一标识 |
| `block_type` | enum | 是 | `title` / `heading` / `paragraph` / `list` / `table` / `figure` / `formula` / `quote` / `code` / `kv` / `record` / `media_ref` |
| `page_no` | int | 否 | 文档页码；非分页来源可为空 |
| `seq_no` | int | 是 | 文档内顺序号 |
| `heading_path` | array | 否 | 当前块所属标题路径 |
| `text` | string | 否 | 文本表达 |
| `html` | string | 否 | 表格、复杂块的 HTML 表达 |
| `attributes` | object | 否 | 表格行列、图片引用、公式表达等附加属性 |
| `source_locator` | object | 是 | 页码、段落号、来源记录主键或表格单元定位信息 |

`normalized_record` 适用于爬虫 JSON、Excel 行记录、数据库同步数据，统一字段包括：`schema_version`、`asset_id`、`version_id`、`source_type`、`record_type`、`record_key`、`record_body`、`metadata`、`governance`、`quality`、`lineage`。结构化记录进入知识加工时可直接转为 `knowledge_chunk`，不经过 MinerU。

**四、多源兼容标准**

| 来源类型 | 标准化方式 | 固定规则 |
|---------|-----------|---------|
| PDF / 扫描件 | MinerU 解析后转为 `normalized_document` | 保留页码、标题路径、块顺序、表格 HTML、图片引用 |
| Word / PPT | 转换后由 MinerU 解析，再标准化 | 保留章节层级、页面顺序、页内块顺序 |
| Markdown / HTML / TXT | 直接进入标准化服务 | 统一编码、标题层级、链接和附件引用表达 |
| Excel / CSV | 转为 `normalized_record` 或表格型 `normalized_document` | 每个 sheet 保留 sheet 名、字段名、主键列、行号定位 |
| JSON / 爬虫事件 | 转为 `normalized_record` | 保留原始字段、来源 URL、抓取时间、发布机构 |
| 平台同步数据 | 转为 `normalized_record` | 保留源表主键、增量游标、同步批次、组织范围 |
| 视频 / 媒体资源 | 生成 `media_meta` 类型标准对象 | 仅入元数据，不参与 MinerU 解析 |

**五、标准化处理流程**

1. 原始对象进入 `ingest_validate`，完成格式校验、哈希计算、来源登记和去重判定。
2. 文档类对象进入 `document_parse`，由 MinerU 生成 `parse_artifact`。
3. `normalize_document` 将 `parse_artifact` 或结构化同步数据转换为 `normalized_document` / `normalized_record`。
4. 标准化服务执行清洗规则、块模型映射、标题路径重建、附件引用整理和治理字段补足。
5. `metadata_enrich` 生成元数据草稿、标签草稿和质量评分。
6. `rag_sync_prepare` 根据资产类型和切片画像生成 RAGFlow 数据集映射、Chunking method 映射和索引字段映射。
7. `index_build` 通过 `ragflow-adapter` 将标准化资产同步至 RAGFlow，触发切片和索引构建。
8. 构建结果、失败原因、切片统计和索引状态回写 `metadata-service` 与作业中心。

**六、切片规范**

切片不再作为单一服务内部自由实现，而是采用平台统一的切片规范，由 `ragflow-adapter` 映射到 RAGFlow 的 Chunking method 与子块策略。

平台切片画像定义如下：

| `chunk_profile` | 适用资产 | 平台规则 | RAGFlow 映射 |
|-----------------|---------|---------|-------------|
| `manual_chapter` | 教材、标准、方案、制度文件 | 以标题路径为主切分，保留章节上下文 | `Manual` |
| `general_text` | 报告、一般文档、网页正文 | 按 token 窗口切分，保留固定重叠 | `General` |
| `paper_article` | 论文、研究报告 | 按摘要、章节、图表说明切分 | `Paper` 或 `General` |
| `laws_policy` | 法规、政策、制度规范 | 按条款、章节、条文编号切分 | `Laws` |
| `presentation` | 课件、演示文稿 | 按页级和页面要点切分 | `Presentation` |
| `table_record` | Excel、CSV、结构化表 | 按记录、字段组或表格块切分 | `Table` |
| `qa_asset` | 问答语料、FAQ | 一问一答为最小切片单元 | `Q&A` |
| `tag_asset` | 标签词典、技能标签体系 | 一标签或一术语为最小切片单元 | `Tag` |

切片执行规则如下：

- 默认父块控制在 600-1,200 tokens，父块重叠 10%-15%。
- 子块用于重排和问答上下文压缩，控制在 150-300 tokens。
- 表格和流程类内容不按自然段切分，而按行组、列组或步骤节点切分。
- 图片、公式、代码块不单独成为主切片，默认挂靠在最近的正文块或标题块下。
- 单切片必须保留 `source_locator`、`heading_path`、`asset_id`、`version_id`、`level`、`org_scope`。

`knowledge_chunk` 字段规范如下：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `chunk_id` | string | 是 | 切片主键 |
| `asset_id` | string | 是 | 资产主键 |
| `version_id` | string | 是 | 版本主键 |
| `chunk_profile` | string | 是 | 平台切片画像 |
| `chunk_level` | enum | 是 | `parent` / `child` |
| `heading_path` | array | 否 | 标题路径 |
| `chunk_text` | string | 是 | 切片文本 |
| `chunk_summary` | string | 否 | 摘要或增强表述 |
| `metadata_refs` | object | 是 | 分类、分级、标签、组织范围、来源定位 |
| `index_status` | enum | 是 | `pending` / `indexed` / `failed` |

**七、元数据抽取规范**

元数据采用"治理必填字段 + 检索增强字段 + 来源追踪字段"三层结构：

| 分组 | 字段 | 生成方式 |
|------|------|---------|
| 治理必填字段 | 数据域、资产类型、分级、组织范围、版本状态、来源类型、采集时间 | 接入登记 + 治理规则计算 |
| 内容字段 | 标题、关键词、专业领域、岗位、学历层次、地区、应用场景 | 标准化处理 + `metadata_enrich` |
| 质量字段 | 解析成功率、标题抽取状态、表格数量、图片数量、异常块数量、质量评分 | `normalize-service` + 质量规则 |
| 追踪字段 | 批次号、原始对象 URI、解析后端、RAGFlow 数据集 ID、索引分区 | 作业中心 + `ragflow-adapter` |

元数据规则如下：

- 治理必填字段缺失时资产状态不得进入 `published`。
- 检索字段允许先生成草稿，审核通过后写入正式元数据。
- 平台主元数据以 `metadata-service` 为准，RAGFlow 中的 metadata 只承载检索执行所需的投影字段。
- 所有标签、分级、组织范围字段在资产、切片和索引侧必须保持一致，不允许出现三套不同口径。

**八、RAGFlow 集成模式**

v7.0 采用"NEXUS 标准化资产 + RAGFlow 切片索引"的集成方式：

1. `normalize-service` 输出 `normalized_document.body_markdown`、`blocks`、`metadata` 和 `governance`。
2. `ragflow-adapter` 根据 `chunk_profile` 创建或映射 RAGFlow 数据集。
3. `ragflow-adapter` 将 `body_markdown` 或 `normalized_record` 渲染内容作为数据集输入，将分类、分级、标签、组织范围、版本状态写入 metadata 字段。
4. RAGFlow 依据 Chunking method、子块策略、metadata 字段和标签集合执行切片与索引。
5. 切片统计、索引状态和失败信息回写 `index_manifest`。

该模式下：

- 标准化处理不放在 RAGFlow 中执行。
- 原始对象不以 RAGFlow 数据集作为唯一留存。
- RAGFlow 承载切片与索引执行完全成立。
- 资产版本治理、权限主策略和审计留痕仍由 NEXUS 负责。

**九、质量评估与人工复核**

质量评估结果写入 `quality_report`，包含以下字段：`parse_score`、`normalize_score`、`metadata_score`、`chunk_score`、`manual_review_status`、`issues`。当出现以下任一情况时，资产进入 `待人工复核`：

- 正文缺失或标题路径重建失败
- 必填治理字段缺失
- 切片数量异常
- 解析后端错误或 RAGFlow 索引构建失败
- 高敏资产的分级和组织范围冲突

**十、资产化入库规则**

| 产物 | 入库位置 | 说明 |
|------|---------|------|
| `raw_object` | `raw/` + 元数据台账 | 原始可信留存 |
| `parse_artifact` | `parsed/` | MinerU 解析产物 |
| `normalized_document` / `normalized_record` | `normalized/` + 元数据库 | 后续知识加工统一输入 |
| `quality_report` | 元数据库 | 质量治理与复核 |
| `knowledge_chunk` | 元数据库 + RAGFlow 数据集投影 | 检索与知识加工输入 |
| `index_manifest` | 元数据库 | 索引分区、状态、同步时间、失败原因 |

说明：二期 D5/D6 平台业务数据不经过 MinerU 解析，但复用同一套标准化契约、元数据规范、RAGFlow 集成和资产入库规则，以保证整个底座在工程模型上保持统一。

### 3.4 分类分级与标签治理

本节基于公司实际数据资产清单，建立与业务价值链对齐的分类体系、差异化分级规则和多维标签体系，并明确这些治理规则如何落在 `document_asset`、`document_version`、`normalized_document`、`knowledge_chunk` 等平台主数据对象上，形成可执行、可审计、可索引同步的治理方案。

#### 3.4.1 数据资产全景与业务价值链

公司数据资产沿以下业务价值链流转，各环节数据具有明确的依赖关系：

```
产业行业数据
    │ 确定区域产业需求与政策方向
    ▼
岗位职业数据
    │ 明确岗位工作任务与能力要求
    ▼
专业建设数据
    │ 将岗位能力转化为专业培养目标
    ▼
课程与教材数据
    │ 支撑课程内容开发与教学实施
    ▼
教学行为与实训数据
    │ 记录真实教学过程与学习轨迹
    ▼
能力评价与就业匹配
```

分类体系以此价值链为主轴设计，确保分类结构与业务使用逻辑一致。

---

│   └── D5-6 实训操作行为数据
│
└── D6 能力评价与就业数据域
    ├── D6-1 能力评估结果
    └── D6-2 职业匹配度数据
```

分类体系由 `metadata-service` 统一维护，并作为后续分级、标签、权限、检索分区和知识加工的共同基础。一级分类主要作用于 `document_asset`，版本、标准化文档和知识切片默认继承所属资产的分类信息。

**分类维护规则：**
- 分类体系在第 1 周由业务专家与数据管理员联合制定，写入 `metadata-service` 后作为治理基准
- 新增数据集默认归入最近匹配的二级分类，三级分类可按需扩展
- 一级、二级分类调整需经数据管理员审批，并触发标签规则、权限策略和索引分区的增量重算
- 三级分类调整由上传部门提出、管理员确认，调整结果自动同步至相关版本、切片和索引清单

#### 3.4.3 数据分级规则

平台数据按开放程度和敏感性分为四级。分级不只作用于"数据集名称"，而是作用于资产对象本身，并随版本、切片、索引结果一路继承和校验。

| 级别 | 定义 | 典型数据 | 可见范围 | 访问控制 |
|------|------|---------|---------|---------|
| **L1 公开** | 官方发布、可公开引用的外部数据 | 国家职业大典、国家专业教学标准、政策文件 | 全员 + 授权外部用户 | 无需审批，直接访问 |
| **L2 内部** | 企业内部通用，不涉及个人或院校隐私 | 行业研究报告、岗位招聘数据、内部教材、课程资料 | 全员 | 登录后直接访问 |
| **L3 受限** | 特定部门或院校范围内使用，含院校私有数据 | 各院校人才培养方案、院校实训数据、院校专业布点 | 按院校 / 部门授权 | 需角色 / 院校权限，跨院校访问需审批 |
| **L4 机密** | 含个人身份信息或高度敏感行为数据 | 用户个人信息（姓名 / 手机号 / 邮箱）、学生学习行为明细、教师行为明细 | 指定角色（数据管理员 / 系统管理员） | 需明确授权，操作全程审计 |

**分级继承与覆盖规则：**
- `document_asset` 在接入登记或结构化同步阶段赋予初始分级，默认基于数据域、来源类型和业务规则自动判定
- `document_version`、`normalized_document` 默认继承所属资产分级；若某一版本新增敏感内容，可单独升级分级
- `knowledge_chunk` 默认继承文档版本分级；若切片中包含更高敏感内容，可由规则引擎或人工复核提升分级
- `ragflow-adapter` 只同步允许被检索的内容，并将分级、组织范围、脱敏策略写入索引元数据
- 个人数据字段（姓名、手机号等）无论所在资产分级如何，在 API 输出时默认脱敏，L4 明文字段仅指定角色可见

| 数据域 | 默认初始分级 |
|--------|-----------|
| D1 产业行业知识域 | L1 / L2（官方发布 → L1，内部积累 → L2） |
| D2 岗位职业知识域 | L1 / L2（官方发布 → L1，招聘爬取 → L2） |
| D3 专业建设知识域 | L1 / L3（官方标准 → L1，院校私有方案 → L3） |
| D4 课程教学知识域 | L2 / L3（内部教材 → L2，院校私有资料 → L3） |
| D5 教学行为与平台数据域 | L3 / L4（课程信息 → L3，个人行为明细 → L4） |
| D6 能力评价与就业数据域 | L3 / L4（汇总报告 → L3，个人评估结果 → L4） |

#### 3.4.6 自动标签与人工校正机制

**自动标签规则**

标准化处理和 `metadata_enrich` 作业完成后，系统基于接入契约、标准化文档内容和知识切片语义自动生成标签草稿，结果写入标签草稿区，等待管理员确认：

| 标签维度 | 自动识别规则 |
|---------|------------|
| 专业领域 | 标题、目录路径、章节关键词和知识切片语义联合判定（如含"电子商务"→ 电子商务标签） |
| 学历层次 | 文档标题、来源路径、模板字段关键词匹配（如含"高职"→ 高职专科标签） |
| 地域范围 | 正文、附件、来源 URL 中的省市名称识别 |
| 数据来源 | 接入方式和来源系统自动赋值（上传 → 内部积累；爬虫推送 → 爬虫采集；官方 URL → 官方发布） |
| 时效状态 | 新版本生效时默认赋值"现行有效"，旧版本自动切换为"历史存档" |
| 更新周期 | 按数据域默认值自动赋值，并可结合来源规则生成复采周期初值 |

**人工校正流程**

```
标准化资产入库
    │
    ▼
metadata_enrich 生成标签草稿
    │
    ▼
管理员在 nexus-console 标签审核页查看草稿
    │
    ├── 确认：标签写回 metadata-service，触发增量索引更新
    │
    ├── 修订：调整标签值后确认，保留修订痕迹
    │
    └── 驳回：标记为"待复核"，返回 reprocess 作业队列
```

批量入库场景支持按数据域、接入批次、上传组织进行批量确认。所有人工校正动作都写入审计日志，并与资产版本绑定留痕。

#### 3.4.7 数据资产与知识资产的关系模型

这是平台数据一致性的核心设计，明确原始对象、数据资产、知识资产和索引对象之间的衍生关系：

```
raw_object（原始对象）
    │  1:N
    ▼
document_asset（资产主实体）
    │  1:N
    ▼
document_version（资产版本）
    │  1:N               │ 1:1
    │                    └────────→ normalized_document（标准文档对象）
    │
    ├────────→ parse_artifact（MinerU 解析产物）
    │
    └────────→ knowledge_chunk（知识切片）
                         │ 1:N
                         ├────────→ knowledge_asset_version（问答语料 / 流程语料 / 图谱 / 评价标准）
                         └────────→ index_manifest（索引分区与索引状态）
```

**版本联动机制**：
- 原始对象更新时，系统创建新的 `document_version`，旧版本不覆盖，只变更状态
- 分类、分级、标签默认从 `document_asset` 继承到 `document_version` 和 `knowledge_chunk`，必要时可在版本级或切片级覆盖
- 新版本处理完成后，自动触发切片重建、知识资产重算和 `index_manifest` 增量更新
- 旧版本切片与知识资产标记为"历史版本"，时效状态自动变更为"历史存档"，默认不参与检索，但保留供审计和回溯查询
- 任意检索结果、问答结果和知识加工产物都必须能追溯到 `document_version` 和 `raw_object`

### 3.5 知识资产精细化加工

分类分级完成后，资产仅处于"可治理"状态，尚不能直接支撑 AI 应用和上层业务系统的使用需求。本节描述如何以 `normalized_document`、`knowledge_chunk` 和结构化业务数据为输入，通过作业中心、知识加工服务、LLM 管道和人工审核流，将治理后的资产精细化为**可直接消费的知识资产**。

#### 3.5.1 加工定位：从数据资产到知识资产

知识资产加工不是直接基于原始解析结果进行"临时拼装"，而是基于底座已经标准化、可追溯、可控权的资产对象进行二次构建。加工层的目标是将非结构化、半结构化内容，按不同应用场景所需的知识形态进行再加工，同时保留来源、版本、审核状态和权限范围：

```
标准化资产层
├── normalized_document
├── knowledge_chunk
└── structured_sync 数据
    │
    ▼
┌───────────────────────────────────────────────────┐
│              知识资产精细化加工层                    │
│                                                   │
│  job-orchestrator（加工编排 / 审核流 / 重处理）    │
│       +                                           │
│  知识加工服务（切片增强 / 图谱构建 / 评价标准化）    │
│       +                                           │
│  LLM 加工管道（抽取 / 生成 / 结构化）              │
│       +                                           │
│  人工校验（教学服务中心 / 教育研究院）               │
└───────────────────────────────────────────────────┘
    │
    ▼
知识数据资产（可直接消费）
├── 检索知识库            → AI 助教、智能检索
├── 问答语料（SFT）        → 模型微调、客服替代
├── 流程语料（Agent）      → AI Agent 编排、自动生成
├── 知识图谱              → 课程推荐、专业规划
└── 评价标准库            → 自动评分、学情分析
```

所有知识资产在平台中都应具备统一的四类属性：来源可追溯、版本可管理、权限可继承、状态可审核。

#### 3.5.2 目标知识资产清单

以下 16 项知识资产是平台需产出的完整目标，按加工类型分组：

##### 管道一：RAG 检索知识库构建（A 类资产）

```

##### 管道二：问答语料生成（SFT 训练数据，A 类资产）

    ▼

[产业]──涵盖──>[行业]──产生需求──>[岗位]

├── 岗位←→技能：从职业能力分析表中映射

### 3.6 权限与可见范围控制

权限治理由 `iam-audit-service`、`metadata-service`、`ragflow-adapter`、`search-service` 和 `nexus-api` 协同实现。平台采用 RBAC（基于角色的访问控制）+ ABAC（基于属性的访问控制）+ 资产分级过滤的复合权限模型，保证权限规则在资产层、索引层和 API 层保持一致。

#### 权限控制对象

| 对象 | 控制粒度 | 作用位置 |
|------|---------|---------|
| 身份主体 | 用户、API 调用方、系统连接器、后台作业账号 | 认证层 |
| 功能资源 | 控制台菜单、管理操作、API 能力 | 应用层 |
| 资产资源 | `document_asset`、`knowledge_asset`、批次、分类目录 | 元数据层 |
| 内容片段 | `knowledge_chunk`、检索结果、问答上下文 | 检索层 |
| 敏感字段 | 姓名、手机号、邮箱、学号、行为明细字段 | API 输出层 |

#### 权限配置维度

| 维度 | 说明 |
|------|------|
| 角色 | 管理员、数据管理员、普通用户、只读用户、API 调用方 |
| 组织范围 | 企业部门、院校、项目组、业务域 |
| 资产分级 | L1-L4 分级控制访问和脱敏策略 |
| 资产分类 | D1-D6 数据域、二级分类、资产类型 |
| 使用目的 | 检索查看、下载导出、审核校正、二次加工、外部开放 |
| 字段敏感性 | 对姓名、手机号、邮箱、行为明细等敏感字段执行掩码或禁出 |
| 审批状态 | 对跨院校、跨部门、高敏资产访问执行临时授权或审批放行 |

#### 权限执行机制

- 权限策略统一存储在 `iam-audit-service`，资产的分级、组织范围和标签存储在 `metadata-service`
- `ragflow-adapter` 在构建索引时，将 `level`、`org_scope`、`asset_type`、`version_status` 等过滤字段同步写入索引元数据
- `search-service` 查询时先做策略求值，再将可见范围编译成底层检索过滤条件
- `nexus-api` 在返回结果前执行字段级脱敏和下载权限校验
- 所有放行、拒绝、脱敏、审批动作都写入审计日志，并与用户、资产、时间和请求 ID 绑定

#### 权限校验流程

```
请求进入（Console 操作 / API 调用）
    │
    ▼
身份认证（Token 校验）
    │
    ▼
构建访问上下文（角色 / 组织 / 数据域 / 请求目的）
    │
    ▼
策略求值（功能权限 + 资产权限 + 院校隔离）
    │
    ▼
元数据范围校验（是否有目标资产的访问权）
    │
    ▼
检索过滤编译（分级 + org_scope + 标签 + 版本状态）
    │
    ▼
结果返回前脱敏（L4 字段掩码 / 禁止导出）
    │
    ▼
审计记录（操作人、时间、请求 ID、访问资产、结果条数、脱敏动作）
```

### 3.7 检索、召回与知识组织

本模块由 NEXUS 的 `ragflow-adapter`、`search-service` 和知识组织服务承载，RAGFlow 作为底层索引与检索执行引擎之一。平台负责索引组织、权限过滤、结果重排、引用回溯和知识上下文编排，底层引擎负责全文检索与向量检索执行。

#### NEXUS 检索服务与 RAGFlow 的角色

RAGFlow 作为底层检索执行组件，与底座的交互关系如下：

| 方向 | 内容 |
|------|------|
| metadata-service → ragflow-adapter | 提供 `normalized_document`、`knowledge_chunk`、标签、分级、组织范围、版本状态 |
| ragflow-adapter → RAGFlow | 推送标准化资产投影、索引文档和过滤字段，触发切片与索引构建 |
| search-service → RAGFlow | 提交查询词、过滤条件、召回策略、Top-K 参数 |
| RAGFlow → search-service | 返回候选切片、相关度分数、索引命中信息 |
| search-service → nexus-api | 输出重排后的结果、来源引用、上下文片段和可见范围说明 |

#### 索引组织原则

| 维度 | 作用 |
|------|------|
| 数据域 / 资产类型 | 用于区分 D1-D6 不同知识空间 |
| 资产分级 | 用于权限过滤和高敏数据隔离 |
| `org_scope` | 用于院校 / 部门 / 项目级隔离 |
| 版本状态 | 控制现行有效、历史存档、待复核是否参与检索 |
| 标签维度 | 支撑专业、学历、地域、应用场景等过滤 |
| 索引清单 `index_manifest` | 记录索引分区、构建时间、版本号和同步状态 |

#### 检索能力

平台提供四类检索模式，可按场景配置：

| 检索模式 | 原理 | 适用场景 |
|---------|------|---------|
| 关键词检索 | 基于全文索引（Elasticsearch） | 精确词汇查找、编号查询 |
| 语义检索 | 基于向量相似度（嵌入模型） | 意图理解、相似内容发现 |
| 混合检索 | 关键词 + 语义融合，RRF 重排 | 通用业务问答，默认配置 |
| 结构化过滤检索 | 分类、标签、图谱节点、组织范围与文本检索联合过滤 | 图谱辅助检索、受限知识空间检索 |

#### 召回与重排

1. 查询解析：识别查询意图、数据域、标签过滤条件和上下文偏好。
2. 权限过滤编译：根据用户角色、组织范围、资产分级生成底层检索过滤条件。
3. 初始召回：按检索模式返回 Top-K 候选切片。
4. 结果重排：基于 Re-ranker 模型和业务规则对候选结果重新排序。
5. 知识组织：将相关切片按章节、来源文档、图谱节点或业务主题聚合，构建结构化上下文。
6. 结果封装：输出结果时附带来源文档、版本号、切片 ID、标签、权限说明和引用位置。

#### 知识资产质量基线

建立以下评估指标，在阶段 1 完成后建立基线，作为后续优化的参照：

| 指标 | 定义 | 目标基线 |
|------|------|---------|
| 解析成功率 | 成功完成解析的文档占比 | ≥ 95% |
| 切片质量评分 | 平均切片质量评分 | ≥ 70 分 |
| 索引更新时效 | 新版本资产完成入库到索引可查询的时间 | ≤ 15 分钟 |
| 检索召回率 | 试点场景测试集的召回率 | ≥ 80%（Top-5） |
| 权限误放行率 | 未授权内容被错误返回的比例 | 0 |
| 答案可追溯率 | 问答结果中有明确来源引用的比例 | 100% |

## 四、两类终端职责

### nexus-console

面向数据管理员、数据治理人员和平台运营人员，承担控制面入口职责。其核心定位不是简单"看文档结果"，而是统一承载数据源管理、资产治理、作业运维、权限审计和运营观测。

**核心功能模块：**

| 模块 | 主要功能 |
|------|---------|
| 数据源管理 | 数据源注册、上传入口配置、NAS 同步配置、爬虫推送配置、平台同步源配置、接入批次查看 |
| 原始数据台账 | 原始对象查询、校验摘要查看、重复判定结果、原始留存状态、回放入口 |
| 作业中心 | `job` 列表、状态监控、失败重试、人工复核、重处理触发、批次执行进度查看 |
| 解析与加工配置 | MinerU 后端策略、切片策略、标准化模板、元数据抽取规则、知识加工模板配置 |
| 分类分级管理 | 分类体系维护（D1-D6）、分级规则配置、标签审核（草稿确认 / 修订 / 驳回）、批量标注 |
| 资产目录管理 | `document_asset` / `document_version` 查询、标准化文档查看、版本比对、时效状态管理 |
| 知识资产管理 | 问答语料、流程模板、图谱节点、评价标准等知识资产的审核、发布和回溯 |
| 权限与审批 | 角色管理、用户授权、组织隔离配置、临时授权、跨院校审批流配置 |
| 运营与审计 | 接入统计、作业统计、检索调用分析、索引同步状态、操作审计日志、L4 访问专项审计 |
| 运维观测 | 服务健康检查、失败告警、SLA 看板、容量趋势、异常事件追踪 |

**终端职责边界：**
- `nexus-console` 负责控制面管理，不直接承担文档解析和检索执行
- 解析、切片、索引等重计算任务统一通过作业中心异步下发
- 所有高敏操作必须经过权限校验和审计记录，控制台仅提供配置与治理入口

### nexus-api

面向上层业务系统、智能应用和外部集成方，以标准化接口开放底座能力。其职责是屏蔽底层元数据、作业、索引和权限实现细节，对外提供稳定的资产查询、检索问答、接入编排和治理能力接口。

**API 接口分类：**

API 接口按调用性质分为两类，采用不同的架构设计：

**同步查询类（低延迟，有 SLA 要求）：**

| 接口 | 说明 | 目标响应时间 |
|------|------|------------|
| `GET /assets` | 资产列表查询，支持分类（D1-D6）、标签、时间过滤 | < 200ms |
| `GET /assets/{id}` | 资产详情及元数据查询 | < 100ms |
| `GET /assets/{id}/versions` | 资产版本列表与状态查询 | < 200ms |
| `POST /search` | 检索召回（关键词 / 语义 / 混合），权限过滤自动生效 | < 1s |
| `POST /qa` | 知识问答（由 search-service 编排，底层可调用 RAGFlow） | < 5s |
| `POST /auth/verify` | 权限校验 | < 50ms |

**异步处理类（高吞吐，队列调度）：**

| 接口 | 说明 |
|------|------|
| `POST /ingest` | 提交接入任务，完成原始落库和作业编排，返回 `job_id` |
| `POST /ingest/batch` | 爬虫系统批量推送接口，返回批次号和首个 `job_id` |
| `POST /jobs/reprocess` | 触发指定资产版本重处理 |
| `GET /jobs/{id}` | 查询作业处理状态、阶段结果和失败原因 |
| `POST /jobs/{id}/retry` | 重新触发失败作业 |
| `POST /knowledge-assets/publish` | 发布审核通过的知识资产版本 |

**通用能力：**
- 鉴权：所有接口需携带 API Key 或 JWT Token
- 限流：按调用方独立限流（默认 100 QPS / 调用方）
- 幂等：接入类接口支持 `idempotency_key`，避免重复提交
- 审计：每次调用记录调用方、接口、参数摘要、响应时间、结果条数和脱敏动作
- 监控：接口可用性、响应时间 P99、错误码分布纳入运营统计面板
- 版本化：统一以 `/v1/` 发布，对底层索引和解析引擎升级保持接口稳定

**接口设计原则：**
- 对外暴露的是资产、知识资产和作业对象，不直接暴露底层解析服务内部状态
- 同步接口只读元数据和检索结果，所有重处理、重建索引、重新加工统一走异步 `job`
- API 返回结果必须包含来源引用、版本信息和可见范围说明，满足可追溯要求

## 五、核心价值

1. 将分散的文档、爬虫数据和平台业务数据统一接入并原始留存，形成可回放、可审计、可重处理的企业数据资产底座。
2. 以 MinerU 为非结构化解析执行引擎，补齐持久化作业中心、标准化文档契约、知识切片和元数据治理能力，形成可长期演进的工程底座。
3. 为 FLOW 及后续业务系统提供统一的数据资产、知识资产和检索问答服务出口，减少重复建设和系统间接口碎片化。
4. 通过分类、分级、标签、权限、审计和索引治理，让知识可用的同时保持可控、可管、可追溯，满足企业级安全与治理要求。
5. 推动数据管理从项目式加工转向平台化、标准化、服务化运营，为 AI 助教、智能检索、智能助手、图谱和评价等场景持续供给高质量知识资产。

## 九、部署方案与容量规划

本章给出 v7.0 的两类部署场景：适合小规模数据场景的单节点部署方案，适合中等数据规模的集群化部署方案。处理能力指标按以下基线测算：

- 文档平均页数：30 页
- 文档结构占比：标准文本型 70%，图文混排型 20%，扫描件 10%
- 解析后端占比：Pipeline 70%，Hybrid 20%，VLM 10%
- 切片基线：父块 800 tokens、子块 200-300 tokens
- 指标口径：为架构容量规划基线，不作为生产 SLA 承诺值

### 9.1 单节点部署方案

#### 9.1.1 适用范围

- 原始文档规模：不超过 10 万份
- 原始数据容量：不超过 2 TB
- 月增量：不超过 1 万份文档
- 使用场景：一期试点、部门级资产平台、有限用户规模的知识检索服务

#### 9.1.2 拓扑结构

单节点承载以下模块：

- `ingest-gateway`
- `metadata-service`
- `job-orchestrator`
- `parse-workers`
- `normalize-service`
- `ragflow-adapter`
- `search-service`
- `iam-audit-service`
- `nexus-api`
- `nexus-console`
- PostgreSQL、Redis、MinIO、RAGFlow
- `ops-observability`（待定）预留部署位

#### 9.1.3 硬件规格

| 资源项 | 规格 |
|------|------|
| CPU | 16 Core |
| 内存 | 64 GB |
| 系统盘 | 500 GB SSD |
| 数据盘 | 2 TB NVMe SSD |
| GPU | 1 张 48 GB 显存 GPU |
| 网络 | 1 Gbps |

#### 9.1.4 处理能力指标

| 指标 | 能力基线 |
|------|---------|
| 标准文本型文档解析吞吐 | 2,500-3,500 份/日 |
| 图文混排文档解析吞吐 | 600-900 份/日 |
| 扫描件解析吞吐 | 250-450 份/日 |
| 标准化资产生成吞吐 | 3,500 份/日 |
| RAGFlow 切片与索引构建 | 60-90 万切片/日 |
| 检索接口吞吐 | 20-40 QPS |
| 问答接口吞吐 | 2-5 QPS |
| 索引更新时效 | 小批量同步 5-15 分钟 |

#### 9.1.5 资源约束

- PostgreSQL、MinIO、RAGFlow 与 MinerU 共机部署时，CPU 与内存竞争明显
- VLM 占比提升时，GPU 成为主瓶颈
- 单节点不具备高可用能力，节点故障将导致解析、检索与控制面同时中断

### 9.2 集群化部署方案

#### 9.2.1 适用范围

- 原始文档规模：10 万至 60 万份
- 原始数据容量：2 TB 至 12 TB
- 月增量：1 万至 6 万份文档
- 使用场景：中等规模知识资产平台、2-5 个业务域并行使用、持续增长的知识检索与问答服务

#### 9.2.2 三节点拓扑结构

集群总计 3 台服务器，按“1 个管控与元数据节点 + 1 个 MinerU 解析节点 + 1 个检索与索引节点”部署，角色固定、资源隔离、对象存储跨节点分布。

- 1 号节点（管控与元数据节点）：`ingest-gateway`、`metadata-service`、`job-orchestrator`、`iam-audit-service`、`nexus-api`、`nexus-console`、PostgreSQL 主实例
- 2 号节点（MinerU 解析节点）：`parse-workers`、`normalize-service`、`metadata_enrich` Worker、MinerU Router（可选），承担文档解析与标准化主算力
- 3 号节点（检索与索引节点）：`ragflow-adapter`、`search-service`、RAGFlow、Redis、重排服务，承担切片执行、索引构建与在线检索
- 共享组件：MinIO 采用 3 节点分布式部署，`raw/`、`parsed/`、`normalized/` 产物跨节点留存；`ops-observability`（待定）以独立容器组或命名空间预留

#### 9.2.3 节点角色与硬件规格

| 节点 | 角色 | 部署模块 | 硬件规格 |
|------|------|---------|---------|
| 1 号节点 | 管控与元数据 | `ingest-gateway`、`metadata-service`、`job-orchestrator`、`iam-audit-service`、`nexus-api`、`nexus-console`、PostgreSQL | 24 Core / 96 GB RAM / 500 GB SSD / 2 TB NVMe |
| 2 号节点 | MinerU 解析与标准化 | `parse-workers`、`normalize-service`、`metadata_enrich` Worker、MinerU Router（可选） | 32 Core / 128 GB RAM / 1 TB SSD / 4 TB NVMe / 1 张 48 GB 显存 GPU |
| 3 号节点 | 检索、索引与缓存 | `ragflow-adapter`、`search-service`、RAGFlow、Redis、重排服务 | 24 Core / 128 GB RAM / 1 TB SSD / 6 TB NVMe |
| 跨节点存储 | 对象存储 | MinIO 3 节点分布式部署 | 三节点各提供独立数据卷，统一构成对象存储池 |
| 集群网络 | 东西向通信 | API 调用、索引同步、对象存储复制 | 10 Gbps |

#### 9.2.4 处理能力指标

| 指标 | 能力基线 |
|------|---------|
| 标准文本型文档解析吞吐 | 6,000-9,000 份/日 |
| 图文混排文档解析吞吐 | 1,200-2,000 份/日 |
| 扫描件解析吞吐 | 400-700 份/日 |
| 标准化资产生成吞吐 | 8,000-12,000 份/日 |
| RAGFlow 切片与索引构建 | 150-250 万切片/日 |
| 检索接口吞吐 | 60-120 QPS |
| 问答接口吞吐 | 6-12 QPS |
| 索引更新时效 | 小批量同步 5-12 分钟，大批量同步 15-40 分钟 |

#### 9.2.5 集群特性与可用性边界

- 3 节点方案实现控制面、解析面、检索面的物理隔离，不把 MinerU、RAGFlow、PostgreSQL 压在同一节点
- MinerU 独占 48 GB 显存 GPU，Hybrid / VLM 任务不与检索服务争抢显存资源
- MinIO 跨 3 节点分布式部署，原始对象、解析产物和标准化产物具备跨节点留存能力
- 该方案的高可用边界清晰：对象存储具备节点级冗余；控制面、元数据主库和 RAGFlow 主服务采用主实例运行方式，节点故障后通过服务重启、数据恢复和作业补偿恢复业务
- `ops-observability`（待定）不单独占用节点，预留为独立服务组，后续可平滑拆分为观测专用节点

#### 9.2.6 扩容规则

| 扩容触发项 | 触发条件 | 扩容动作 |
|-----------|---------|---------|
| 解析排队时长 | 连续 3 天超过 20 分钟 | 新增第 4 节点作为专用解析节点，2 号节点继续承担 MinerU 主解析 |
| GPU 利用率 | 连续峰值超过 80% | 在 2 号节点增加第 2 张 48 GB 显存 GPU，或将 VLM 任务拆分至新增解析节点 |
| 检索延迟 | P95 超过 800 ms | 新增检索节点，拆分 RAGFlow 与 `search-service` 负载 |
| 元数据库负载 | PostgreSQL CPU 连续超过 70% 或活跃连接数持续过高 | 拆出独立数据库节点，1 号节点保留控制面服务 |
| 数据盘使用率 | 连续超过 70% | 扩容 MinIO 数据卷或增加对象存储节点 |
| 问答并发 | 连续超过 10 QPS | 增加 `search-service` 与重排服务实例，并扩展检索节点算力 |
