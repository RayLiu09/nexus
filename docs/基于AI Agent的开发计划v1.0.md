# 基于 AI Agent 的开发计划 v1.0

## 一、结论

基于 NEXUS v2.2 最终版 P0 范围，采用“AI Agent 作为开发主力，人工进行 Review 和关键决策”的方式，建议采用如下交付策略：

| 交付口径 | 建议周期 | 按 2026-05-06 启动的目标日期 | 管理建议 |
|----------|----------|------------------------------|----------|
| P0 可演示版 | 4 周 | 2026-06-02 | 可用于中期汇报，不建议作为正式验收口径。 |
| P0 可验收版 | 6 周 | 2026-06-16 | 作为内部交付目标，要求主链路、权限、审计、AI 可追溯和核心异常流可验收。 |
| P0 稳妥交付版 | 8 周 | 2026-06-30 | 作为对外交付窗口，保留 2 周用于缺陷收敛、环境联调、演示打磨和正式验收。 |

推荐管理口径：对内按 6 周完成 P0 可验收版推进，对外保留 8 周交付窗口。

原因：

1. AI Agent 可以显著加速 CRUD、API、页面、迁移、测试、文档和样例代码产出。
2. 业务样本、治理规则、Prompt 口径、LiteLLM / MinerU / RAGFlow 环境联调和验收仍需要人工确认。
3. 权限误放行率 0、追溯率 100%、关键动作审计覆盖 100% 不能只依赖生成代码，必须通过人工 Review 和测试证据确认。
4. 2 周缓冲不应被提前消耗为 P1/P2 需求，应保留给质量、集成和正式汇报。

## 二、本次 Review 发现与优化动作

| Review 发现 | 风险 | 优化动作 |
|-------------|------|----------|
| 原方案强调 AI 主力开发，但 Agent 任务边界不够细 | Agent 容易一次生成大块代码，带来架构漂移和 Review 成本上升 | 引入“任务包”机制，每个任务包限定上下文、文件范围、禁区、验收证据和 Review 清单。 |
| 人工 Review 只有原则描述，缺少门禁 | Review 可能流于代码风格检查，漏掉权限、审计、状态机、AI 治理边界 | 设置人工 Review Gate：数据模型、AI 调用、规则引擎、权限审计、版本状态、RAGFlow 集成都必须人工批准。 |
| 周期压缩后容易牺牲测试 | AI 生成代码能跑通主链路，但异常流和边界条件不足 | 让 AI Agent 同步生成单元测试、契约测试、E2E 样例和验收证据，测试与功能同 PR 交付。 |
| 前后端可能并行失配 | API、字段、状态枚举和页面状态不一致 | 采用 contract-first：先冻结 OpenAPI / Pydantic Schema / 状态枚举 / UI 状态口径，再并行实现。 |
| 业务专家投入有限 | Prompt、规则、样本和验收口径若滞后，会阻塞 AI 治理链路 | 保持业务专家 75 小时投入，但集中到第 1、3、4、5、6、8 周关键确认点。 |
| AI Agent 可能引入 P1/P2 或旧架构概念 | 范围膨胀，或违反 v2.2 边界 | 每个任务包必须声明禁区：不做企业 IAM、不做 `llm-gateway`、不拆独立 `ai-governance-orchestrator`、不新增反向指针。 |
| 外部系统联调可能低估 | LiteLLM、MinerU、RAGFlow 任一不稳定都会影响验收 | 第 2 周前打通 MinerU 最小适配，第 3 周前确认 LiteLLM 模型别名，第 5 周前完成 RAGFlow 最小索引闭环。 |

## 三、开发组织方式

### 3.1 人员与 AI Agent 分工

| 角色 | 工作定位 | 主要职责 |
|------|----------|----------|
| 后台开发 / 项目负责人 / AI 工程师 | 人工负责人 | 任务拆解、架构决策、后端 Review、AI 治理 Review、接口验收、里程碑汇报。 |
| 前端开发 | 人工负责人 | 前端架构、页面 Review、交互验收、演示路径稳定性。 |
| 业务专家 | 关键节点参与 | 样本、字段、规则、Prompt、质量评分、检索问题、权限用例和验收意见。 |
| Backend Agent | AI 开发主力 | API、数据库模型、迁移、作业、治理、权限、审计、测试。 |
| Frontend Agent | AI 开发主力 | 页面、组件、表单、状态、抽屉、弹窗、接口联调。 |
| Test Agent | AI 质量辅助 | 单元测试、契约测试、E2E 用例、权限用例、回归清单。 |
| Docs Agent | AI 文档辅助 | API 说明、部署说明、规则说明、AI 治理说明、验收材料。 |
| Review Assistant Agent | AI Review 辅助 | 对照 `ARCHTECT.md`、`SPEC.md`、Prototype 文档检查架构边界和需求覆盖。 |

说明：AI Agent 可以并行产出，但最终合并和关键决策必须由人工负责人确认。

### 3.2 工作流

```text
需求 / 架构契约确认
    -> 人工拆分任务包
    -> AI Agent 生成代码、测试和文档
    -> AI Review 辅助检查契约偏差
    -> 人工 Review 高风险点
    -> 小批量合并
    -> 自动化测试 / 演示路径验证
    -> 每周可演示切片
    -> 每 2 周里程碑汇报
```

关键原则：

1. 先契约，后代码。重要 API、状态枚举、数据模型和页面字段先冻结。
2. 小任务包，小 PR。单个任务包建议 0.5 到 1.5 天内可完成。
3. 代码、测试、文档同步交付。没有测试或验收证据的 AI 产物不得进入主分支。
4. 人工 Review 看边界和风险，不把时间浪费在普通样板代码上。
5. 每周必须保留一个可演示切片，避免最后两周集中联调。

## 四、AI Agent 任务包模板

每个任务包必须使用以下结构：

```text
任务名称：
背景来源：
  - 引用 ARCHTECT.md / SPEC.md / Prototype v2.2 的具体章节或约束。

目标：
  - 本任务要完成的用户价值或工程能力。

范围：
  - 允许修改的模块、文件、API、页面或测试。

禁区：
  - 不允许引入企业 IAM。
  - 不允许开发 llm-gateway。
  - 不允许新增独立 ai-governance-orchestrator。
  - 不允许新增 document_asset.current_version_id。
  - 不允许新增 document_version.normalized_ref_id。
  - 不允许让 AI 输出绕过规则护栏直接进入 governance_result。

交付物：
  - 代码。
  - 数据库迁移。
  - 测试。
  - API / 页面说明。
  - 审计或追溯证据。

验收方式：
  - 可运行测试命令。
  - 可演示路径。
  - 关键断言。
```

## 五、人工 Review Gate

### 5.1 必须人工批准的变更

| Gate | 触发条件 | 人工 Review 重点 |
|------|----------|------------------|
| 数据模型 Gate | 新增或修改主数据表、状态字段、关系字段 | 是否违反单向引用、是否新增反向指针、是否满足唯一约束和审计要求。 |
| AI 治理 Gate | LiteLLM 调用、Prompt 配置、AI 输出采纳、质量评分 | 是否仍依赖 LiteLLM，Prompt 是否版本化，L3/L4 是否脱敏，AI 输出是否经过 Schema 和规则护栏。 |
| 规则引擎 Gate | 规则表达式、规则发布、冲突处理 | 是否禁止任意代码执行，规则输入是否来自标准化对象，冲突是否可追踪。 |
| 权限审计 Gate | 检索、QA、API Key、组织范围、L4 脱敏 | 是否先鉴权再返回内容，权限误放行是否可测试，关键动作是否审计。 |
| 状态流 Gate | `processing`、`available`、`review_required`、`archived`、`disabled`、`failed` | 是否只允许一个 `available` 版本，复核条件是否正确，索引状态是否一致。 |
| RAGFlow 集成 Gate | 索引、切片、检索、QA 引用 | 是否能回查 `index_manifest`，结果是否可追溯到版本、标准化引用、chunk 和原始对象。 |
| 验收 Gate | 里程碑前和最终验收前 | 是否满足 12 个 E2E 用例、权限误放行率 0、追溯率 100%、审计覆盖 100%。 |

### 5.2 Review Checklist

每次合并前至少检查：

1. 是否违反 v2.2 架构禁区。
2. 是否新增 P1/P2 范围。
3. 是否有对应测试。
4. 是否有失败路径和异常状态处理。
5. 是否有审计和 trace_id。
6. 是否有权限过滤和脱敏。
7. 是否能从结果追溯到资产版本、标准化引用和原始对象。
8. 是否更新 API、字段或页面说明。

## 六、质量门禁

| 门禁 | 要求 |
|------|------|
| 编译与静态检查 | 后端、前端基础检查必须通过。 |
| 单元测试 | 状态机、规则执行、AI 输出校验、权限过滤、幂等逻辑必须覆盖。 |
| 契约测试 | `/v1` API 请求/响应、Pydantic Schema、前端字段映射必须一致。 |
| E2E 测试 | 覆盖静态文档入库、爬虫 JSON、AI 自动采纳、冲突复核、规则重治理、权限隔离、QA 引用、重处理、索引失败恢复、幂等、无 DingTalk 闭环、AI 重评分。 |
| 安全测试 | L4 默认脱敏、API Key 不明文日志、跨组织默认拒绝、规则表达式不能执行任意代码。 |
| 审计测试 | Prompt、规则、权限、版本状态、AI 采纳、人工覆盖、API Key 变更必须留痕。 |
| 演示门禁 | 每 2 周里程碑前必须通过一条端到端演示路径。 |

| 周期 | 目标 | 工作内容 | 业务专家投入 |

4. 所有缓冲期修复必须有测试或验收证据。

## 十、AI Agent 可独立承担与必须人工介入的边界

| 工作类型 | AI Agent 可主导 | 必须人工介入 |
|----------|-----------------|--------------|
| CRUD、API、页面、表单、迁移 | 是 | 合并前 Review。 |
| 主数据模型设计 | 可出草案 | 人工确认字段、关系、约束和迁移。 |
| Prompt 模板 | 可生成候选 | 业务专家和项目负责人确认。 |
| 规则配置样例 | 可生成候选 | 业务专家确认业务口径，负责人确认表达式安全。 |
| 权限过滤 | 可生成实现和测试 | 人工必须 Review，权限误放行为 No-Go。 |
| AI 输出采纳 | 可生成流程 | 人工必须 Review Schema、脱敏、规则护栏和审计。 |
| E2E 测试 | 可生成并维护 | 人工确认验收覆盖是否充分。 |
| 交付文档 | 可生成初稿 | 人工确认准确性和范围边界。 |

## 十一、任务并行策略

推荐并行方式：

1. 后端 Agent 和前端 Agent 以 OpenAPI / Schema / Mock 数据为契约并行。
2. Test Agent 与功能开发同步生成测试，不等功能完成后补测。
3. Docs Agent 在每个里程碑前更新接口说明、演示说明和验收证据。
4. Review Assistant Agent 在每次合并前对照 `ARCHTECT.md`、`SPEC.md` 和 Prototype 文档做契约偏差检查。

不推荐并行方式：

1. 多个 Agent 同时修改同一核心模型文件。
2. 未冻结 API 契约就让前后端大规模并行。
3. 让 Agent 一次生成跨数据模型、后端、前端、测试、文档的大型改动。
4. 在缓冲期继续并行开发新增功能。

| 里程碑 | 必须提供的证据 |
|--------|----------------|
| M1 接入到资产化演示 | 接入请求、原始对象、作业记录、标准化引用、资产目录、资产详情、当前版本读取模型截图或接口返回。 |
| M2 AI 治理与规则护栏演示 | Prompt 配置版本、LiteLLM 模型别名、AI 执行记录、质量报告、规则命中、决策日志、`available` / `review_required` 示例。 |
| M3 P0 全链路验收演示 | RAGFlow 索引清单、检索/QA 结果、权限过滤、L4 脱敏、引用追溯、重处理、重治理、AI 重评分、审计日志。 |
| M4 正式验收汇报 | 12 个 E2E 用例执行结果、NFR 指标、Go/No-Go 表、业务验收意见、遗留问题分级、交付文档清单。 |

## 十三、最终建议

最优推进方式不是简单把 8 周压缩为 4 周，而是：

1. 用 AI Agent 把代码、测试、文档产出速度提升到 6 周可验收。
2. 用人工 Review 守住架构边界、数据模型、权限、审计、AI 治理和验收质量。
3. 用 2 周缓冲吸收外部系统联调、业务确认延迟和正式汇报风险。
4. 用任务包、Review Gate、质量门禁和里程碑证据防止 AI 快速产出带来的隐性质量债。

因此，本项目建议采用“6 周内部验收 + 2 周正式交付缓冲”的 AI Agent 开发节奏。
