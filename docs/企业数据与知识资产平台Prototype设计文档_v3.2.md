## 一、v3.2 设计理念

### 1.1 设计愿景

NEXUS v3.2 控制台定位为**轻量化 SaaS 风格的现代数据资产管理平台**，面向中小型组织（教研团队、院校、中型企业），强调"开箱即用、可视化驱动、零学习曲线"。同时兼顾**企业数据资产管理特有的操作模式**——批量处理、SLA 驱动的审核队列、全链路可追溯、安全合规操作。

### 1.2 七大设计原则

| 原则 | 说明 | 设计表现 |
|------|------|----------|
| **简洁高效** | 一屏一事，核心操作 ≤ 3 步完成 | 减少嵌套弹窗，优先内联编辑和侧边抽屉 |
| **可视化驱动** | 数据状态一目了然，决策有据可依 | 仪表盘、趋势图、状态卡片、交互式血缘 DAG |
| **操作效率优先** | 支持批量处理和快速扫描 | 批量采纳、批量标签、保存筛选视图 |
| **模块化** | 功能独立、按需组合、可扩展 | 组件化设计系统，页面即模块拼装 |
| **云原生美学** | 现代、轻盈、有呼吸感 | 柔和渐变、微阴影、圆角卡片、充足留白 |
| **安全可信** | 高风险操作可防误触，变更可追溯 | 软删除回收站、输入确认、操作预览 |
| **离线兼容** | 不依赖 WebSocket，中小规模轮询友好 | HTTP 轮询 + 乐观更新 + 渐进增强 |

### 1.3 v3.2 关键设计决策

| 决策点 | 结论 | 理由 |
|--------|------|------|
| 状态更新机制 | **HTTP 轮询（3-5s）+ 手动刷新** | 中小规模单节点部署，不引入 WebSocket 依赖 |
| 规则管理方式 | **外部 JSON 注入，保存即生效** | 低频变更，无需发布审批流 |
| Prompt 管理方式 | **内置模板 + 自动选用 + 试验场** | 低频变更，治理环节自动选用对应的 Prompt+规则集组合 |
| NAS/Web 同步导入 | **异步模式：先同步后补元数据** | 大批量场景无法提前预知校验状态和元数据 |

### 4.11 轮询状态指示器（v3.2 新增）

用于中小规模部署下的作业和流水线状态更新：

```
┌─ 状态栏 ────────────────────────────────────┐
│ ◉ 自动刷新中 · 每5秒    [手动刷新] [暂停]  │
│ 上次更新: 2s前 · 响应: 45ms                │
└─────────────────────────────────────────────┘
```

| 状态 | 表现 | 说明 |
|------|------|------|
| 轮询中 | 蓝色脉冲 ◉ + "自动刷新中" | 默认每 5 秒 GET 作业状态 |
| 暂停轮询 | 灰色 ◎ + "已暂停自动刷新" | 用户手动暂停，减少请求 |
| 轮询失败 | 红色 ● + "刷新失败·重试中" | 网络错误，3 次后退避 |
| 手动刷新 | 按钮 loading | 用户主动点击刷新按钮 |

**轮询策略（P0 实现）：**
- 活跃作业（状态 `queued`/`running`）：每 3 秒轮询
- 关注作业（用户在详情页查看的）：每 3 秒轮询
- 后台作业（非关注的其他活跃作业）：每 10 秒轮询
- 已完成/失败作业：不轮询
- 工作台统计卡片：每 30 秒轮询
- 手动刷新按钮始终可用，不受轮询间隔限制

## 八、页面设计 — 数据接入 NX-03（v3.2 重构）

### 8.1 设计目标

数据接入需要支持两种截然不同的场景：

| 场景 | 数据量 | 元数据 | 校验 | 交互模式 |
|------|--------|--------|------|----------|
| **交互式上传** | 少量文件（≤50） | 可提前指定 | 即时校验 | 三步向导 |
| **异步批量同步** | 大批量（NAS/Web） | 无法提前指定 | 无法提前校验 | 异步提交+事后补齐 |

v3.2 的核心设计思路：**入口统一，模式自适应**。

### 8.5 异步同步后的元数据补齐流程

提交同步任务后，系统异步完成以下流程。用户在作业中心和治理中心逐步完成元数据补齐：

```
提交同步任务
  │
  ▼
作业中心：扫描作业（scan_directory / fetch_webhook）
  │  状态：◉ 扫描中... 已发现: 127 个文件
  ▼
作业中心：接入校验（ingest_validate × N）
  │  结果：✓ 120 通过 · ⚠ 5 格式不支持 · ✕ 2 大小超限
  ▼
原始台账：批次详情（可查看全部文件列表和校验结果）
  │
  ▼
作业中心：assetize → parse → normalize（批量并行）
  │
  ▼
治理中心 → AI 治理建议 Tab
  │  所有资产以「待分类」状态进入 AI 建议队列
  │  用户在此补齐：数据域、分级、标签、组织范围
  │  支持：批量采纳高置信度 AI 建议、逐条裁定低置信度条目
  ▼
治理中心 → 决策追踪
  │  元数据补齐完成，资产进入 available
  ▼
资产目录：可检索、可索引
```

**关键交互要点：**

1. **不阻塞接入** — 文件可以先接入（raw_object），元数据可以事后在治理中心补齐
2. **批量补齐** — 治理中心 AI 建议队列中，批量筛选、批量采纳/修改
3. **逐步收敛** — AI 给出初步分类建议 → 人工批量确认高置信度 → 低置信度逐条裁定 → 全部资产生效
4. **来源可追溯** — 每个文件即使元数据未补齐，也保留完整的 raw → ingest → parse → normalize 链路

### 10.5 交互细节

| 操作 | 行为 |
|------|------|
| 进入作业详情页 | 立即开始 3s 轮询，页面标题显示脉冲 ◉ |
| 离开详情页 | 降级为列表 5s 轮询 |
| 点击「暂停」 | 停止自动轮询，显示 ◎ 已暂停，保留手动刷新按钮 |
| 轮询失败 3 次 | 按 1s/3s/9s 退避重试，显示 ⚠ 连接异常 |

| 手动刷新 | 点击 ↻ 按钮立即请求最新状态，不影响轮询节奏 |

### 10.6 渐进增强：WebSocket 预留

### 14.4 规则集的四类规则详解

#### A. 数据域分类规则 (`domain_classification`)

| 字段 | 说明 |

#### B. 数据分级规则 (`data_classification`)

| 字段 | 说明 |

| 典型规则 | 高敏关键词检测、数据域默认分级、来源可信度加权 |

#### C. 标签设置规则 (`tag_labeling`)

| 字段 | 说明 |
|------|------|
| 目的 | 从 7 个标签维度自动设置推荐标签 |
| 输入 | 标题、摘要、内容片段、数据域、分级 |
| 输出 | 7 维标签推荐列表 + 置信度 |
| 典型规则 | 关键词→专业领域映射、数据域→学历层次推断 |

#### D. AI 质量评分规则 (`quality_scoring`)

| 字段 | 说明 |
|------|------|
| 目的 | 评估资产内容质量的 5 个维度 |
| 输入 | 标准化内容、解析结果、元数据完整性 |
| 输出 | 综合分 + 5 维度分（完整性/结构性/可读性/安全性/可追溯性） |
| 典型规则 | 正文缺失阈值、章节结构完整性、切片连贯性检查 |

### 15.3 治理环节自动选用机制

**自动选用规则：**
- 系统根据治理目标自动匹配对应的 Prompt 模板和规则集组合
- 用户无需手动选择，系统始终使用当前生效的模板版本
- 如果某类模板不存在（未配置），对应治理环节跳过（不阻塞流水线）
- 自动选用记录写入 `ai_governance_run` 审计日志，包含模板名和版本

## 二十一、端到端流程 v3.2

### 流程 A：NAS 大批量异步同步 + 元数据事后补齐（v3.2 新增）

```
数据源管理 [NAS-共享盘]
  → 数据接入 [异步批量同步模式]
  → 配置同步范围（路径、文件模式、递归深度）
  → 选择元数据策略：「暂不指定，AI 自动分类」
  → 提交同步任务
  → 作业中心：scan_directory ◉ → ingest_validate ● → assetize ● → parse ● → normalize ●
  → 治理中心 [AI 建议 Tab]：全部资产以「待分类」状态进入队列
  → 筛选「高置信度 ≥85%」→ 批量采纳 80 条
  → 逐条裁定低置信度条目 14 条
  → 全部资产生效 → 资产目录可检索
```

### 流程 B：更新规则集 + 仅影响新资产（v3.2 新增）

```
规则配置 [数据分级规则 v2]
  → 点击 [编辑 JSON]
  → 上传新的 JSON 配置文件（外部编辑器编辑）
  → 系统自动校验：语法 ✓ · Schema ✓ · 字段引用 ✓ · 优先级冲突 ⚠
  → 展示影响范围：「131 个历史资产不受影响」
  → [保存并生效]
  → Toast：「规则已生效，新接入资产将使用 v3 分级规则」
  → 已在处理中的 8 个作业继续使用 v2 规则完成
```

### 流程 C：Prompt 试验场验证后更新（v3.2 新增）

```
AI Prompt 配置 [元数据治理 v3]
  → 点击 [复制为新版本] → v4 (草稿)

  → 根据测试结果调整 Prompt
  → 重新测试 → 全部通过
  → [应用此版本到生产（保存并生效）]
  → 后续新接入资产自动使用 v4 模板
```
