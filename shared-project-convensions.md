# Shared Project Convensions

> 本文档沉淀 2026-04-26 至 2026-04-28 期间在需求 Spec、技术架构、Prototype、任务排期、AI Agent workflow、任务包拆解等工作中体现出的通用工作习惯、协作方式和项目推进偏好。
>
> 注意：本文档不是某一个具体项目的功能说明，而是可复用于后续项目的工作约定。文件名按当前约定暂用 `shared-project-convensions.md`。

## 1. 总体工作方式

### 1.1 先理解，再 Review，再优化

处理任何项目资料时，默认遵循：

```text
认真阅读和理解上游文档
  -> 从专业角色视角 Review
  -> 找出不一致、冗余、边界不清和落地风险
  -> 直接优化并输出新版本
  -> 再做一致性校验
```

常见 Review 视角包括：

- 资深软件架构师视角。
- 资深产品经理视角。
- 工程交付负责人视角。
- AI Agent 编程协作视角。

要求不是只给建议，而是要基于 Review 结果直接形成可落地的优化版本。

### 1.2 契约先行，避免边做边漂移

项目推进时优先建立契约，再进入实现：

- 先定义技术架构契约。
- 再定义需求 Spec 契约。
- 再定义 Prototype 契约。
- 再定义排期和交付节奏。
- 再定义 AI Agent workflow。
- 最后拆成可执行任务包。

对于研发实施，不允许仅凭聊天上下文、记忆或宽泛产品描述直接编码。每个任务周期应先读取对应任务包，确认范围、禁区、交付物和验收方式。

### 1.3 版本化输出，保留历史

文档和方案默认采用版本化方式演进：

- 不覆盖旧版本，除非明确要求。
- 每次优化生成新版本，例如 `v1.0`、`v1.1`、`v1.2`、`v2.0`。
- 小幅优化版本号增加 `0.1`。
- 架构发生较大基线变化时，可以升级主版本。
- 每个版本应包含版本记录，说明变化原因和核心调整。

版本化的目的不是形式化管理，而是保证决策可追溯、反馈可回放、方案演进有上下文。

### 1.4 架构文档版本归档规则

架构文档版本演进时遵循以下约定：

- 旧版本归档至 `docs/archived/`，文件名保留原版本号，不删除。
- `ARCHTECT.md` 始终指向当前有效版本，文件头标注版本号和源文档路径。
- 新版本在正文中包含"Key Changes from 上一版本"变更表，列明每条变更的分类和内容。
- 更新 `ARCHTECT.md` 时同步补充新引入的架构约束（如接入层 Adapter 规则、存储 Key 命名规则等）。
- 旧版本契约不得在代码或测试中继续引用；切换版本时需检查是否存在旧版本残留引用。

## 2. 文档工作习惯

### 2.1 文档之间必须相互对齐

需求、架构、Prototype、排期、Workflow 之间必须形成一致闭环：

```text
战略/业务方案
  -> 技术架构
  -> 需求 Spec
  -> Prototype
  -> 排期计划
  -> AI Agent workflow
  -> 周任务包
```

如果架构变更，必须回看需求 Spec 和 Prototype 是否需要同步调整。

如果 Spec 不能指导 Prototype，必须补充 Prototype 设计文档。

如果排期或开发方式改变，必须同步更新 workflow 和任务包。

### 2.2 文档要可执行，而不是只可阅读

文档应尽量能指导后续动作：

- 架构文档应能指导模块边界、数据模型、状态机、接口分层和技术选型。
- Spec 应能指导产品功能、角色权限、优先级、验收标准和 API 范围。
- Prototype 文档应能指导页面、字段、交互、状态、弹窗、空态和演示路径。
- 排期计划应能指导谁在什么时间做什么、交付什么、如何汇报。
- Workflow 应能指导 AI Agent 如何开始任务、如何并行、如何 Review、如何合并。
- 任务包应能指导 AI Agent 直接开始实现。

### 2.3 优先使用结构化表达

偏好的表达方式：

- 表格。
- 状态流。
- 端到端链路。
- 角色和职责矩阵。
- P0/P1/P2 范围表。
- Review 发现、风险、优化动作三列表。
- 任务、责任人、时间、工时、进度、备注的计划表。
- Gate、触发条件、Review 重点的门禁表。

结构化表达用于减少歧义，方便人工和 AI Agent 同时理解。

## 3. 范围控制习惯

### 3.1 强调 P0 最小闭环

项目推进时优先识别 P0 最小可用闭环，而不是一次性展开完整蓝图。

典型格式：

```text
输入
  -> 处理
  -> 治理
  -> 状态流转
  -> 服务开放
  -> 追溯和验收
```

P0 应具备可演示、可测试、可验收的完整主链路。

P1/P2 可以保留为后续演进，但不得挤占 P0 交付。

### 3.2 明确不做项

每个版本或计划都应明确“不做什么”：

- 不做超出一期范围的能力。
- 不重复已有平台能力。
- 不引入不必要的新组件。
- 不在 P0 中塞入 P1/P2。
- 不把预留能力伪装成已实现能力。

不做项要写清楚，避免后续执行中范围膨胀。

### 3.3 对外承诺和对内目标分离

排期上偏好：

- 对内设定更激进的可验收目标。
- 对外保留合理缓冲。
- 缓冲期用于质量、联调、演示、验收和缺陷收敛。
- 缓冲期不承接新需求。

典型节奏：

```text
内部目标：6 周完成可验收版
外部窗口：8 周完成稳妥交付
```

## 4. 架构工作习惯

### 4.1 明确边界，避免重复建设

技术方案中必须清楚说明：

- 哪些能力由本系统承担。
- 哪些能力依赖已有平台。
- 哪些能力只做架构预留。
- 哪些能力不进入本期。

如果已有外部平台可以承担能力，倾向于依赖已有平台，而不是重复开发。

### 4.2 简化模型，避免冗余字段

数据模型设计中强调：

- 能由关系和约束推导的字段，不应重复落主表。
- 当前状态优先通过唯一约束和读取模型表达。
- 避免双向指针导致一致性风险。
- 主数据和执行投影要分离。
- 正式输入和中间产物要分清。

典型偏好：

- 用只读视图或读取模型表达当前版本。
- 用单向引用表达标准化对象归属。
- 避免反向指针、重复指针和双写一致性。

### 4.3 自动优先，人工兜底

业务流程设计中倾向：

- 默认自动流转。
- 人工只处理异常、低置信、冲突、高风险、抽检和申诉。
- 避免所有资产都进入人工审核。
- 状态基线要简单、清晰、可测试。

### 4.4 配置化优先

分类、分级、标签、组织范围、质量准入、复核触发、索引准入等规则应配置化。

不应把业务规则硬编码在接口、Worker、页面或一次性脚本中。

### 4.5 可追溯和可审计是默认要求

重要决策必须能说明：

- 输入是什么。
- 规则是什么。
- 模型或算法是什么。
- 谁做了人工覆盖。
- 最终结果是什么。
- 为什么这样决策。
- 后续如何回放和追踪。

审计、trace_id、决策日志、证据引用是默认工程要求，不是后补项。

## 5. AI 能力引入习惯

### 5.1 AI 主导，但不能黑盒

AI 可以作为治理、评分、生成、建议的主力，但不能直接成为不可解释黑盒。

AI 输出必须具备：

- 结构化结果。
- Schema 校验。
- 证据引用。
- 置信度。
- 采纳状态。
- 人工反馈。
- 可追溯输入摘要。

### 5.2 AI 输出不能绕过规则和人工兜底

AI 结论进入正式结果前必须经过：

- 字段白名单。
- 脱敏策略。
- 输出 Schema。
- 规则护栏。
- 置信度阈值。
- 状态机判定。
- 必要时人工复核。

### 5.3 Prompt 和模型调用边界要清楚

偏好将 Prompt、输出 Schema、评分权重、脱敏策略放在业务平台内维护。

模型路由、模型供应商适配、模型凭据和网关限流等能力，如果已有平台承载，则不重复开发。

### 5.4 AI Agent 是开发主力，但不是决策主体

AI Agent 可以主导：

- 代码生成。
- CRUD。
- API。
- 页面。
- 测试。
- 文档。
- 迁移脚本。
- 样例数据。

人工必须负责：

- 架构边界。
- 数据模型。
- 权限和审计。
- AI 治理安全。
- 规则口径。
- 验收结论。
- 合并决策。

## 6. AI Agent 工作流习惯

### 6.1 每周任务从任务包开始

每周任务启动时，优先阅读对应任务包：

```text
docs/task-packages/wk_<week>_task_package.md
```

不允许仅凭记忆、聊天上下文或宽泛文档直接开始实现。

如果当前工作不在任务包范围内，应先补充或调整任务包，再开始实现。

### 6.2 任务包必须边界清楚

非平凡任务必须使用任务包方式描述，至少包含：

- 任务名称。
- 来源上下文。
- 目标。
- 范围。
- 不在范围。
- 禁止变更。
- 交付物。
- 验收方式。

任务包建议控制在 0.5 到 1.5 天可完成。

### 6.3 多 Agent 并行时必须契约先行

该约束仅限存在多 AI Agent 并行场景。

多 Agent 并行开始前，必须先创建并行契约，明确：

- 共同目标。
- 参与 Agent。
- 人工 owner。
- 冻结的 API、Schema、状态枚举、事件、UI 状态语义。
- 各 Agent 的写入范围。
- Mock 数据和 fixture。
- 集成顺序。
- Review Gate。
- 冲突解决方式。

如果任务中途引入并行 Agent，需要暂停实现，先补并行契约。

### 6.4 小步合并，不做大爆炸集成

偏好：

- 小任务包。
- 小 PR。
- 单一职责。
- 明确写入范围。
- 每周可演示切片。
- 每 2 周里程碑汇报。

不偏好：

- 一次性跨后端、前端、数据模型、测试、文档的大型改动。
- 未冻结 API 就让前后端大规模并行。
- 多个 Agent 同时改同一核心模型或迁移文件。

## 7. Review Gate 习惯

### 7.1 高风险变更必须过门禁

以下变更必须人工 Review：

- 数据模型。
- AI 治理。
- 规则引擎。
- 权限和审计。
- 版本状态。
- 外部系统集成。
- API 契约。
- P0 核心 UX。
- 里程碑和最终验收。

### 7.2 分层 Review 模式

复杂系统的人工 Review 应按关注点分层推进，而不是一次性全量 Review：

典型分层顺序（以数据平台为例）：

```text
第一层：数据契约（枚举、模型字段、命名、约束、状态机）
  -> 第二层：接入层（幂等、去重、Adapter 边界、存储 Key 设计）
  -> 第三层：流水线层（阶段划分、错误分类、重试策略、事务边界）
  -> 第四层：治理层（AI 输入、规则护栏、状态流转、审计事件）
  -> 第五层：API 和前后端契约（路由、Schema、错误码、状态语义）
```

说明：

- 每层 Review 完成后才进入下一层，避免底层问题传导导致上层返工。
- 每次 Review 后应更新 `docs/Human_Review_Feedbacks.md`（见 §7.3）。
- 不同项目可根据实际情况调整分层顺序和粒度，但分层原则保持。

### 7.3 Review Feedback 文档约定

项目根目录 `docs/Human_Review_Feedbacks.md` 是人工 Review 意见的持久化记录，每次 Review 后必须更新。

文件结构：

- 每次 Review 为一个独立章节，标注时间和审查范围。
- 每条意见包含：原始意见引用、处置动作、理由。
- 处置动作分三类：
  - **Action: 已实施** — 按意见修改了代码/文档，说明具体改动和相关文件。
  - **Action: 保持不变** — 经评估决定维持现状，必须附上充分理由（"保持不变"不等于忽略意见）。
  - **Action: 延期到 wk_N** — 认可意见但推迟处理，说明推迟原因和规划的任务包。
- 文件末尾标注最后更新时间（"最后更新：YYYY-MM-DD（Review N 完成后）"）。

维护规则：

- AI Agent 不得在没有人工确认的情况下将意见标记为"保持不变"。
- 每次 Review 后当次 commit 应包含 `Human_Review_Feedbacks.md` 的更新。
- 文件随项目版本一起纳入 git 管理。

### 7.4 Review 重点不是代码风格

人工 Review 重点是：

- 是否违反架构禁区。
- 是否引入范围膨胀。
- 是否破坏数据一致性。
- 是否绕过权限和审计。
- 是否缺少测试和证据。
- 是否影响追溯。
- 是否让 AI 输出直接进入正式结果。
- 是否隐藏未完成能力。

普通样板代码、简单 CRUD 和页面细节可更多交给 AI Agent 和自动化检查。

## 8. API 和前后端协作习惯

### 8.1 API 风格统一 RESTful

API 默认遵循 RESTful 风格：

- 面向资源命名。
- HTTP 方法语义清楚。
- 状态码和错误体明确。
- 请求和响应 Schema 稳定。
- 变更有契约测试。
- 需要幂等的写操作必须设计幂等键或等价机制。

### 8.2 业务 API 和控制面 API 分层

面向外部业务方、上层系统、API 调用方、智能应用的业务 API，应由业务 API 包承载。

控制台控制面 API 原则上只服务管理控制台，不对外提供业务方调用。

如果一个能力既被控制台使用，也需要被外部业务方复用，应优先沉淀为业务 API，而不是只放在控制台内部 API。

### 8.3 前后端并行前先冻结契约

前后端并行开发前必须先确认：

- API path。
- 请求 Schema。
- 响应 Schema。
- 状态枚举。
- 错误码。
- UI 状态语义。
- Mock 数据。
- 字段命名。

## 9. 技术栈与依赖管理习惯

### 9.1 Git Commit Message 约定

commit message 应简要说明本次变更的意图和范围，不追加变更文件列表（文件列表由 `git diff` 或 `git show` 获取，写入 message 是冗余信息）。

格式约定：

```text
<type>: <一句话说明变更意图>

<可选正文：补充背景、设计决策或不可从代码直接读出的原因>
```

常用 type：`feat`（新功能）、`fix`（缺陷修复）、`refactor`（重构）、`docs`（文档）、`test`（测试）、`chore`（工程配置）。

示例（正确）：

```text
refactor: adapter pattern for ingest gateway, eliminate duplicate submit logic
```

示例（错误，不要这样写）：

```text
update files: gateway.py, adapter_base.py, adapter_file.py, adapter_crawler.py, enums.py
```

### 9.2 Python 依赖统一使用 uv

Python 技术栈默认统一采用 `uv` 进行依赖包管理。

约定：

- 使用 `pyproject.toml` 作为 Python 项目的依赖声明入口。
- 使用 `uv.lock` 锁定依赖版本，保证本地、CI 和部署环境一致。
- 新增、升级、删除 Python 依赖时优先使用 `uv add`、`uv remove`、`uv sync` 等 uv 工作流。
- 不默认使用 `pip freeze > requirements.txt` 作为主依赖管理方式。
- 不默认引入 Poetry、Pipenv 或其他 Python 包管理器，除非项目已有强约束或用户明确要求。
- 文档、Agent 任务包和执行命令中应优先使用 uv 命令。

目的：

- 降低多环境依赖漂移。
- 提升安装和同步速度。
- 让 AI Agent、人工开发和 CI 使用一致的依赖入口。

## 10. 排期和里程碑习惯

### 10.1 排期表字段固定化

项目排期偏好使用固定字段：

- 工作阶段/事项。
- 工作内容。
- 责任人。
- 计划开始时间。
- 计划结束时间。
- 计划工作量。
- 实际开始时间。
- 实际结束时间。
- 实际工作量。
- 完成进度。
- 立项阶段。
- 备注。

实际字段在计划阶段保持为空，执行中再更新。

### 10.2 工时计算清楚

默认工时计算方式：

```text
计划工作量 = 计划工作日数 * 7.5 小时/人
```

资源不是每周都必须满投入，尤其是业务专家，应集中在关键确认点。

### 10.3 里程碑必须可演示

里程碑不是只汇报进度，而应能演示：

- 主链路。
- 数据和状态。
- 页面。
- API。
- 追溯证据。
- 风险和缺口。

每 2 周设置一个演示汇报节点是一种偏好节奏。

## 11. 验收习惯

### 11.1 验收要有证据

验收不只看“功能完成”，还要看：

- 测试结果。
- 接口返回。
- 页面截图。
- 状态流转。
- 审计日志。
- 追溯链路。
- 错误和异常样本。
- Go / No-Go 结论。

### 11.2 可演示版和可验收版要区分

偏好清楚区分：

- 可演示版：主链路能讲清楚、跑起来，但可能还有缺口。
- 可验收版：权限、审计、异常流、E2E、追溯和质量指标达到要求。
- 稳妥交付版：在可验收基础上完成缺陷收敛、文档、汇报和归档。

不能把可演示版包装成正式验收版。

### 11.3 缓冲期只处理质量和交付风险

缓冲期原则：

- 不接新需求。
- 不做 P1/P2。
- 不重构已验收主链路。
- 只处理阻塞、安全、追溯、演示稳定和交付文档问题。

## 12. 输出偏好

### 12.1 命名清晰，可直接落盘

文件命名偏好：

- 带版本号。
- 能表达用途。
- 使用项目内约定目录。
- 根目录放通用契约。
- `docs/` 放方案、排期、任务包和设计文档。

### 12.2 输出要有最终校验

生成或修改文档后，应做校验：

- 文件是否存在。
- 文件名是否正确。
- 关键章节是否存在。
- 关键约束是否命中。
- 版本号是否正确。
- 是否有旧概念残留。
- 是否符合用户指定结构。

### 12.3 优先直接产出，而不是只提建议

当目标明确时，偏好直接修改或生成文件。

只有在信息不足、会造成重大风险或存在不可判断分歧时，才需要先询问。

## 13. 通用禁区

除非用户明确变更方向，以下行为应避免：

- 未读上游文档就开始写方案或代码。
- 覆盖历史版本而不留痕。
- 只提出 Review 意见但不优化文档。
- 把 P1/P2 偷偷塞进 P0。
- 将外部已有平台能力重复开发。
- 数据模型中增加冗余反向指针。
- 将 AI 输出直接当作正式结果。
- 缺少审计、追溯和测试证据。
- 未冻结契约就并行开发。
- 控制面内部 API 对外充当业务 API。
- 里程碑只讲进度而无演示证据。
- Python 项目中绕过 uv 引入另一套依赖管理方式，除非用户明确要求。
- git commit message 中追加变更文件列表（文件列表属于 git diff，不属于 commit message）。
- Review 意见标记为"保持不变"但不附理由。
- 架构版本升级后不归档旧版本、不更新 ARCHTECT.md 指向。
- Review 完成后不更新 `docs/Human_Review_Feedbacks.md`。
- 一次性对所有层级展开全量 Review 而不按层推进。
